2.3 YANG Module Introduction 18
The data models representing the O1 interface are organized as a set of reusable YANG modules , defined according to 19
[6]. The YANG modules need to be able to support a variety of deployment scenarios, including  20
i) where the O -RU functionality is co -located with corresponding O -DU functionality, in a classical 3GPP 21
“DU” approach,  22
ii) where the O-RU is deployed using O-RAN WG4 defined hierarchical approach where the O-DU is used to 23
manage all aspects of the remote O-RU connected using the WG4 defined fronthaul interface, and 24
iii) where the O -RU is deployed using O -RAN WG4 defined hybrid approach where the SMO and O-DU in 25
combination are used to manage the remote O-RU connected using the WG4 defined fronthaul interface. 26

configuring the O -RU functionality. Because an O -DU managed function will likely support multiple O -RU functions, 28
the O-RAN O1 Interface specification for O -DU defines new models which are used to aggregate the configuration of 29
individual O-RU instances. Figure 2 illustrates how the SMO is responsible for aggregating the individual configuration 30

manual configuration is successful, the NETCONF server shall be able to recover this configured information and use the 16
o-ran-mplane-int.yang model to communicate this operational-state to a NETCONF client. 17
b) DHCP server provides O-DU’s transport layer address information together with the identity of the SMO/MnS 18
Consumer(s). This identity encodes either the transport layer address or FQDN of the SMO/MnS Consumer. If an FQDN 19
is signaled, the O -DU shall use the DNS server address provided by the DHCP server to recover the  IP address 20
corresponding to FQDN of the SMO/MnS Consumer. 21
c) If IPv6 is suppo rted, Stateless Address Auto -Configuration (SLAAC) is used to configure the O-DU’s transport 22
address with the DHCP server providing the identity of the SMO/MnS Consumer . This identity encodes either the  23
transport layer address or FQDN of the SMO/MnS Consumer. If an FQDN is signaled, the O-DU shall use the DNS server 24
address provided by the DHCP server to recover the IP address corresponding to FQDN of the SMO/MnS Consumer.  25

interfaces supported by the O-DU and using the presence of the augmented ipv6 container in the o-ran-interfaces module 27

ports may be configured either as an access port, where untagged Ethernet frames are used, or as a trunk port, where 35
multiple VLANs are  configured. At start up, the O -DU will typically not be able to immediately determine whether its 36
ports are attached to remote transport equipment configured for access or trunk mode operation.   37

3.1.3 O-DU IP Address Assignment used for O1 signalling 18
Automatic IP address assignment for the O-DU management plane can be achieved using different techniques: 19
1. IPv4 configuration using DHCPv4, RFC2131 [1 2] enables DHCP servers to configure IPv4 network address(es) on 20
the O-DU. An O-DU shall support the behavior specified in RFC 4361 [8], using stable DHCPv4 node identifiers in their 21
dhcp-client-identifier option. 22
Note: a network realized with multiple DHCP servers should ensure that their configurations are co -ordinated to ensure 23
a common default gateway is provisioned in an O -DU which receives multiple DHCPv4 responses, e.g., when received 24
over different interfaces. 25
Note: an O-DU may indicate that it supports configuration of routing information using RFC 3442, enabling static routes 26
to be used by the O-DU when determining how to route uplink packets, e.g., when the O-DU supports multiple interfaces. 27
For O-DUs that support IPv6, both stateful and stateless address assignment procedures are supported:  28
2. IPv6 Stateless Address Auto -Configuration (SLAAC), RFC4862 [1 3] enables the O -DU to generate link -local and 29

Chapter 11 Fronthaul Delay Management Support 13
O-RAN WG4 [4] specifies delay management between O-DU and O-RU, where transmission of data over fronthaul is 14
governed by O-DU transmission/reception window and O-RU transmission/reception window. WG5 M-Plane needs to 15
support the function. O-RU transmission/reception window is decided by O-RU itself, and the relevant procedures and 16
data models are specified in WG4 spec [4]. This chapter addresses how O -DU transmission/reception window can be 17
determined and the relevant procedures and data models. 18
  19
WG4 spec [4] does not define how O-DU transmission/reception window is determined, but this spec assumes the 20
below 2 ways.  21
A) Not Configured by M-Plane 22
➢ O-DU determines O-DU transmission/reception window. How O-DU determines the window is up to O-23
DU, it may be hardcoded value at O-DU, or calculated from O-RU transmission/reception window and 24
FH delay based on O-RAN WG4 CUS specification [4] 25
Note: WG4 IOT profile describes O-DU transmission/reception window values. O-DU 26
transmission/reception window can be hardcoded based on these values.  27
➢ There is no impact to WG5 M-Plane spec since O-DU transmission/reception window is determined at 28
O-DU. 29
B) Configured by M-Plane 30
➢ SMO configures parameters that control the O-DU transmission/reception windowby M-Plane. The SMO 31
configured parameters should be taken by the O-DU as the maximum or minimum allowed values for the 32
respective transmission/reception window parameters. Having the SMO configure maximum or 33
minimum allowed values instead of the exact value, provides necessary flexibili ty for the O-DU 34
implementation. SMO may take into account O-RU reported window and FH delay to determine O-DU 35
transmission/reception window. FH delay is pre-defined (Defined Transport Method) or measured 36
(Measured Transport Method).  37
➢ There is impact to WG5 M-Plane spec where the details are described below. 38

Note: This version focuses on hierarchical model. Next version will include hybrid model since this scheme can be 40
extended to hybrid model. 41
