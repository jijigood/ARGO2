1.4 Philosophy 39 
The O-RAN O1 management services follow existing standards wherever possible.  The focus of this document  is to 40 
identify the use cases which conform to existing standards, identify gaps in management services for O -RAN and define 41

and update the references in this document as the standards evolve to cover the gaps. If extensions and gaps are not 2 
specified, it is expected that the management services producers and consumers are conforming to referenced 3GPP 3

1.5 Open Points 8 
As each Management Service is evaluated, the Use Cases and relevant specifications need to be assessed and 9 
augmented as needed to support O-RAN. The current list of Use Cases in Chapter 2 below is not exhaustive, and the list 10 
of specification references may not be complete. 11 
Some Use Cases referred to in the standard may not be applicable to O-RAN and this needs to be addressed, as an 12 
exception.  13 
The O1 team needs to be clear when citing a reference whether we are using it as a citation (meaning it will be strictly 14 
followed) or as a reference.  Precise terminology needs to be included as this draft matures.  15 
The O1 interfaces are defined assuming that the O-RAN OAM Architecture will ensure that Management Service 16 
Providers behind a NAT are able to communicate with their Management Service Consumer using the same interfaces 17 
as Management Service Providers not behind a NAT.  The O-RAN OAM Architecture team has studied the topic and 18 
has a CR proposed for v02.00 confirming our assumption.  When the document is approved this open point will be 19 
removed. 20

Chapter 2. Management Services 1 
2.1 Provisioning Management Services 2 
Provisioning management services allow a Provisioning MnS Consumer to configure attributes of managed objects on 3 
the Provisioning MnS Provider that modify the Provisioning MnS Provider’s capabilities in its role in end-to-end 4 
network services and allows a Provisioning MnS Provider to report configuration changes to the Provisioning MnS 5 
Consumer. NETCONF is used for the Provisioning Management Services to Create Managed Object Instance, Delete 6 
Managed Object Instance, Modify Managed Object Instance Attributes and Read Managed Object Instance Attributes. 7 
A REST/HTTPS event is used to notify the Provisioning MnS subscribed Consumers when a configuration change 8 
occurs.   9 
Use cases are from 3GPP TS 28.531. 10 
Reference documents include 3GPP TR 28.890, TS 28.531, RFC 6241, “Network Configuration Protocol 11 
(NETCONF)”, IETF, RFC 7950, “The YANG 1.1 Data Modeling Language” . 12 
2.1.1 General NETCONF Requirements 13 
REQ-GNC-FUN-1:  The provisioning management service provider and consumer SHALL support the following 14 
NETCONF operations: 15 
• get 16 
• get-config 17 
• edit-config 18 
• lock 19 
• unlock 20 
• close-session 21 
• kill-session 22 
Other operations are optional. 23 
REQ-GNC-FUN-2:  The provisioning management service provider and consumer SHALL support the following 24 
NETCONF capabilities: 25 
• writable-running 26 
• rollback-on-error 27 
• validate 28 
• xpath 29 
Other capabilities are optional. 30 
REQ-GNC-FUN-3:  The provisioning management service provider and consumer SHALL support a running datastore 31 
for NETCONF.  Support for a candidate datastore is optional. 32 
REQ-GNC-FUN-4:  The provisioning management service provider and consumer SHALL support YANG1.1, defined 33 
in RFC 7950, including coexistence with YANG Version 1 as specified therein . 34 
2.1.2 Create Managed Object Instance 35 
2.1.2.1 Description 36 
Provisioning MnS Consumer sends a synchronous provisioning update request to the Provisioning MnS Provider to 37 
create a Managed Object Instance (MOI) on the Provisioning MnS Provider and set its attribute values.  38

2.1.2.2 General NETCONF Session Requirements 1 
Requirements are to be specified in a 3GPP spec for the Provisioning management services.  Until that time, the 2 
requirements are provided in this O1 Interface Specification.  3 
REQ-NCS-FUN-1:  The provisioning management service provider SHALL have the capability to establish a 4 
NETCONF session with its authorized consumer upon request from the consumer. 5 
REQ-NCS-FUN-2:  The provisioning management service provider SHALL support an established NETCONF session 6 
until the authorized consumer terminates the session.  NOTE: The consumer may want to perform multiple provisioning 7 
management services operations during a single NETCONF Session.   8 
REQ-NCS-FUN-3:  The provisioning management service provider SHALL have the capability to terminate a 9 
NETCONF session with its authorized consumer when requested to do so by the authorized consumer.    10 
2.1.2.3 Create MOI Requirements 11 
Requirements are to be specified in a 3GPP spec for the Create Managed Object Instance use case.  Until that time, the 12 
requirements are provided in this O1 Interface Specification.  13 
REQ-CMO-FUN-1:  The provisioning management service provider SHALL have the capability to allow its authorized 14 
consumer to create MOI(s) and set attribute values of those MOIs via the NETCONF <edit-config> operation. 15 
REQ-CMO-FUN-2:  The provisioning management service provider SHALL have the capability to make the new 16 
MOI(s) persistent over a reset.  17

severity set to major or critical.  19

2.8 PNF Software Management Services 17 
Software management services allow a PNF Software MnS Consumer to request a physical PNF Software MnS 18 
Provider to download, install, validate and activate a new software package a nd allow a physical PNF Software MnS 19 
Provider to report its software versions.  O-RAN will utilize the liaison to 3GPP to initiate enhancements to the 3GPP 20

the O-RAN Alliance Working Group 4 Management Plane Specification [29].  23 
2.8.1 Software Package Naming and Content 24 
PNF Software Package naming, content and format are vendor specific and do not require standardization in O -RAN.  25 
A PNF Software Package may contain one or more files.  Some of the files in the Software Package may be optional for 26 
the PNF (example: a file that has not changed version).  The PNF is aware of the content and format of its available 27 
Software Packages and can determine which files it needs to download. 28 
The softwarePackage Managed Object Class (MOC) contains attributes about a software package such as:  software 29 
package name, version, fileList, integrityStatus (valid, invalid, empty), runningState (active, passive), vendor, 30 
productName, softwareType (operational, factory), etc. This MOC is applicable to VNFs and PNFs and is a generic 31 
term that O-RAN will use to refer to the software available on the PNF rather than the legacy term of software slot  32
