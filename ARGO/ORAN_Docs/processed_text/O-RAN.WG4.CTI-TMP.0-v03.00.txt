The present document describes the Transport Management Procedures for the Cooperative Transport Interface (CTI). 14 
CTI is an interface between O-DUs and Transport Nodes of a packet-based transport network that is used to interconnect 15 
the O-DUs to a variety of O-RUs. CTI specifically targets transport Nodes that manage a shared point-to-multipoint access 16 
network. Intermediate transport nodes (routers and switches) that only manage point -to-point links do not exchange CTI 17 
messages with the O -DUs. CTI consists of a Transport Control plane (TC) and a Transport Management (TM). This 18 
document specifies the TM Information Model / Dara Model and procedures. The TC-plane is described in [1]. The reader 19 
is referred to the TC -plane document for a description of the protocol as well as background, context, motivation and 20 
topologies with CTI. 21 
This document focuses on the management aspect of CTI. The goals are to define the CTI-related configurable parameters 22 
at TN and at O-DU, to define a suitable set of YANG modules to capture these parameters, to describe necessary steps in 23 
the configuration of the parameters, and to define an auto -discovery method to automate the correlation between TN 24 
specific parameters and CTI parameters. 25 
Disclaimer; the integration of the CTI Information Model into the broader O-DU Information Model is only partial in this 26 
version of the  specification, full integration is subject for further study and completion in a future version of this 27 
specification. 28

4 High Level Description 1 
4.1 General CTI Management Architecture 2 
4.1.1 Foreword on management domains  3 
This document refers to separate “transport domain OSS” and “ RAN domain SMO”, without any implication of 4 
ownership of the domain s or use of particular processes or protocols in the transport domain OS S. E ach domain is 5 
responsible for managing its nodes and can do so independently, but for CTI some level of mutual coordination is needed. 6 
Note that using two domains in this document does not preclude a lternative approaches such as an integrated OSS for 7 
both domains, or an overarching orchestrat ing function that would select the parameters and forward them to the OSS 8 
domains.  9 
4.1.2 Summary of architecture  10 
As the configuration of CTI parameters in the O-DU is non real-time, it will be done over the O-RAN O1 interface using 11 
NETCONF/YANG. 12 
This document does not make assumptions about the management process in the Transport OSS.  13 
Several CTI parameters are common for O-DU and TN side and must be aligned. There is a need for coordination between 14 
both operation domains. This document specifies the needs, but does not make assumptions about the process or protocols 15 
to ensure such coordination. The only process described in this document is an automated auto -discovery which allows 16 
to link a TN specific transport parameter with each individual CTI session ID. 17 
This document does not make assumptions about the integration of functional features into physical nodes. For example, 18 
the document does not restrict the realization of the RAN domain DHCP server, which may be integrated with the O-DU, 19 
may be provided by the transport system, or may be accessed via a relay. 20

6 CTI procedures 1 
Proper interpretation of CTI mes sages by the TNs requires a level of cooperation between the CTI server side and CTI 2 
client side to guarantee consistency of the parameter values and and correlation between the parameters. Although it is 3 
possible to rely on manual correlation actions, it is beneficial to automate the process. This section describes an automated 4 
autodiscovery method of the correlation of the CTI session IDs with the TU/TN transport parameter they correspond to.  5 
The same method can also be used to determine which CTI server must be associated per given CTI session ID. 6 
6.1 Correlations to be performed 7 
For proper operation of CTI, several parameters need to be consistent at both TN and at O-DU sides. This means that the 8 
same values must be consistently configured at both CTI server  and CTI client side by their respective management 9 
systems. The method or protocols for this coordination between Transport OSS and RAN SMO are out of scope of this 10 
document. 11 
Static parameters such as timer values, CTI Flow ID filters, CTI Pattern IDs etc … can be coordinated at OSS level 12 
independently of the topology of the O-RUs and transport nodes. 13 
Dynamically allocated parameters such as IP addresses of CTI servers and CTI clients are dependent of the 14 
interconnection topology of TNs and O -DUs, but not of the topology of the O -RUs. They can be coordinated at OSS 15 
level. 16 
The topology of O-RU deployments will require the following correlations for every CTI session ID, based which TU 17 
interface each O -RU interface is connected to  (see Figure 6-1 for a logical view of the correlations. This figure is 18 
conceptual only and does not represent how or where the correlation must be executed (in the TN or in its management 19 
system)): 20 
• The TN must determine  which transport parameter must be linked to the corresponding CTI session ID 21 
(combined with a CTI flow ID if used) in the CTI report messages (note that multiple transport parameters could 22 
be configured on a TU interface, only o ne pertains to a combination CTI session ID + CTI flow ID).  Note that 23 
CTI flow IDs are only unique per CTI client (different clients could use the same Flow ID for different 24 
meanings), so the CTI client must be taken into account in the correlation. 25 
• The TN must also keep track of which CTI client is reporting the CTI session ID, to send messages related to 26 
that CTI session ID. A single CTI client can report about multiple CTI session IDs.  27 
• At O-DU side, the CTI client must correlate which CTI session ID to use for each O -RU interface it reports 28 
about. This is a one-to-one relationship. For reporting between different flows on the same CTI session ID the 29 
O-DU must also be aware of the class of service associated to each flow, which in turn is represented by a given 30 
set of L2/L3/L4 filter parameters reflecting how uplink traffic is tagged by the O -RU. This tagging could be 31 
different in different transport networks (a single O -DU can in principle manage O -RUs in multiple transport 32 
networks), so the CTI server (reflecting the TN) must be taken into account in the correlation. 33

6.2 Auto Discovery for correlation of CTI session ID to transport 4 
parameter in TN 5 
This optional method is a means to automate the correlation of CTI session ID with the required transport parameter on 6 
the corresponding TU (or TU UNI) on the corresponding TU or TU UNI . It avoids relying on manual correlation during 7 
the installation phase of the O-RU to a given TU interface. 8 
The method is based on  re-using the DHCP exchange from the O -RU that is performed during start up of the O -RU to 9 
achieve IP connectivity for the M -plane [4]. Both cases of layer 2 and layer 3 connectivity are very similar , using the 10 
DHCP relay agent in the TN to add the agent circuit ID as the key for the correlation. The only differences are the use of 11 
giaddr by the DHCP relay agent in case of layer 3 connectivity, and the mandatory use of IP in the C -plane and U-plane 12 
traffic in case of layer 3 connectivity. 13 
If the M-plane uses IPv4, the DHCP relay agent adds Option 82 [11], which contains the agent circuit ID (sub option code 14 
1), which is a string uniquely identifying the TU interface on which the DHCP request was received.  15 
If the M-plane uses IPv6, the DHCPv6 relay agent adds Option 18 [12], which contains the agent circuit ID. 16 
Additionally, the possible use of DHCP Option 60 [13] and DHCP Option 124 [14] by the O-RU is specified in [4]. Even 17 
if such options are not used, the O-RU interface will be discovered by the RAN SMO during the autoconfiguration of the 18 
O-RU through M-plane exchanges. 19 
Once the O-RU is configured, the RAN SMO can correlate the agent circuit ID with the O-RU interface ID based on the 20 
allocated IP address. In turn the O-RU interface is uniquely linked to a CTI session ID. The correlation [agent circuit ID, 21 
CTI session ID ] can then be notified back to the transport OSS, which pushes the correlation to the corresponding TN 22 
that terminates the given TU. 23 
It is recommended to use the  unique MAC address of the  O-RU interface as the CTI session ID. For IPv4 t his is 24 
communicated to the DHCP server in the field “chaddr” of DHCP option 82. For IPv6 it can be communicated as “client 25 
link layer address” in DHCPv6 Option 79 [15]. If the O -RU does not use  Option 79 in DHCPv6, there needs to be a 26 
preconfigured association [O-RU ID; MAC address] in the RAN SMO. 27 
The following sections 6.2.1 and 6.2.2 supersede section 3.2.1.2 in [1]. 28

7 CTI Information Model 1 
7.1 General structure of the Information Model  2 
The model in this document applies to the O -DU. Part of this model is common for the TN, but th e specification of the 3 
IM of the TN  is out of scope of this document. TN considerations (IM and DM) in this document are provided as 4 
informational examples.  5 
This version of the document provides the CTI IM itself and a partial integration of the CTI IM into the broader O -DU 6 
IM (see Figure 7-6). The full integration into the broader O -DU IM (choice of IOC as attachment point  for CTIConfig 7 
IOC) is subject to further study and optimization in a future version of the document.  8 
 9 
The logical structure of the CTI IM at O-DU is determined by following facts; 10 
• When an O-DU supports the optional CTI functionality, it can have one or multiple CTI clien ts. Each CTI 11 
client has a separate IOC “CTIClient” that must be configured. 12 
• For each O-DU, the possible CTI server(s) that its CTI client(s) will connect to will be represented by a MOI 13 
“CTIServer”. 14 
• In one O-DU, each CTI client can have connectivity to multiple remote CTI servers, and each such CTI server 15 
can have connectivity to multiple CTI clients in the O-DU (in other words a full mesh of CTI servers and CTI 16 
clients is supported). 17 
• Each combination of CTI server and CTI client will use a “CTIConnProfile”. 18 
• CTISessions of a given CTI client are grouped per corresponding CTI Server into a “CTISessionGroup”.  19 
• CTIFlow(s) are configured per CTI server and can be referred by “CTIFlowsInUse” in one or multiple CTI 20 
sessions (of one or multiple CTI clients). 21 
• “CTISession” for each CTIClient and “CTIFlowsInUse” for each CTISession will also be configured before 22 
the CTI functionality starts to work. 23 
• There are network-wide definitions of CTI patterns. The same pattern has the same meaning for all CTI ser vers 24 
and CTI clients. 25 
 26 
Depending on its position in the YANG structure, a parameter can be unique for the whole node, or only within a smaller 27 
context in the node allowing for multiple values to be configured inside the same node. Additionally, if a parameter must 28 
be unique across the whole network, this consistency must be assured by the SMO system to configure the same value in 29 
all its managed nodes. 30 
 31 
CTI Session ID:  32 
The CTI Session ID must be unique across the whole network. Recommendation is to use the MAC address of the O-RU 33 
interface it is linked to. 34 
The CTI session ID element is used as the cornerstone in the TN and O-DU YANG models. 35 
 36 
CTI Session Group: 37 
In each CTI Client the CTI sessions are regrouped per associated CTI Server.  38 
 39 
O-RU ID or O-RU interface ID:  40 
They are used locally in the O -DU, so in principle only unique within one O -DU. But it is more practical if it is unique 41 
across the whole network.  42 
They are not used in the TN. 43

Filter values linked to a CTI Flow ID:  2 
From O-DU point of view a Flow ID can either represent the same filter values over the whole network (all TNs), or the 3 
same ID could represent different filter values in different transp ort networks (eg if TNs in different transport networks 4 
use different VLAN tagging). Therefore the CTI Flow ID is defined at per -CTI server level in the O-DU.  5 
From the TN point of view a Flow ID can either represent the same filter values over the whole n etwork (all O-DUs), or 6 
the same ID could represent different filter values in different RAN networks, eg if the same ID would be re -used by 7 
accident by two different RAN networks (sets of O -DUs). Therefore the CTI Flow ID is defined at per -CTI client level 8 
in the TN. 9 
 10 
Pattern characteristics linked to a CTI Pattern ID:  11 
From O-DU point of view a Pattern ID will represent the same pattern characteristics no matter on which TN an O-RU is 12 
connected to. The Pattern ID can therefore be unique at O-DU node level. It is up to the RAN SMO to configure consistent 13 
values across the different O-DUs. 14 
From TN point of view a same Pattern ID could be re -used by accident by two different RAN networks (sets of O-DUs) 15 
while representing different pattern characteristics. The CTI Pattern ID is therefore defined at a per -CTI client level in 16 
the TN, allowing to make such differentiation.  17

L2, L3, L4 Parameters for CTI connectivity: 19 
As mentioned in section 5.1, different sorts of connectivity can be used between pairs of clients and servers. The use of 20 
L2, L3 and L4 parameters are part of the network-wide planning. 21 
In the O-DU the parameters are kept in the CTI Client, CTI Server, and CTI connectivity profile. The CTI client keeps 22 
its MAC address (in case of Ethernet connectivity). The CTI Server keeps its connectivity type, its MAC address (in case 23 
of Ethernet connectivity) or its host (in case of UDP/IP connectivity). The CTI connectivity profile can be used between 24 
one or mulitple pairs of CTI Client and CTI Server and keeps its connectivity type, the VLAN tag in case of Ethernet 25 
connectivity or the VLAN tag in case of UDP/IP connectivity, and the client host (in case of UDP/IP connectivity).  26 
In the TN the parameters are grouped into a so -called “CTI connectivity profile”, and a pair of CTI client – CTI server 27 
can have its own connectivity profile. In the TN the connectivity profiles are defined at top level, and each CTI session 28 
ID can refer to a given connectivity profile.  29 
Note that there is no distinction between message types (all message between a CTI client and CTI server use the same  30 
connectivity parameters).  31 
 32 
CTI timers 33 
It is assumed that there will be no difference in timer values throughout the network, their values will be unique.  34 
 35 
UDP listening port 36 
When UDP/IP is used for CTI transport between a CTI client and a CTI server, all messages between all CTI servers and 37 
all CTI clients over all networks  will use the same destination port value . Without a well -known IANA port, a single 38 
value must be consistently configured in all TN and O-DU nodes.  39 
 40 
CTI versions 41 
Each TN can support one or more versions, each O -DU can support one or more versions. The supported versions are a 42 
list structure in the O-DU and in the TN. Each CTI client must know what versions are supported by a  given CTI server, 43 
and vice versa. The CTI version is defined per CTI client in the TN, per CTI server in the O -DU. 44 
Please note there is no direct version negociation mechanism in CTI between a CTI client and a CTI server, it is up to the 45 
management to select the appropriate version to be used. 46

8 CTI YANG Data Model  1 
This document specifies the CTI YANG Data Model to be used in the O-DU 2 
This document also proposes a CTI YANG Data Model to be used in a TN from a generic perspective, with a specific 3 
declination for PON OLT and DOCSIS CMTS . The TN YANG Data Model is provided here as information for further 4 
distribution towards the corresponding external bodies for their consideration (Broadband Forum for PON, Cable Labs 5 
for DOCSIS). 6 
  7 
The YANG Data Model follows a modular approach.  8 
Structures that are used in the O-DU and that are also proposed for consideration at the TN are defined in module o-ran-9 
cti-common.  10 
The specific structures for the O -DU are defined in module o-ran-o1-ctiOdu. Together, the modules o-ran-cti-11 
common and o-ran-o1-ctiOdu specify CTI for the O-DU side and fit into the wider O-DU YANG model. 12 
The specific structures for the TN are defined in module o-ran-cti-tn-generic, which contains no TN technology 13 
specific structures. Technology-specific structures are defined in the separate modules o-ran-cti-pon and o-ran-14 
cti-tn-docsis respectively. A TN then has to combine o-ran-cti-common, o-ran-cti-tn-generic and one tn-15 
technology module for a full description of CTI YANG, which in turn has to f it into the wider TN YANG model. 16
