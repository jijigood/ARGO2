1.3.1 Definitions 21
For the purposes of the present document, the terms and definitions given in 3GPP TR 21.905 [1] and the following apply. 22
A term defined in the present document takes precedence over the definition of the same term, if any, in 3GPP 23
TR 21.905 [1]. 24
E2 Node: A logical node terminating E2 interface. In this version of the specification, O-RAN nodes terminating E2 25
interface are: 26
• for NR access: O-CU-CP, O-CU-UP, O-DU or any combination as defined in [19], 27
• for E-UTRA access: O-eNB as defined in Section 4.3.7. 28
 29
Managed Application: The definition of Managed Application is given in O -RAN Operations and Maintenance 30
Architecture [14]. 31
Managed Element: The definition of a Managed Element (ME) is given in 3GPP TS 28.622 [4], Section 4.3.3.    32
Managed Function: The definition of a Managed Function (MF) is given in 3GPP TS 28.622 [4], Section 4.3.4.   33
Near-RT RIC: O-RAN Near-Real-Time RAN Intelligent Controller : A logical function that enables near -real-time 34
control and optimization of RAN elements and resources via fine -grained data collection and actions over E2 interface. 35
It may include AI/ML workflow including model training, inference and updates. Please refer to [19] for more information 36
Non-RT RIC: O-RAN Non-Real-Time RAN Intelligent Controller: A logical function within SMO that drives the content 37
carried across the A1 interface.  It is comprised of the Non-RT RIC Framework and the Non-RT RIC Applications (rApps) 38
whose functions are defined below.  Please refer to [17] for more information. 39
Non-RT RIC Applications (rApps): Modular applications that leverage the functionality exposed via the Non-RT RIC 40
Framework’s R1 interface to provide added value services relative to RAN operation , such as driving the A1 interface, 41
recommending values and actions that may be subsequently applied over the O1/O2 interface and generating “enrichment 42
information” for the use of other rApps.  The rApp functionality within the Non -RT RIC enables non-real-time control 43
and optimization of RAN elements and resources and policy-based guidance to the applications/features in Near-RT RIC.  44
Please refer to [18] for more information. 45

Non-RT RIC Framework: That functionality internal to the SMO that logically terminates the A1 interface to the Near-1

The Non-RT RIC Framework functionality within the Non-RT RIC provides AI/ML workflow including model training, 3

NMS: A Network Management System for the O- RU as specified in [21]  to support legacy Open Fronthaul M- Plane 5
deployments (prior to version 5 of [21]).  6
O-Cloud: O-Cloud is a cloud computing platform comprising a collection of physical infrastructure nodes that meet O-7
RAN requirements to host the relevant O-RAN functions (such as Near-RT RIC, O-CU-CP, O-CU-UP, and O-DU), the 8

appropriate management and orchestration functions. Please refer to [23] for more information. 10
O-CU-CP: O-RAN Central Unit – Control Plane: a logical node hosting the RRC and the control plane part of the PDCP 11
protocol. Please refer to Section 4.3.3 for more information. 12
O-CU-UP: O-RAN Central Unit – User Plane: a logical node hosting the user plane part of the PDCP protocol and the 13
SDAP protocol. Please refer to Section 4.3.4 for more information. 14
O-DU: O-RAN Distributed Unit: a logical node hosting RLC/MAC/High-PHY layers based on a lower layer functional 15
split. Please refer to Section 4.3.5 for more information. 16
O-eNB: An eNB [6] or ng-eNB [8] that supports E2 interface. Please refer to Section 4.3.7 for more information. 17
O-RU: O-RAN Radio Unit: a logical node hosting Low-PHY layer and RF processing based on a lower layer functional 18
split.  This is similar to 3GPP’s “TRP” or “RRH” but more specific in including the Low-PHY layer (FFT/iFFT, PRACH 19
extraction). Please refer to Section 4.3.6 for more information. 20
O1: Interface between SMO framework as specified in Section 4.3.1 and O-RAN managed elements, for operation and 21
management, by which FCAPS management, PNF (Physical Network Function) software management, File management 22
shall be achieved.  23
O2: Interface between SMO framework as specified in Section 4.3.1 and the O -Cloud for supporting O -RAN virtual 24
network functions. Please refer to [23] for more information. 25
SMO: A Service Management and Orchestration system as described in Section 4.3.1. 26
xApp: An application designed to run on the near -RT RIC. Such an application is likely to consist of one or more 27
microservices and at the point of on -boarding will identify which data it consumes and which data it provides. The 28
application is independent of the near-RT RIC and may be provided by any third party. The E2 enables a direct association 29
between the xApp and the RAN functionality [19]. 30

Chapter 2.  O-RAN Overview 1
2.1 Scope and Objectives 2
O-RAN activities are guided by the following objectives [13]: 3
• Leading the industry towards open, interoperable interfaces, RAN virtualization, and big data and AI enabled 4
RAN intelligence. 5
• Maximizing the use of common-off-the-shelf hardware and merchant silicon and minimizing proprietary 6
hardware 7
• Specifying APIs and interfaces, driving standards to adopt them as appropriate, and exploring open source 8
where appropriate 9
• The O-RAN Architecture identifies the key functions and interfaces adopted in O-RAN.  10

Chapter 4. O-RAN Architecture 1
4.1 Overall Architecture of O-RAN 2
Figure 4.1-1 below provides a high-level view of the O-RAN architecture.  It shows that the four key interfaces – 3
namely, A1, O1, Open Fronthaul M-plane and O2 – connect SMO (Service Management and Orchestration) framework 4
to O-RAN network functions and O-Cloud. Figure 4.1-1 below also illustrates that the O-RAN network functions can 5
be VNFs (Virtualized Network Function), i.e., VMs or Containers, sitting above the O-Cloud and/or PNFs (Physical 6
Network Function) utilizing customized hardware.  All O-RAN network functions are expected to support the O1 7
interface when interfacing the SMO framework.   8
The Open Fronthaul M-plane interface, between SMO and O-RU, is to support the O-RU management in hybrid mode, 9
as specified in [21]. The O-RU termination of the O1 interface towards the SMO as specified in [14] is under study.  10
The management architecture of the flat mode [14] and its relation to O1 interface for the O-RU is a subject for future 11
study.  12
 13
Figure 4.1-1: High Level Architecture of O-RAN 14
Within the logical architecture of O-RAN, as shown in Figure 4.1-2 below, the radio side includes Near-RT RIC, O-15
CU-CP, O-CU-UP, O-DU, and O-RU functions. The E2 interface connects O-eNB to Near-RT RIC.  Although not 16

4.3.1.2 Non-RT RIC  14
Non-Real Time RAN Intelligent Controller (Non-RT RIC) is the functionality internal to the SMO in O-RAN 15
architecture that provides the A1 interface to the Near-Real Time RIC.  16
The primary goal of Non-RT RIC is to support intelligent RAN optimization by providing policy-based guidance, ML 17
model management and enrichment information to the near-RT RIC function so that the RAN can optimize, e.g., RRM 18
under certain conditions [17]. It can also perform intelligent radio resource management function in non-real-time 19
interval (i.e., greater than 1 second).  20
Non-RT RIC can use data analytics and AI/ML training/inference to determine the RAN optimization actions for which 21
it can leverage SMO services such as data collection and provisioning services of the O-RAN nodes. 22
The Non-RT RIC is comprised of two sub-functions: 23
• Non-RT RIC Framework – Functionality internal to the SMO Framework that logically terminates the A1 24
interface and exposes the required services to rApps through its R1 interface. 25
• Non-RT RIC Applications (rApps) – Modular applications that leverage the functionality exposed by the Non-26
RT RIC Framework to perform RAN optimization and other functions via the A1, O1, O2 and R1 interfaces. 27
 28
The Non-RT RIC Framework is responsible for exposing all required functionality to the rApps, whether “inherent” to 29
the Non-RT RIC Framework, “inherent” to the SMO Framework, or “implementation variable” functionality.   30
