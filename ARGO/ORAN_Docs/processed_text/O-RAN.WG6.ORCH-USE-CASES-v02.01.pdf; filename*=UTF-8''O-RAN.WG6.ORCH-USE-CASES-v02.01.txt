Managed Function  Term used in O-RAN OAM to refer to a distinct logical function that is managed.  Examples 118
include Near-RT RIC, O-CU-CP, O-CU-UP, O-DU, and O-RU.     119
From the OAM Framework document:  3GPP TS 28.622 states that a Managed Function 120
(MF) can represent a telecommunication function either realized by software running on 121
dedicated hardware or realized by software running on NFVI. Each managed function 122
instance communicates with a manager (directly or indirectly) over one or more management 123
interfaces exposed via its containing managed element instance. 124
Managed Element  Term used in O-RAN OAM to refer to a distinct physical or virtual element that is managed.  125
The ME will expose its contained MFs and their information models to the SMO through its 126
management interfaces. 127
From the OAM Framework document:  3GPP TS 28.622 states that a Managed Element 128
(ME) communicates with a manager (directly or indirectly) over one or more management 129
interfaces for the purpose of being monitored and/or controlled. Managed elements may or 130
may not additionally perform element management functionality. 131

2.2 Objectives of this Document 182
The O-RAN ALLIANCE seeks to improve RAN flexibility and deployment velocity, while at the same time reducing 183
the capital and operating costs through the adoption of cloud-based architectures.  184
A key principle is the decoupling of RAN hardware and software for all compone nts including O-CU, O-DU, and O-185
RU, and the deployment of software components on commodity server architectures supplemented with programmable 186
accelerators where necessary. 187
Given that the RAN environment will consist of a range of different components that  can be realized by either PNFs or 188
VNFs/CNFs it is critical to understand the coordination of RAN components done by the Service Management and 189
Orchestration Framework to deploy and operate RAN services in an O-RAN architecture. 190
This document defines end-to-end use cases in clause 3 and specifies OAM-related general and interface requirements 191
in clause 4 to support the O-RAN architecture.  192

3.1 O-Cloud Basic Use Cases 230
Cloud Platforms must be identified for the SMO to take advantage of them. In traditional data center clouds these have 231
been manually or externally added to the SMO for homing of NF deployments  since the numbers are typically low from 232
hundreds of cloud regions to even several thousand. However, in the case of edge clouds the number of deployments 233
scale into the 10K to 100K range, in which case this process needs to more closely resemble the way RAN elements are 234
“discovered”. Even in this latter case, the rationale for edge clouds is for specific performance requirements, therefore 235
not all edge clouds are equivalent. This Use Case describes how the SMO discovers and manages a cloud 236
implementation that meets the O-RAN specifications as described in the Cloud Architecture and Deployment Scenarios 237
for O-RAN virtualized RAN [7]. Such a cloud instance is generally referred to as an O-Cloud which is comprised of O-238
Cloud Nodes (Compute, Storage, and Network), dedicated resources for the O-Cloud. 239
This group of Use Cases includes the following: 240
- Pre-processing where before the O-Cloud is activated the SMO is configured with the identity of the O-Cloud so 241
that registration can be done 242
- Basic platform software installation by the SMO and IMS  243
- O-Cloud deployment processing where the O-Cloud registers with SMO and has its identity and software version 244
checked 245
- Capabilities Exchange where the SMO either queries for O-Cloud capabilities or is notified autonomously that 246
there has been a change in O-Cloud capabilities 247
- O-Cloud hardware upgrade where a new O-Cloud Node is added  248
- O-Cloud software upgrade where new platform software is added to the O-Cloud  249
 250
The end result of the group of use cases is an active O-Cloud registered with SMO and ready for the deployment of 251
Network Functions. 252
NOTE: Where Personnel are shown as part of the Use Case, the roles and terminology used are given for 253
purposes of illustration only. 254

3.1.6 O-Cloud Platform Software Upgrade Use Case 326
3.1.6.1 High Level Description 327
Edge clouds deployments can range in the 10K to 100K range across an operator network. Rolling O -Cloud software 328
upgrades need to be performed without service disruption for both VM and container -based workloads. Across a region, 329
it may be necessary to upgrade all edge O-Clouds in a serial fashion in order to minimize potential disruptions or in a 330
parallel fashion where a set amount of edge O-Clouds in a region are simultaneously upgraded. Compute and storage 331
nodes of individual edge O-Clouds can also be upgraded in a serial or parallel hitless fashion. Within an individual O -332
Cloud, the IMS migrates Network Function deployments from the affected O-Cloud Nodes before the software upgrade 333
in order to avoid service disruption (in this use case, it is assumed that IMS does this autonomously).  From an SMO 334
perspective, offloading the upgrade strategy of an edge O-Cloud to a regional and local controller substantially reduces 335
the upgrade complexity. 336
Fault handling of an O-Cloud upgrade is critical. Individual nodes can fail during the upgrade process and may need to 337
be replaced. Rolling back of the upgrade prior to a successful completion must also be supported.  338
In this use case, we examine how the SMO will upgrade a single O-Cloud.  It is assumed that the software upgrade does 339
not affect the IMS itself, and that the IMS is capable of performing the software upgrade and migrating NF deployments 340
from the affected O-Cloud Nodes without affecting active services. 341

3.6 Recovery Use Cases 675
3.6.1 Introduction 676
When an operator experiences problems with an O-Cloud NF Deployment(s), O-Cloud Node(s), or O-Cloud instance, it 677
is possible for the operator to initiate a reset at different levels of the O-Cloud. These can include the following: 678
1. Application level reset 679
This causes an application within an NF on the O-Cloud to be reset on operator request, and is accomplished through 680
the O1 interface between SMO and the associated NF. Typically, the network operator tries to execute the Application 681
level reset as part of recovering from NF’s application failures via O1. 682
The Application level reset is out of scope of the present document since it is foreseen that the interactions are only on 683
the O1 interface. 684
2. NF Deployment level reset 685
Another level of reset is by acting upon the NF Deployments that realize an NF instance, where the network operator 686
can execute the NF Deployment level reset (refer to clause 3.6.2) via O2dms. This causes an NF Deployment within an 687
O-Cloud to be reset on operator request, and is accomplished through the O2dms interface between SMO and the DMS 688
responsible for the deployment management services of the NF Deployment of concern.  689
3. O-Cloud Node level reset 690
This causes one or more O-Cloud Nodes within an O-Cloud to be reset on operator request, and is accomplished 691
through the O2ims interface between SMO and the IMS responsible for infrastructure management services for that O -692
Cloud Node. This can be used as part of recovery of the resources within the O-Cloud, i.e., the O-Cloud Node level 693
reset (refer to clause 3.6.3). This level of reset needs extra trigger considerations because it can affect multiple NF’s 694
supported by the O-Cloud Node or Nodes. 695
4. O-Cloud instance level reset 696
This causes an O-Cloud instance to be reset on operator request.  This last level of reset, i.e., O-Cloud instance level 697
reset, can be seen as a last resort action and it needs extra trigger considerations because it can affect  multiple NFs and 698
O-Cloud Nodes residing in the O-Cloud instance. 699
Editor’s Note: It is for further study whether this can be accomplished through the O2 interface. 700
The ordering or which one of the resets to use or execute is up to network operator policies, e.g. in some cases a 701
cascading might be tried, while in others cases the operator might decide to activate a specific reset without cascading 702
among them. 703
For illustrative purposes, the following is explanation about typical system reset cases in legacy deployments. In legacy 704
deployment cases, when the xNB experiments some issue at the software level, e.g. the software on the xNB is not 705
responding to the operations of the remote management tool due to possible software failures, “physical hardware” level 706
reset is performed to re-gain software level control. For performing this action, typically an OSS sends a “system reset” 707
signal to the target xNB. This procedure is necessary to reset the system (CU and DU) in xNB without performing a 708
“manual intervention” at the radio antenna site, e.g., “personnel pushing power button”. Figure x-x illustrates an 709
example of remote system reset in the legacy deployment case.  System reset does not apply in exactly the same way for 710
recovery of NF and O-Cloud systems but reset at different levels can be supported using O2 as described above. 711

3.7.3.2 Sequence Description 1001
Use Case Stage Evolution / Specification <<Uses>>
Related use
Goal
This Use Case describes the query for alarm(s) from resources. The SMO
queries for specific kinds of alarms or groups of alarms through query criteria.
Alarms are managed by the O-Cloud and this use case allows a user (SMO) to
request for a set of alarms based on the query criteria.
The alarm query criteria specify the alarm characteristics which must be
matched by IMS alarms to be returned as an alarm query results.
In the IMS case, A fault query is a message which is expecting a response
message from the IMS which matches the query criteria. FOCOM (SMO)
queries the IMS for alarms over O2ims.
Alarm queries within certain periods of time can be used for reconciliation.
Combinations of alarm queries can be used for analysis.
An “alarm status” for all the O-Cloud resources can be accomplished by an
alarm query with a criterion of “all O-Cloud resources” and “all alarms” (on
those resources).

Actors and Roles
SMO – The FOCOM within the SMO is used in this use case to represent any
consumer of the query service.
IMS – The IMS within the O-Cloud is used to represent any provider of the query
service.

Assumptions
The NFO (SMO) can query for alarms WITHOUT an alarm subscription to the
IMS.
There is no relationship between an alarm subscription’s filter criteria and an
alarm query’s criteria.

Preconditions SMO has connectivity to IMS/DMS.
Begins when  This use case begins when the cloud operator at the SMO (subscriber) initiates
an Alarm Query.
Step 1.1 (O)
(Alt) ALARM QUERY  INITIATED FROM OPERATOR  – An alarm  query is
initiated from the Operator to the FOCOM with query criteria that specifies what
type of alarms the operator is interested in.

Step 1.2 (O)
(Alt) ALARM QUERY INITIATED FROM SMO – An alarm query may also be
initiated from the FOCOM (SMO) with query criteria that specifies what type of
alarms to retrieve from the IMS.

Step 2 (M) ALARM QUERY FROM FOCOM TO IMS – An alarm query is processed by the
FOCOM (SMO) towards the IMS in the O-Cloud with the alarm query criteria.
Step 3 (M)
(ALT) SUCCESSFUL ALARM RESPONSE – The response is returned in
response for the Alarm Query with alarm content matching the alarm query
criteria.
Note: An alarm query on an O-Cloud resource with no alarms will return a
successful alarm response with a null set of alarms.
Note: To get historical alarms the cloud operator would just perform an alarm
query with the appropriate criteria. For example, an alarm query with the
criteria to request for all alarms with cleared status would return historical
alarms based on the retention policy of the alarm list.

Step 4 (M)
(ALT) ALARM OBJECT NOT FOUND – This is an alt step. This is an
exception case for the Alarm Query basic case. In this situation, the alarm that
was to be queried was not found. A “alarm not found” response is sent to the
subscriber. For example, an alarm query was performed on a non-existent O-
Cloud resource.

Step 5 (M)
(ALT) ALARM NO CONTENT  – This is an alt step. This is an exception case
for the Alarm Query basic case. In this situation, the query parameters returned
no content (an empty set). A “no content” response is sent to the subscriber.
For example, if the given criteria [( all severity = major ) and ( all severity = not
major)] would never return any alarms.

discovered and what version of software is required for O-Cloud Nodes in each pool.
cloud blueprint This is information about the functionality and resources needed in an O-Cloud build
