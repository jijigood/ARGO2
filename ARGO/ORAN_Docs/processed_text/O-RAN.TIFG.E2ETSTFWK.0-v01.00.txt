Chapter 2 Introduction 1 
This document defines the End to End (E2E) testing framework for comprehensive E2E testing and validation of the Open 2 
RAN (O-RAN) Systems ensuring robust interoperability and operate with high performance as intended in real -world 3 
deployments.  4 
Figure 2-1 shows the focus on the O -RAN System as the System under Test (SUT) in the context of the E2E testing 5 
framework specified in this document and in relation to the E2E KPIs defined between the UEs and Services which will 6 
be further elaborated in this specification. Refer to [3] and Section 3.1 in this specification for more details on the O-RAN 7 
System.   8 
 9 
Figure 2-1: Focus on O-RAN as the System under Test (SUT) 10 
Mobile operators will likely plan for many different variants of O-RAN Systems depending on their strategies and use 11 
cases which they may be interested to test, deploy and evolve their O -RAN compliant networks over time . The E2E 12 
testing framework will therefore need to take into consideration the Operators’ O -RAN deployment variants given its 13 
influence on the scope of E2E testing.  14 
The concept of O -RAN Deployment Blueprint is introduced in this specification  and is used to describe the different 15 
deployment variants of O-RAN Systems for E2E system testability purposes. This document provides the definition of 16 
the O -RAN Deployment Blueprint template which contains the necessary descriptors to build O -RAN Deployment 17 
Blueprints. 18 
A set of typical O -RAN Deployment Blueprints are defined using the most common deployment scenarios abstracted 19 
from various surveys which O -RAN Alliance has gather ed from global operators. O -RAN Test and Integration Focus 20 
Group (TIFG) focuses on the definition of the E2E testing framework based on addressing the needs of these most 21 
common and probable deployment scenarios. 22 
Operators and O -RAN Software Community (OSC) have submitted additional detailed inputs on their O -RAN 23 
Deployment Blueprints and testing practices respectively, to assist O-RAN Alliance with developing comprehensive E2E 24 
testing methodologies and test cases. These inputs are included in this specification for referencing purposes. 25 
Chapter 3  starts with a brief introduction of the O -RAN logical architecture, O -RAN network functions and open 26 
interfaces between the O-RAN network functions. These O-RAN network functions and open interfaces are mapped to a 27 
list of the most considered use cases driven O-RAN cloud deployment scenarios. 28 
Chapter 4 defines the O-RAN Deployment Blueprint, Template and its associated Profiles which are used to describe 29 
the O-RAN Deployment Blueprint. The O -RAN network functions and open interfaces are mapped to the various 30 
descriptors of the O-RAN Deployment Blueprint. Detailed definition of the various profiles’ templates will be considered 31 
in the future releases of this specification. 32 
Chapter 5 details the O-RAN E2E Testing Framework by providing guidance on the Testing Methodology for holistic 33 
testing and evaluation of the O -RAN Deployment Blueprints ensuring that these Blueprints can be robustly tested and 34 
optimized, ensuring consistent high quality and performance in a deterministic, repeatable and reproducible manner across 35 
the entire technology lifecycle. 36 
Chapter 6 provides guidance on the key considerations along with information to help guide testing and validating the 37 
O-RAN Deployment Blueprints with the relevant E2E Test Cases and KPIs. Recommendations on the specific E2E Test 38 
Cases and KPIs will be considered in the future releases of this specification. 39

Chapter 3 O-RAN Architecture and Deployment Scenarios 1 
This chapter gives an overview of the O-RAN Logical Architecture, O-RAN network functions, open interfaces and a list 2 
of the most considered use cases driven O-RAN Deployment Scenarios which show how the O-RAN network functions 3 
are mapped to physical implementations in terms of how these network functions are realized as physical, virtual or cloud 4 
native network functions, aggregation/dis-aggregation of these network functions and where these network functions can 5 
be deployed in the network (e.g. Cell Site, Edge Cloud, Regional Cloud).  6 
The information provided in this Chapter facilitates definition of the O-RAN Deployment Blueprint in Chapter 4. The O-7 
RAN Deployment Blueprint is used to further describe the O-RAN Deployment at the E2E System and Subsystem levels, 8 
and the Open Interfaces between the specified Subsystems. This would then allow definition and documentation of the 9 
testing methodology and the test cases using the Blueprint specification.  10

The O-RAN Deployment Blueprint is defined as the set of inputs used to describe a specific O -RAN deployment from 5 
several aspects. These aspects include the specification of the O-RAN deployment at the System level (e.g. architecture, 6 
performance metrics) as well as Subsystem level and the Open Interfaces between the specified Subsystems. This would 7 
then allow definition and documentation of the testing methodology and the test cases using the blueprint specification. 8

One of the key objectives of Subsystem testing is to validate the functionality of production grade DUT when the DUT 23 
is not operating in test mode and in few cases in standardized test mode as when required. One example of standardized 24 
test mode can be 3GPP test modes.  25

support Subsystem testing, i.e., DUT is not expec ted to be testing tools when deployed in production networks and 27 
therefore DUT should not be used as testing tool during Subsystem tests. 28 
Subsystem tests are performed with a set of testing tools which are used to both apply active stimulus and as well as 29 
passive monitoring and measurements of the DUTs. 30

under Test (DUT) and Interfaces under Test (IUT), therefore providing a testing framework for comprehensive evaluation 32 
and testing prior to Multi-vendors Interoperability (MV-IOT) Subsystem pairing and Open interfaces testing. 33

surrounding the Subsystem (DUT) as per defined in the O-RAN Deployment Blueprint. Reference test platforms can be 35 
Subsystems designed for commercial purpose or test applications which are used to support the wraparound test setup. 36

O-RAN network function designed to operate on the O-Cloud Subsystem. 38

The O-RAN E2E testing framework aims to evaluate the entire O-RAN Deployment Blueprint holistically as a System 3 
under Test (SUT) based on a set of well-defined E2E Test Cases and Key Performance Indicators (KPIs). 4 
The purpose of this Chapter is to outline the key considerations along with information to help guide testing and validating 5 
the O-RAN Deployment Blueprints with the relevant E2E Test Cases and KPIs. 6

6.2  Use Cases and Services Requirements 15

1. Use Cases and Services 17 
2. O-RAN Deployment Blueprint which are optimized according to the Use Cases and Services requirements 18

6.3  O-RAN Deployment Blueprint dependencies 2 
An O-RAN Deployment Blueprint can consist of multiple Subsystems from possibly multiple vendors. The KPIs can be 3 
measured as E2E KPIs across any combinations of vendors and as well as Subsystems and Subsystems Pairs KPIs for the 4 
different Subsystems and the Open interfaces between the Subsystems respectively.  5 
The specific Performance Measurements, KPIs and their respective values defined for Subsystem Testing, Subsystem 6 
Pair and Open Interfaces Testing and E2E System Testing will be dependent on the O-RAN Deployment Blueprints.  7 
For example, from the E2E System Testing perspective, the beam forming system performance gain KPIs specified for 8 
outdoors macro cells would not be applicable for indoors small cells which may not require beam forming capabilities. 9 
Similarly, the beam forming performance gain measurements specified for the O-RUs which are designed and optimized 10 
for outdoors macro cells systems would not be applicable for O-RUs which are designed and optimized for indoors small 11 
cells.  12 
It is therefore important to ensure that testing is performed for each of the 13 
1. Subsystem Instances with their defined Subsystem Instance profiles and associated KPIs 14 
2. Subsystem Instances pairs with their defined Subsystem Pairs Interoperability profiles and associated KPIs 15 
3. System Deployment Blueprints with the defined E2E System KPIs 16

6.4  Relevance of the 3GPP KPIs 18 
As highlighted in Section 6.2, operators will require validation of the O -RAN Deployment Blueprints that the overall 19 
3GPP System characteristics is as expected. 20 
In general, 3GPP defines two categories of RAN KPIs  21 
1. Performance data for technology tests, trials, and evaluation such as in 3GPP TR 38.913. 22 
2. Performance data for network monitoring, assessment, analysis, optimization, and assurance which will be 23 
further discussed in Section 6.4.1. 24 
 25 
The same set of E2E KPI categories and KPIs which are defined for 3GPP compl iant networks and services will be 26 
directly applicable for O-RAN compliant networks.  27 
Additional considerations for the application of 3GPP defined KPIs for O-RAN Deployment Blueprint testing and 28 
verification will be discussed in Section 6.4.2. 29 
6.4.1  3GPP KPIs defined for network monitoring, assessment, analysis, 30 
optimization and assurance 31 
From the 3GPP system perspective, the E2E KPIs defined for network monitoring, optimization and assurance are 32 
typically defined for the following categories  33 
1. Availability (e.g. cells ready to service connections) 34 
2. Accessibility (e.g. connection establishment rates, attempts/failures, latency) 35 
3. Integrity (e.g. throughput, latency, jitter, retransmission rates) 36 
4. Mobility (e.g. intra/inter RATs handover rates, interruption times for control / user planes) 37 
5. Reliability (e.g. amount of sent packets which are successfully delivered to the destination within the time 38 
constraint required by the target service, divided by the total number of sent packets) 39 
6. Retainability (e.g. drop rates for PDN/PDU sessions, call session) 40 
7. Utilization (e.g. resources utilization based on certain traffic conditions and profiles such as fronthaul, compute) 41 
8. Energy Efficiency (e.g. power consumption based on certain traffic conditions and profiles) 42

6.6.2  Functional, Performance including normal and abnormal conditions  3 
Multi-vendors interoperability testing typically includes functional validation for handling normal conditions.  4 
However, similar to how the 3GPP Radio Access Networks are expected to be tested and optimized for Performance and 5 
Abnormal conditions handling, operators would be expecting the O -RAN Deployment Blueprints to be tested and 6 
optimized in the same manner.  7 
O-RAN Deployment Blueprints consisting of Subsystems from multiple vendors will therefore require Multi -vendors 8 
interoperability testing to include testing for Performance and Abnormal conditions handling.  9 
Performance testing can be used to evaluate how well each individual Subsystem, each pair of Subsystem Instances and 10 
all the Subsystems in the O-RAN Deployment Blueprint inter-operate to achieve the best in class cost-performance ratio.  11 
This can include quantifying the O-RAN Deployment Blueprint when Subsystems from multiple vendors inter -operate 12 
under Performance test conditions through  13 
1. Performing testing using the same set of KPIs, for example examining if the latency figures can be achieved in 14 
performance testing conditions. 15 
2. Measuring resources utilization efficiency for example with Radio resource blocks, Open Fronthaul and xHaul 16 
transport, O-Cloud computing / networking / storage and power consumption / energy efficiency. 17

Abnormal testing will in some cases be required to be performed to validate fault management and zero -touch network 19 
automation techniques such as self-healing functions.  20 
6.6.3  Radio channel variations and impacts on the overall system 21 
performance and QoE  22 
Radio access networks should be tested with close to real -world radio channel conditions, variations of the channel 23 
conditions and mobility patterns in the lab environment for a proper evaluation of the overall system performance of the 24 
RAN and QoE of the services handled by the RAN. 25 
For example testing key 3GPP RAN functions such as Radio resource, Beam / Cell level mobility management and 26 
advanced use cases defined by O -RAN such as Traffic Steering and Massive MIMO Beamforming Optimization will 27 
require the test environment to be setup with the appropriate network and slices topology, radio channel conditions, UEs 28 
distributions and mobility patterns, which otherwise these functions and use cases cannot be properly tested and evaluated.  29 
Testing O-RAN Deployment Blueprints consisting of Subsystems from multiple vendors  under the appropriate radio 30 
channel conditions and mobility patterns will be required to validate how well the multi-vendors Subsystems inter-operate 31 
under these test conditions.  32 
6.6.4  Management, Orchestration and Network Automation  33 
In general, operators require the Management & Orchestration (MANO) and Network Automation capabilities of their 34 
networks to be robustly validated.   35 
In order for operators to be able to deploy O -RAN networks and network slices which can consist of Subsystems from 36 
multiple vendors, multi-vendors interoperability testing will be required for validating the management and orchestration 37 
functions of all the Subsystems defined in the O-RAN Deployment Blueprints.  38 
O-RAN Alliance has specified a set of prioritized use cases for O-RAN orchestration of virtualized RAN and the interfaces 39 
used for management and orchestration, in particular the O1 interface between the service management and orchestration 40 
framework and the RAN managed functions and the O2 interface between the service management and orchestration 41 
framework and the O -Cloud Infrastructure Management Services/Deployment Management Services that controls 42 
resource assignment for Virtualized Network Functions.  43

6.6.5  Additional considerations related to practical deployment aspects  4 
In addition to performing testing based on 3GPP, ITU -T, IEEE specified functionalities, it is important as well to take 5 
into considerations the practical deployment aspects which operators will need to consider and therefore tested as part of 6 
the E2E Testing framework. These considerations include 7 
1. Performance Evaluation and Monitoring 8 
2. Robustness, Resilience, Reliability, Security  9 
3. Troubleshooting and Root Cause Analysis (RCA) 10 
4. Practical Network Operating Conditions such as fronthaul transport loading and non-zero variable delays 11 
5. FCAPS including fault, configuration, accounting, performance, and security management 12 
6. System Deployment lifecycle (installs, upgrades)  13 
7. Virtualization, interaction with O-Cloud (NFVi), Management and Orchestration (MANO) 14 
8. Evolution towards Zero Touch Automation network management  15 
 16 
System Deployment lifecycle (installs, upgrades) involves multiple phases of the System Deployment lifecycle which 17 
will include new installs and as well as future field upgrades scenarios such as  18 
1. Performing software or firmware upgrades of the existing installed O-RAN Network Functions (e.g. increased 19 
performance, enhanced capabilities such as dynamic spectrum sharing, bug fixes and others) 20 
2. Installing new O -RAN Network Functions which are required to interoperate with existing installed O -RAN 21 
Network Functions (e.g. adding new O-RUs from a new vendor to the field operational O-DU currently serving 22 
O-RUs from the current vendor, new xApp to be installed onto the current near-RT RIC) 23 
3. Installing new or Replacing existing hardware (or O -Cloud) to support existi ng installed O -RAN Network 24 
Functions (e.g. adding, replacing storage, networking, computing resources) 25 
 26 
It is therefore important to define testing templates ensuring that the essential testing is performed for the specific 27 
Subsystem Instances, Subsystem I nstances Pairs and Open Interfaces and System Deployment Blueprint which are 28 
affected by the System Deployment lifecycle scenarios.   29

Annex B: O-RAN Software Community inputs on Testing 1 
Methodology  2 
This annex provides an overview on the O-RAN Software Community (OSC) work and its current software releases and 3 
testing practices. Refer to the OSC wiki site for the most current information (https://wiki.o-ran-sc.org/).  4 
The O-CU referred in this annex should be considered as O-CU CP/UP. 5 
B.1 O-RAN Software Community 6 
The O-RAN Software Community (OSC) is an open source software development project aimed at providing an open 7 
source implementation of an end -to-end RAN system following O-RAN Alliance architecture and specifications.  The 8 
OSC is hosted by the Linux Foundation and funded by the O-RAN Alliance.  9 
The OSC is led by a 12 -member Technical Oversight Committee (TOC) with its current members representing AT&T, 10 
China Mobile, Deutsche Telekom, Ericsson, Nokia, NTT DOCOMO, Orange, Radisys and Verizon.   11 
The working effort of the OSC is organized into the following development projects: 12 
• Operations and Maintenance (OAM) 13 
• Non-Realtime RAN Intelligent Controller (NONRTRIC) 14 
• Near-Realtime RAN Intelligent Controller Platform (RIC) 15 
• Near-Realtime RAN Intelligent Controller Applications (RICAPP) 16 
• O-RAN Central Unit (OCU) 17 
• O-RAN Distributed Unit High (ODUHIGH) 18 
• O-RAN Distributed Unit LOW (ODULOW) 19 
• Simulations (SIM) 20 
• Infrastructure (INF) 21 
• Documentation (DOC) 22 
• Integration and Testing (INT)  23 
and subcommittee(s) 24 
• Requirements and Software Architecture Committee (RSAC) 25 
 26 
The OSC conducts its business on open communication and development platforms including Zoom teleconferencing 27 
bridge, Groups.io mailing lists, Confluence wiki, and Git/Gerrit source code repositories.  All O -RAN SC information 28 
and O-RAN SC developed software can be accessed via resources listed in Annex B.6. 29 
B.2 OSC Development Cycles 30 
The OSC developed software is released on a bi -annual schedule, first in May/June time frame and the second in 31 
November/December.   The naming scheme of releases follows color names in alphabetical order, with individual release 32 
names selected by community member voting.  The first four releases are named Amber, Bronze, Cherry, and Dawn.  The 33 
OSC delivered its first release, Amber release, in November 2019.   34 
The high-level objectives and use cases for the next release are usually selected by the RSAC during the later portion of 35 
the current release cycle.  The first several weeks of each release cycle typically are for the design and planning phase 36 
where the high-level release objectives are designed into implementable features and tasks, which are distributed to the 37 
OSC development projects.  Following the planning phase is the development phase, during which the development 38 
projects and teams implement the agreed-upon features.  This phase is divided into three-week sprints where each sprint 39 
has its own more manageable implementation and testing goals.  Finally, prior to the release of the software, it is the 40 
testing phase, during which the integrated system testing and end-to-end use case testing are conducted, and the viability 41 
and quality of the software are verified and evaluated.   42

B.5 OSC Testing 25 
OSC testing effort is divided into five levels 26 
1. Unit testing (UT) 27 
o UT is performed by the developers by implementing test cases using programming language specific 28 
unit testing and static code analysis framework.   29 


 
________________________________________________________________________________________________ 
Copyright © 2020 by the O-RAN Alliance. 
Your use is subject to the terms of the O-RAN Adopter License Agreement in Annex ZZZ          57 
                                                                                          O-RAN.TIFG.E2ETSTFWK.0-v01.00 
o UT is triggered automatically every time when there is source code change submitted into the code 1 
repository for review. 2 
o At OSC level the INT project team provides support for unit test and code scan (i.e. static code analysis) 3 
integration into Linux Foundation Continuous Integration flow, and result reporting into the SONAR 4 
Cloud platform.  Using the reports, TOC sets quality gates such as percentage of code covered by unit 5 
test cases, no major issues discovered, etc. 6 
2. Project level integrated testing (PIT) 7 
o Each development project is expected to conduct project level integrated testing, where all software 8 
components developed by the project are deployed into a single system for testing internal flows. 9 
o PIT is expected to be conducted by individual project teams, at the end of each development sprints or 10 
other development milestones. 11 
o At OSC level the INT project team organizes sprint demonstration fests regularly where development 12 
project teams gather and demonstrate achieved features in the previous sprint, using project level system 13 
deployment in project team’s own testing and demonstration environment.   14 
o The INT project team is also working with individual development projects on Robot testing 15 
framework-based testing flows that are to be integrated into Linux Foundation Continuous Integration 16 
flow.  Such test flows serve as regression tests for implementation upda tes and can be triggered 17 
automatically by the CI system on a regular basis (e.g. daily) or upon new code submission. 18 
3. Project pairwise testing (PPT) 19 
o For projects that deliver O -RAN architectural components that interact with each other, PPT is 20 
conducted to ensure communication compatibility. 21 
o PPT sessions are organized by the INT project team, using either community testing lab or self-arrange 22 
testing facilities by involved projects. 23 
4. System integrated testing (SIT) 24 
o SIT is whole OSC software testing where all OSC development components are deployed into a single 25 
system.  SIT focuses on overall system health, inter -project communication and application 26 
compatibility, overall system deployment flow, and resource requirements.  27 
o SIT is conducted by the INT project team using OSC community testing lab. 28 
o SIT is conducted during the testing phase of each release cycle. 29 
o Robot test framework-based health check test flows used as automated regression test. 30 
5. Use case testing (UCT) 31 
o UCT is whole OSC software testing where all OSC development components are deployed into a single 32 
system.  UCT tests OSC deployment that has passed SIT. 33 
o UCT is conducted by the INT project team using OSC community testing lab. 34 
o UCT is conducted during the testing phase of each release cycle. 35 
o UCT uses Open Testing Framework (OTF), a test flow definition and orchestration tool that is 36 
developed as a sub-project under the INT. 37
