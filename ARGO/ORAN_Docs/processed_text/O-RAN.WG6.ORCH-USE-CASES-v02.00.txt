2.2 Objectives of this Document 158 
The O-RAN ALLIANCE seeks to improve RAN flexibility and deployment velocity, while at the same time reducing 159 
the capital and operating costs through the adoption of cloud-based architectures.  160 
A key principle is the decoupling of RAN hardware and software for all components  including O-CU, O-DU, and O-161 
RU, and the deployment of software components on commodity server architectures supplemented with programmable 162 
accelerators where necessary. 163 
Given that the RAN environment will consist of a range of different components that ca n be realized by either PNFs or 164 
VNFs/CNFs it is critical to understand the coordination of RAN components done by the Service Management and 165 
Orchestration Framework to deploy and operate RAN services in an O-RAN architecture. 166 
This document defines end-to-end use cases in clause 3 and specifies OAM-related general and interface requirements 167 
in clause 4 to support the O-RAN architecture.  168

3.1 O-Cloud Basic Use Cases 205 
Cloud Platforms must be identified for the SMO to take advantage of them. In traditional data center  clouds these have 206 
been manually or externally added to the SMO for homing of NF deployments since the numbers are typically low from 207 
hundreds of cloud regions to even several thousand. However, in the case of edge clouds the number of deployments 208 
scale into the 10K to 100K range, in which case this process needs to more closely resemble the way RAN elements are 209 
“discovered”. Even in this latter case, the rationale for edge clouds is for specific performance requirements, therefore 210 
not all edge clouds are equivalent. This Use Case describes how the SMO discovers and manages a cloud 211 
implementation that meets the O-RAN specifications as described in the Cloud Architecture and Deployment Scenarios 212 
for O-RAN virtualized RAN [7]. Such a cloud instance is generally referred to as an O-Cloud which is comprised of O-213 
Cloud Nodes (Compute, Storage, and Network), dedicated resources for the O-Cloud. 214 
This group of Use Cases includes the following: 215 
- Pre-processing where before the O-Cloud is activated the SMO is configured with the identity of the O-Cloud so 216 
that registration can be done 217 
- Basic platform software installation by the SMO and IMS  218 
- O-Cloud deployment processing where the O-Cloud registers with SMO and has its identity and software version 219 
checked 220 
- Capabilities Exchange where the SMO either queries for O-Cloud capabilities or is notified autonomously that 221 
there has been a change in O-Cloud capabilities 222 
- O-Cloud hardware upgrade where a new O-Cloud Node is added  223 
- O-Cloud software upgrade where new platform software is added to the O-Cloud  224 
 225 
The end result of the group of use cases is an active O-Cloud registered with SMO and ready for the deployment of 226 
Network Functions. 227 
NOTE: Where Personnel are shown as part of the Use Case, the roles and terminology used are given for 228 
purposes of illustration only. 229

3.1.2 O-Cloud Platform Software Installation Use Case 246 
3.1.2.1 High Level Description 247 
The Use Case describes the deployment of the O-Cloud and installation of O-Cloud Platform Software on an O-Cloud 248 
Node.  The Cloud Installer begins the installation by installing the hardware for the O-Cloud Nodes and notifying SMO 249 
to begin Cloud Build.  SMO then loads O-Cloud Platform Software on a first O-Cloud Node in the O-Cloud, which then 250 
brings up any additional O-Cloud Nodes in the O-Cloud.   251 
During activation of other O-Cloud Nodes, the O-Cloud Management Plane activates IMS, which then notifies the 252 
SMO of its availability.  The SMO then sends IMS the required set of O-Cloud Node Roles and Personalities to support.  253 
The IMS loads necessary basic software for O-Cloud onto the other O-Cloud Nodes and also configures and activates 254 
one or more DMSs. 255 
It is assumed that hardware to deploy O-Cloud Management Software is pre-configured and ready. Network 256 
connectivity between O-Cloud network gateways(s) and SMO is assumed to be ready, as per the "O-Cloud Pre-257 
Deployment Processing Use Case" described in clause 3.1.1. 258

3.1.6 O-Cloud Platform Software Upgrade Use Case 301 
3.1.6.1 High Level Description 302 
Edge clouds deployments can range in the 10K to 100K range across an operator network. Rolling O-Cloud software 303 
upgrades need to be performed without service disruption for both VM and container-based workloads. Across a region, 304 
it may be necessary to upgrade all edge O-Clouds in a serial fashion in order to minimize potential disruptions or in a 305 
parallel fashion where a set amount of edge O-Clouds in a region are simultaneously upgraded. Compute and storage 306 
nodes of individual edge O-Clouds can also be upgraded in a serial or parallel hitless fashion. Within an individual O -307 
Cloud, the IMS migrates Network Function deployments from the affected O-Cloud Nodes before the software upgrade 308 
in order to avoid service disruption (in this use case, it is assumed that IMS does this autonomously).  From an SMO 309 
perspective, offloading the upgrade strategy of an edge O-Cloud to a regional and local controller substantially reduces 310 
the upgrade complexity. 311 
Fault handling of an O-Cloud upgrade is critical. Individual nodes can fail during the upgrade process and may need to 312 
be replaced. Rolling back of the upgrade prior to a successful completion must also be supported. 313 
In this use case, we examine how the SMO will upgrade a single O-Cloud.  It is assumed that the software upgrade does 314 
not affect the IMS itself, and that the IMS is capable of performing the software upgr ade and migrating NF deployments 315 
from the affected O-Cloud Nodes without affecting active services. 316

3.5 Reconfiguration of O-RAN Virtual Network Function(s) 410 
3.5.1 High Level Description 411 
A common orchestration request is the reconfiguration of one or a set of cloud -based O-RAN Virtual Network 412 
Functions in order to affect the behavior of the RAN.  The action taken could be, for example, to re-optimize network 413 
traffic patterns or update the load balancing of traffic across the RAN network.  The trigger for this action could be any 414 
number of events such as non-real time traffic analysis by the Non-RT RIC, or input from non-RAN sources such as 415 
upcoming natural events or scheduled human events.  416 
In such cases the SMO will either be told or determine by its own analysis which Virtual Network Functions need 417 
reconfiguration and in which order this is done if multiple Virtual Network Functions are involved.  If a problem 418 
develops during reconfiguration then SMO may optionally invoke a fallback procedure to revert the Virtual Network 419 
Functions to their original configuration.  As in reconfiguration of PNFs, O1 Interface [6] Provisioning Management is 420 
used. 421 
NOTE: There may be a need for SMO to initially reroute traffic away from the affected Virtual Network 422 
Functions and subsequently revert traffic to the original patterns.  This could be done by using Network 423 
Function Reconfiguration on adjacent Network Functions that send or receive information from the 424 
affected Network Functions. 425
