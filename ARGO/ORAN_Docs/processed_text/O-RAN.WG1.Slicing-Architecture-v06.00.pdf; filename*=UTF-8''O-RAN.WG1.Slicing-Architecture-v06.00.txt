Chapter 2.  Slicing Overview 1
Network Slicing is expected to play a critical role in 5G networks because of various use cases a nd services 5G will 2
support. It allows a network operator to provide services tailored to customers’ requirements. Network slice is defined as 3
a logical network with a bundle of specified network services over a common network infrastructure.  A single phy sical 4
network is sliced into multiple virtual networks that can support different service types over a single RAN. 3GPP has 5
standardized 4 different service types: eMBB, URLLC, MIoT and V2X  [3].  6
3GPP defined 5G architecture and procedures containing network slicing and related concepts in Release 15. Furthermore, 7
management and orchestration of 5G networks featuring slicing was defined in the 3GPP specifications. Othe r standard 8
groups e.g. GSMA, ETSI NFV-MANO, ETSI ZSM and ONAP focus on the different aspects of network slicing. Further 9
information regarding network slicing and other SDO’s contributions was discussed in the Study on O-RAN Slicing 10
Technical Report [24]. 11
A sample RAN slicing deployment of O-RAN network functions based on the select initial deployment option, option B 12
as described in [20], is shown in Figure 2-1, with some of the network functions shared between RAN slice subnets (such 13
as O-CU-CP, O-DU, O-RU) and some network functions dedicated to a particular RAN slice subnet (such as O-CU-UP). 14

Chapter 3.  High-Level O-RAN Slicing Use Cases 1
This section contains high-level O-RAN slicing use cases that O -RAN is expected to support. Slicing requirements will 2
include the requirements derived from the specified use cases.   Additional use cases will be added as prioritized by the 3
O-RAN community in future versions of this document. It should be noted that not all of the use cases presented here are 4
currently supported by O-RAN specifications and these use cases will be addressed in future O-RAN work. 5
3.1 O-RAN Slice Subnet Management and Provisioning Use 6
Cases 7
Network slicing is conceived to be an end -to-end feature that includes the core network, the transport network and the 8
RAN. Although 3GPP has started defining network slicing support with Release 15, slicing in O-RAN needs to be further 9
addressed in line with 3GPP to achieve deployable network slicing in an open RAN environment.   10
Management aspects of network slicing such as Network Slice Instances (NSI) and Network Slice Subnet Instances 11
(NSSI) are defined by 3GPP. While NSI refers to an end -to-end network slice, NSSI refers to a part of it such as a RAN 12
slice subnet. The provisioning of network slicing includes the four phases which are preparation, commissioning, 13
operation and decommissioning. The NSI/NSSI provisioning operations include:  14
- Create an NSI/NSSI; 15
- Activate an NSI/NSSI; 16
- De-active an NSI/NSSI; 17
- Modify an NSI/NSSI; 18
- Terminate an NSI/NSSI. 19
Further details of NSI and NSSI lifecycle management and provisioning can be found 3GPP TS28.531 [6], Section 4.1. 20
This chapter provides the use cases and procedures necessary for O -RAN Slice Subnet Management that is in -line with 21
3GPP Slice Management framework. For this purpose, use cases such as sli ce subnet creation, activation, modification, 22
deactivation, termination, configuration and feasibility check are considered for O-RAN architecture. 23

3.2 O-RAN Slicing Use Cases 1
3.2.1 Use Case 1: RAN Slice SLA Assurance 2
In the 5G era, network slicing is a prominent feature which provides end-to-end connectivity and data processing tailored 3
to specific business requirements. These requirements include customizable network capabilities such as the support of 4
very high data rates, traffic densities, service availability and very low latency. According to 5G standardization efforts, 5
the 5G system should support the needs of the business through the specification of several service needs such as data 6
rate, traffic capacity, user density, latency, reliability, and availability. These capabilities are always provided based on a 7
Service Level Agreement (SLA) between the mobile operator and the business customer, which brought up interest for 8
mechanisms to ensure slice SLAs and prev ent its possible violations. O -RAN’s open interfaces and AI/ML based 9
architecture will enable such challenging mechanisms to be implemented and help pave the way for operators to realize 10
the opportunities of network slicing in an efficient manner.   11
RAN slice SLA assurance scenario involves Non-RT RIC, Near-RT RIC, E2 Nodes and SMO interaction.  The scenario 12
starts with the retrieval of RAN specific slice SLA/requirements (possibly within SMO or from NSSMF depending on 13
Operator deployment options). Based on  slice specific performance measurements from E2 Nodes, Non -RT RIC and 14
Near-RT RIC can fine-tune RAN behavior aligned with O -RAN architectural roles to assure RAN slice SLAs. Non -RT 15
RIC monitors long-term trends and patterns for RAN slice subnets’ performance, and employs AI/ML methods to perform 16
corrective actions through SMO (e.g. reconfiguration via O1) or via creation of A1 policies. Non -RT RIC can also 17
construct/train relevant AI/ML models that will be deployed at Near -RT RIC. A1 policies possibly incl ude scope 18
identifiers (e.g. S-NSSAI) and statements such as KPI targets. On the other hand, Near -RT RIC enables optimized RAN 19
actions through execution of deployed AI/ML models in near-real-time by considering both O1 configuration (e.g. static 20
RRM policies) and received A1 policies, as well as received slice specific E2 measurements.  21

respective SLAs through SMO. In addition, Non-RT RIC may retrieve enrichment information from 3rd party applications 18
enabling advanced RAN slicing technology to be applied in O-RAN framework.  19
In order to construct AI/ML models to be deployed in the Near-RT RIC, Non-RT RIC retrieves slice specific performance 20
metrics, configuration parameters and required attributes of the RAN slice subnets from the SMO framework. Complex 21

Near-RT RIC is the component which enables near-real-time RAN slice subnet optimization through execution of slicing 8

as well as their requirements. This information will be received through O1 interface during provisioning of RAN slice  13
subnets. Therefore similar to Non -RT RIC, Near-RT RIC will be aware of RAN slice  subnets through O-RAN specific 14

and E2 interfaces respectively, where the PMs can be used for slice performance monitoring and slice SLA assurance 32

management and enrichment information services [18]. These three services will be utilized for various slicing use cases, 6
such as slice SLA assurance. Policy management will be used by Non-RT RIC to send slice specific (e.g. S-NSSAI based) 7
policies to guide Near -RT RIC with slice resource allocations and slice specific control activities, as well as to receive 8
slice specific policy feedback for the policies deployed on the Near-RT RIC. 9
For the use cases that make use of external enrichment data or where Non-RT RIC produces enrichment information, A1 10
enrichment interface will be used to send slice specific enrichment data to Near -RT RIC. 11
It should be noted that slice specific A1 policies are not persis tent (do not survive the restart of Near-RT RIC) and while 12
they may take precedence over O1 slice specific configurations, they should be aligned and not deviate significantly from 13
O1 configurations.  14
Note: It is intended to add examples for the usage of A1 services for slicing use cases in the next version of this document. 15
5.6 E2 Interface 16
E2, which is the interface between the Near -RT RIC and the E2 nodes, supports E2 primitives (Report, Insert, Control 17
and Policy) to control the services exposed by E2 nodes [19]. These primitives will be used by slice specific applications 18
(xApps) to drive E2 nodes’ slice configurations and slice specific behaviour, such as slice based radio resource 19
management, radio resource allocations, MAC scheduling policies and other  configuration parameters used by various 20
RAN protocol stacks.  21
E2 will be used to configure and receive slice specific reports and performance data from E2 nodes. Thes e reports may 22
include 3GPP defined slice specific PMs (such as PRB utilization, average delay, etc.  [10]) and new PMs that can be 23
defined by O-RAN to support various slicing use cases. 24
Note: It is intended to add examples for the usage of E2 primitives for slicing use cases in the next version of this document. 25
5.7 O1 Interface 26
O1, which is the interface between O-RAN managed elements and the management entity as defined in [14], will be used 27
to configure slice specific parameters of O-RAN nodes based on the service requirements of the slice. 3GPP have defined 28
some of the slice specific information models in TS 28.541 [9], including the RRM policy attributes to provide the ratio 29
of PRBs and the split of these PRBs among slices. To support O-RAN slicing use cases and their requirements , 3GPP 30
information models may be extended and additional information models may be defined to capture slice profiles and slice 31
specific configuration parameters, which will be carried over O1 interface as well. 32
O1 will also be used to configure and gather slice specific performance metrics and slice specific faults from O -RAN 33
nodes. 34
Note: It is intended to add examples for the usage of O1 for the configuration, performance and fault management of 35
slicing use cases in the next version of this document. 36

C.1 Slicing and 3GPP Slice Modeling Terminology Awareness 9
This section intends to provide further information and clarified awareness for some of the fundamental network slicing 10
concepts and 3GPP modeling aspects.  11
C.1.1 Conceptual Differences between 3GPP SA2 and SA5 12
(System Architecture) Network Slice (3GPP SA2): Even though this is a system architecture group, the RAN architecture 13
is out of the scope of SA2. RAN3 owns the RAN architecture. SA2 service -based architecture (SBA) is a functional 14
architecture with mostly service-based interfaces. 15
(Management Aspect) Network Slice (3GPP SA5): SA5 describes all the management aspects of a network, which is 16
modular and model driven. It is a service -based management architecture ( SBMA), using service -based API allowing 17
implementations to pick and choose appropriate API and compliance per API. See 3GPP TS 28.533 [8].  18
A Network Slice Instance (NSI), a SA2 term, is a set of functions. SA2 also introduced the concept of an instance because 19
there would be more than one slice of the same type for the purpose of scaling. What is managed by an operator is a group 20
of functions (an NSI). 21
A NetworkSlice Managed Object Instance (MOI), a SA5 usage, is an object exposing to an external consumer. Once an 22
operator builds a something to satisfy expectations, it needs to build something that is exposed to an external consumer 23
which is a NetworkSlice MOI. Scale i n/Scale out is an operation within Life Cycle Management which is under the 24
purview of SA5. In the context of SA5, a NSI is just an attribute carrying signaling “NSI” value semantic. From a 25

C.1.2 Name Containment Limitation 5
NSSI stands for NetworkSliceSubnet Instance a.k.a. Managed Object Instance (MOI) of the NetworkSliceSubnet 6
Information Object Class (IOC). 7
NetworkSliceSubnet IOC has been introduced to group instances (of any IOC) in a way that is NOT restricted by the 8
Name Containment (DN) rules. Those name containment rules are described in 3GPP TS 32.300 [12]. Name containment 9
defines a managing hierarchy . This is ok; however,  it restricts the flexibility needed for Network Slice management 10
(especially for shared Network Functions). 11
This short-coming of the TS 32.300 [12] is addressed by the NetworkSliceSubnet which allows for multiple views or 12
“overlays” to augment the management hierarchies. This grouping is independent of how the network is man aged and 13
structured via distinguished names. Note that the NetworkSliceSubnet inherits from the top IOC (see TS 28.541 [9] clause 14

of the generic collection instance that may support one or more NetworkSliceSubnet instances. For an end-to-end Network 23
Slice to exist, it is expected that it would have an associated slice profile containing a pLMNInfoList.  24

C.1.7 Single Network Slice Selection Assistance Information (S-NSSAI) 3
A Single Network Slice Selection Assistance Information (S-NSSAI) is used to define a slice. It is a 32-bit number defined 4
by 3GPP TS 23.003 [3] Clause 28.4.1. S-NSSAI is comprised of Slice / Service Type (SST) and Slice Differentiator (SD). 5
SST have values 0 to 127 standards reserved and 128 values are operator defined. These are defined in TS 23.501 [4]. 6
The Standardized SST values are given in 3GPP TS 23.501 [4] clause 5.15.2.2. 7
