1.4 Philosophy 27 
The O-RAN O1 management services follow existing standards wherever possible.  The focus of this document is to 28 
identify the use cases which conform to existing standards, identify gaps in management services for O -RAN and define 29 
needed extensions.  For identified gaps, the goal is to modify the standards to include the needed O-RAN extensions 30 
and update the references in this document as the standards evolve to cover the gaps.  31 
It is recognized that the O-RU follows the O-RAN Front-Haul m-plane specification [29] initially but will evolve over 32 
time to comply with the O1 specification supported by other O-RAN Managed Functions. 33 
 34

As each Management Service is evaluated, the Use Cases and relevant specifications need to be assessed and 36 
augmented as needed to support O-RAN. The current list of Use Cases in Chapter 2 below is not exhaustive, and the list 37 
of specification references may not be complete. 38 
Some Use Cases referred to in the standard may not be applicable to O-RAN and this needs to be addressed, perhaps as 39 
an exception.  40

Chapter 2. Management Services 1 
2.1 Provisioning Management Services 2 
Provisioning management services allow a Provisioning MnS Consumer to configure attributes of managed objects on 3 
the Provisioning MnS Provider that modify the Provisioning MnS Provider’s capabilities in its role in end-to-end 4 
network services and allows a Provisioning  MnS Provider to report configuration changes to the Provisioning MnS 5 
Consumer. NETCONF is used for the Provisioning Management Services to Create Managed Object Instance, Delete 6 
Managed Object Instance, Modify Managed Object Instance Attributes and Read Managed Object Instance Attributes. 7 
A REST/HTTPS event is used to notify the Provisioning MnS Consumer when a configuration change occurs on the 8 
Provisioning MnS Provider which was not initiated by the Provisioning MnS Consumer.  YANG 1.1 Data Models are 9 
used by O-RAN Provisioning MnS Providers. 10 
O-RAN Provisioning MnS Provider must support a running datastore.  The support of candidate datastore on the 11 
Provisioning MnS Provider is optional. 12 
O-RAN ME must support the XPATH capability. 13 
Version 2.0 of the O1 interface will define more of the espected capabilities the O-RAN ME should support.   14 
Use cases are from 3GPP TS 28.531. 15 
Reference documents include 3GPP TR 28.890, TS 28.531, RFC 6241, “Network Configuration Protocol 16 
(NETCONF)”, IETF, RFC 7950, “The YANG 1.1 Data Modeling Language” . 17 
2.1.1 Create Managed Object Instance 18 
2.1.1.1 Description 19 
Provisioning MnS Consumer sends synchronous provisioning updates to Provisioning MnS Provider to create a 20 
Managed Object Instance (MOI) on the Provisioning MnS Provider and set its attribute values.  21 
2.1.1.2 General NETCONF Requirements 22 
Requirements are to be specified in a 3GPP spec for the Provisioning management services.  Until that time, the 23 
requirements are provided in this O1 Interface Specification.  24 
REQ-GNC-FUN-1:  The provisioning management service provider shall have the capability to establish a NETCONF 25 
session with its authorized consumer upon request from the consumer. 26 
REQ-GNC-FUN-2:  The provisioning management service provider must support an established NETCONF session 27 
until the authorized consumer terminates the session.  NOTE: The consumer may want to perform multiple provisioning 28 
management services operations during a single NETCONF Session.   29 
REQ-GNC-FUN-3:  The provisioning management service provider shall have the capability to terminate a NETCONF 30 
session with its authorized consumer when requested to do so by the authorized consumer.    31 
2.1.1.3 Create MOI Requirements 32 
Requirements are to be specified in a 3GPP spec for the Create Managed Object Instance use case.  Until that time, the 33 
requirements are provided in this O1 Interface Specification.  34 
REQ-CMO-FUN-1:  The provisioning management service provider shall have the capability to allow its authorized 35 
consumer to create MOI(s) and set attribute values of those MOIs via the NETCONF <edit.config> operation. 36 
REQ-CMO-FUN-2:  The provisioning management service provider shall have the capability to make the new MOI(s) 37 
persistent in its running datastore.  38

2.8.1 Software Package Naming and Content 21 
PNF Software Package naming, content and format are vendor specific and do not require standardization in O -RAN.  22 
A PNF Software Package may contain one or more files.  Some of the files in the Software Package may be op tional for 23

term that O-RAN will use to refer to the software available on the PNF rather than the legacy term of software slot.  On 29 
a PNF, a softwarePackage MOI corresponds to a hardware software slot.  30
