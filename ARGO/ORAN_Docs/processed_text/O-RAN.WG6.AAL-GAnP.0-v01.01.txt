Chapter 2. General Aspects 1 
2.1 Hardware Acceleration 2 
In the design of digital computing systems, ranging from general-purpose processors to fully customized hardware, 3 
there is a tradeoff between flexibility and efficiency, with efficiency increasing by orders of magnitude when any given 4 
application is implemented in hardware. The range of implementation options includes general -purpose processors such 5 
as CPUs, more specialized processors such as GPUs, functions implemented on field- programmable gate arrays 6 
(FPGAs), and fixed-functions implemented on application-specific integrated circuits (ASICs). Hardware accelerator is 7 
a specialized HW implementation that can perform some pre-defined work offloaded from the General-Purpose 8 
Processor. Any transformation of data or routine that can be computed, can be calculated purely in software running on 9 
a generic CPU, purely in a custom-made hardware, or using a combination of both. The implementation of computing 10 
tasks in hardware to reduce latency and to increase throughput is known as hardware acceleration. The hardware 11 
acceleration is implemented in the form of lookaside or inline mode where in the former case, the host CPU invokes an 12 
accelerator for data processing and receives the result after processing is complete, while in the latter case,  the host 13 
CPU, after invoking an accelerator for data processing, does not necessarily retrieve the post processed data. The 14 
principle of hardware acceleration and functional offloading is illustrated in  15 
Figure 2.1, allowing the application to offload workload to a hardware accelerator and to continue performing other 16 
work in parallel- this could be to continue to execute other software tasks in parallel or to sleep and wait for the 17 
accelerator hardware to complete. The hardware acceleration boosts application performance in environments with 18 
compute-intensive, deeply pipelined, massively parallel operations as shown in Figure 2.1. This model requires the API 19 
to support two operations, one for initiating the offload and another for retrieving the operation once complete.  20

2.2 AAL Architecture  24 
The goal of the AAL is to specify a common and consistent interface for HW Device accelerators to the applications 25 
which facilitates decoupling of an application, e.g. O-RAN Cloudified Network Function, from a specific HW 26 
Accelerator device implementation. In order to accommodate the many different combinations of HW and SW 27 
implementation and also many different network deployment scenarios,  the AAL introduces the concept of an AAL 28 
profile which is used to distinguish between the different combinations of accelerated functions to be offloaded. The 29 
end to end high-level AAL architecture block diagram is shown in Figure 2.2. The AAL specification will define what 30 
interfaces to use when exposing these management functions to the O-Cloud IMS/DMS. The orchestration of the HW 31 
Accelerator Manager and the details of the IMS/DMS or O2 is outside the scope of this  AAL GAnP document and shall 32 
be specified in the ORAN WG6 O2 specifications [12]  33

2.5 General Interface Principles  10 
2.5.1 Generic Principles   11 
2.5.1.1 Extensibility  12 
O-RAN has defined the functions that can be accelerated by the cloud platform based on 3GPP specifications and O-RAN 13 
deployment scenarios. However, the AAL API should not limit innovation of future implementations and should evolve 14 
as the specification requires. In this way, the API shall be extensible to accommodate future revisions of the specification. 15 
2.5.1.2 HW Independence  16 
An AAL profile API should be independent of the underlying HW. 17 
2.5.1.3 Interrupt and Poll Mode 18 
The API shall allow multiple design choices for application vendors and shall not preclude a vendor from adopting an 19 
interrupt-driven design or poll -mode design or any combination of both. As such, the API shall support both interrupt 20 
mode, poll mode and any combination of interrupt and poll modes for the data -path application interface.   21

2.5.1.7 Look-aside Acceleration Model  15 
The AALI shall support look-aside acceleration model where the host CPU invokes an accelerator for data  processing 16 
and receives the result after processing is complete. A look-aside architecture, illustrated in Figure 2.7, allows the 17 
application to offload work to a hardware accelerator and continue to perform other work in parallel—this could be to 18 
continue to execute other software tasks in parallel or to sleep and wait for the accelerator hardware to complete. This 19 
model requires the AALI to support two operations, one for initiating the offload and another for retrieving the 20 
operation once complete. 21

The AALI shall support inline acceleration model where acceleration by function and I /O-based acceleration are 2 
performed on the physical interface as the packet ingresses/egresses the platform. Figure 2.8. AAL i nline acceleration 3 
model shows one possible implementation of an inline acceleration model. 4 
 5 
Figure 2.8. AAL inline acceleration model 6

an Ethernet interface, while “Rx” refers to the reception of data from the Ethernet interface to the acceleration device.  8

accelerator and directly from the accelerator to the front-haul interface, instead of being sent back to the CPU. The typical 11 
user plane data flows for accelerating the O -DU high-PHY functions for the look- aside and inline architectures are as 12 
follows. 13 
Look-aside architecture user plane dataflow 14 
CPU ↔ accelerator ↔ CPU ↔ front-haul: for a set of consecutive PHY functions offload (e.g., FEC) 15 
CPU ↔ accelerator ↔ CPU ↔ accelerator ↔…↔ CPU ↔ front-haul: for a set of non-consecutive PHY functions offload 16 
Inline architecture user plane dataflow 17 
CPU ↔ accelerator ↔ front-haul: for a set of consecutive PHY functions offload (up to the end of the PHY pipeline)  18

The AAL interface has two distinct parts, the first a set of common methods (‘AAL Common’) which allows an 2

across the HW Accelerators accelerating the same set of functions. It enables the applications to be able to efficiently 6 
offload the accelerated workload process to HW Accelerators in a consistent way without requiring them to know every 7 
single detail of the accelerators. Figure 4.1 shows examples of the AALI APIs presented to an application in three 8

5.1.3 O-DU AAL Profile Definitions 11 
O-DU AAL profiles are defined below with future specification(s) to define the AALI  for each profile.  12 
5.1.3.1 Profile Definitions General Guidelines  13 
5.1.3.1.1 Naming  14 
As discussed above O-DU AAL profiles are specific to one or more physical channel(s) or signal(s) as such should 15 
follow the naming guidelines  16 
• O-DU AAL profiles shall be prefixed with “AAL_”  17 
• O-DU AAL Profiles when specific to a single channel or signal shall include the channel or signal in the na me 18 
e.g. “AAL_PUSCH”  19 
• O-DU AAL Profiles when common across multiple channels or signals shall not include the channel or signal 20 
name, instead just reference the Accelerated Function(s), e.g. AAL_RE-MAPPING 21 
• O-DU AAL Profiles that include a subset of the functional blocks within a channel or signal shall include a 22 
functional description after the channel name e.g. AAL_PUSCH_CHANNEL_ESTIMATION  23 
 24 
5.1.3.1.2 Data Flow 25 
O-DU AAL Profiles shall specify the data flow supported by the profile as discussed in section 2.5.1.7 and 2.5.1.8 26 
above e.g. Look aside, Inline or other.  27 
Look aside data flow implies the remaining functions not included in the Profile that comprise the channel or signal are 28 
implemented in SW on the host CPU and not implemented in the HW Accelerator.  29 
Inline data flow implies that the set of accelerated functions is constituted of the entire U -plane processing of high-PHY 30 
channel or signal (with 7-2x PHY functional split) and the IQ data (in DL) or decoded bits (in UL) (post processing) are 31 
transferred directly from the accelerator to the Fronthaul interface. (in DL) or Layer 2 (in UL).  32 
The profile shall specify if the data flow includes only user plane, or only control plane, or both. 33
