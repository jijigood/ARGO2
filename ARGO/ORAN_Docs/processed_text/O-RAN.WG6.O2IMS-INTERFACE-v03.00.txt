2.1.1.1.3 ResourceType 267 
The ResourceType service object represents a resource type in an O-Cloud. 268 
Resource types provide the categorization of O-Cloud Nodes as described in the O-RAN Cloud Architecture and 269 
Deployment Scenarios [7] specification. The values for these types are used to build the inventory and enable an 270 
authorized consumer to track resource changes over time. The O-Cloud provides a mechanism to discover the resource 271 
types with respective operations on resource types. 272 
Please note that the list of resource classes is not exhaustive, and additional classes are FFS.  273 
The information representing the ResourceType shall follow the provisions indicated in table 2.1.1.1.3 -1. 274

model M Information about the model of the resource type. 
version M Version or generation of the resource type.

2.1.1.1.7 Subscription 309 
The Subscription service object represents an instance of a subscription to infrastructure inventory notifications for 310 
when resources within the inventory change. 311 
Subscriptions are required for a notification to be generated. Information about the change are provided to the consumer 312 
when the consumers specified criteria is met. The absence of a filter means all events are sent.  313 
The information representing one subscription shall follow the provisions indicated in table 2.1.1.1.7-1. 314 
Table 2.1.1.1.7-1: Attributes of the Subscription 315 
Attribute Qualifier Description 
subscriptionId M Identifier of the subscription. Locally unique within the scope of an 
O-Cloud instance. 
callback M The URL of the callback to which notifications are delivered to. 
consumerSubscriptionId O The consumer may provide its identifier for tracking, routing, or 
identifying the subscription used to report the event. 
filter O Criteria for events which do not need to be reported or will be 
filtered by the subscription notification service. Therefore, if a filter 
is not provided then all events are reported.

2.1.1.2.2 OCloud 419 
The OCloud service object represents an instance of an O-Cloud within the SMO. For more information about O-Cloud, 420 
refer to clause 3.1 in the "O-RAN O2 General Aspects and Principles Specification" [24]. 421 
The SMO has a view of the O-Cloud and initiates its deployment. From the O2 General Aspects and Principles [24] the 422 
SMO generates an ID and inventory record for the O-Cloud. Once the O-Cloud has reached a level of completeness, 423 
when it is ready to be provisioned for used to deploy workloads, the O-Cloud invokes a callback identified at the onset 424 
of the lifecycle process passing enough information for the SMO to associate the assigned globalCloudID, with the 425 
internal oCloudID and store its Infrastructure Management Service end point. The relationship to the globalCloudID 426 
and oCloudID is one-to-one in that there is a single, unique globalCloudID for each O-Cloud. The O-Cloud generates 427 
its own ID for its internal data relationships. Thus, reducing the coupling of the O-Cloud and the SMO which could be 428 
from different vendors. There is no guarantee that two different O-Clouds will not have the same internal oCloudID and 429 
therefore it is the responsibility of the SMO to discern and federate the two with different gl obalCloudID values. 430
