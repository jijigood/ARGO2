implementations in Non/Near-RT RICs that are based on different versions of this specification. 21

added or that an incompatible change has been made to an A1 service. An incremented second digit could indicate that 23
an optional feature has been added, or that clarifications or corrections have been made.  24

and which version(s) of each A1 service that are implemented. The version of an A1 service is indicated by the API 26
version in the URI (see chapter 4) and compatibility is governed by the version of the OpenAPI document for the A1 27
service (see Annex A). The present document handles the service compatibility aspects while A1 interface: Type 28
Definitions [5] handles the compatibility for data types used by the A1 servic es. 29

Chapter 2 A1 Application Protocol 30
This document contains a REST method realization of the interface architecture , and policy and EI procedures 31
identified in A1 interface: Generic Aspects and Principles [3]. It is based on HTTP transport as defined in A1 interface: 32
Transport Protocol [4] and an application data model defined in A1 interface: Type Definitions [5] . 33
This definition of the A1 Application Protocol (A1AP) is based on the 3GPP service framework for network functions 34
specified in 3GPP TS 23.501 [6]. It corresponds to a REST-based Solution Set and is based on the structure in the 3GPP 35
specification TS 29.501 [7] and the related TS template [8]. The design patterns for HTTP procedures are based on 36
3GPP TS 32.158 [9] and the design patterns for JSON objects are based on 3GPP TS 32.866 [10]. 37

3.2.1.2 Policy representation 8
The following principles are used for A1 policies when JSON is used as resource representation format:  9
- A policy corresponds to a resource (in the REST sense); 10
- A policy is represented as a JSON object referred to as a PolicyObject; 11
- A PolicyObject contains a scope identifier and at least one policy statement (e.g. one or more policy objective 12
statements and/or one or more policy resource statements); 13
- A policy is identified by a policyId that is included in the URI when operation is for a single policy; 14
- The policyId is assigned by the A1-P Consumer when the policy is created; 15
- The A1-P Producer cannot modify or delete a policy; 16
- Policy feedback for a specifc policy is subscribed to when the policy is created by providing a callback URI in 17
the Create policy operation; 18
- A PolicyObject does not contain any information related to which internal function in the Near-RT RIC that is 19
to evaluate the policy;  20
- The A1-P Producer indicates for which policy types policy creation is supported, and the JSON schemas for 21
those policy types can be retrieved by the A1-P Consumer; 22
- The A1-P Consumer cannot create, modify or delete policy types. 23
3.2.1.3 Representation objects 24
The following JSON objects are used within the service operations of the A1-P service: 25
PolicyTypeObject 26
 The PolicyTypeObject contains the JSON schemas used to validate a PolicyObject and a PolicyStatusObject. 27
PolicyObject 28
 The PolicyObject is the JSON representation of an A1 policy. 29
PolicyStatusObject 30
The PolicyStatusObject is the JSON representation of the enforcement status of an A1 policy. 31
ProblemDetails 32
The ProblemDetails object is the JSON representation of the content in a response message with ot her HTTP error 33
response codes (4xx/5xx). 34

Figure 3.2.2.2-1 Create policy procedure. 12
The procedure is as follows: 13
1) The A1-P Consumer generates the policyId and sends a HTTP PUT request to the A1-P Producer. The target 14
URI identifies the resource (policyId) under which the new policy shall be created. The message body carries a 15

2) The A1-P Producer returns the HTTP PUT response. On success, “201 Created” is returned. The location 17
header is present and carries the URI of the new policy and the message body the PolicyObject. On failure, the 18
appropriate error code is returned, and the message body may contain additional  error information. 19
As the A1-P Producer has indicated which policy types it supports, when creating a policy, the A1-P Consumer includes 20
a policyTypeId in the URI for the PUT request. The policyTypeId is used by the A1-P Producer to select the appropriate 21
schemas to use for validation of the PolicyObject and for PolicyStatus. 22
If the provided policyTypeId is not supported or validation of the PolicyObject fails, “400 Bad Request” is returned and 23
the message body may contain additional error information. 24
If the A1-P Consumer likes to receive policy status updates related to the created policy, it includes the 25
notificationDestination as a query paremeter in the PUT request. 26

3.3.1.2 EI representation 7
The following principles are used for A1 Enrichment Information when JSON is used as resource representation format:  8
- The A1-EI Producer can indicate the EI types that are available;  9
- An EI type is identified by an EI type identifier and the schemas for available EI types can be retrieved by the 10
A1-EI Consumer; 11
- An EI job can be created for delivery of information of a specific EI type;  12
- An EI job corresponds to a resource (in the REST sense); 13
- An EI job, when transferred over HTTP,  is represented as a JSON object referred to as an EI job object; 14
- An EI job object contains a scope identifier and parameters and conditions related to the EI type the delivery is 15
for; 16
- An EI job is identified by an EI job identifier that is included in the URI when operation is for an EI job; 17
- The EI job identifier is assigned by the A1-EI Consumer when the EI job is created; 18
- Status for a specific EI job can be queried and notifications can be subscribed to when the EI job is created by 19
providing a callback URI in the create EI job operation; 20
- An EI job object does not contain any information related to which source that produces it nor which internal 21
function in the near-RIC that is to consume it;  22
- EI job results are deliverd to a callback URI provided during create EI job operation;  23
- Delivered EI that is represented as a JSON object is referred to as an EI job result object. 24

A.1.1 Versioning of A1 OpenAPI documents 8
The OpenAPI documents for the A1 services found in this chapter are versioned by Semantic Versioning 2.0.0 [12] as 9
described in the OpenAPI Specification [16]. When included in the present specification, the OpenAPI documents are 10
considered as released and are versioned using three digit major.minor.patch where the main compatibility expectations 11
stated for Sematic Versioning [12] implies: 12
major version is stepped up when incompatible API changes are made to the OpenAPI document. This corresponds 13
to saying that implementations of an A1 service in Non/Near-RT RICs are incompatible in case the API version is 14
different. The major version in the OpenAPI document corresponds to the API version in the URI for of the A1 15
service defined in chapter 4. 16
minor version is stepped up when features are added to the OpenAPI document in way that keeps implementations 17
compatible although all features are not supported by both the service producer and the service consumer of the A1 18
service. 19
patch version is stepped up when errors are corrected in a backward compatible way, when or editorial changes are 20
made to the OpenAPI document, but no features are added.  21
Note: Non/Near-RT RIC products that implement various API versions of an A1 service can be compatible by 22
supporting several API versions of an A1 service. The present specification specifies only one API version, and 23
contains only one OpenAPI document, for each A1 service. 24
