1.1 Scope 2 
This Technical Specification has been produced by the O-RAN Alliance. 3 
The contents of the present document are subject to continuing work within O-RAN and may change following formal 4 
O-RAN approval. Should the O-RAN Alliance modify the contents of the present document, it will be re-released by O-5 
RAN with an identifying change of release date and an increase in version number as follows: 6 
Release xx.yy.zz 7 
where: 8 
xx the first two-digit value is incremented for all changes of substance, i.e. technical enhancements, corrections, 9 
updates, etc. (the initial approved document shall have xx=01). 10 
yy the second two-digit value is incremented when editorial only changes have been incorporated in the 11 
document. 12 
zz the third two-digit value is included only in working versions of the document indicating incremental 13 
changes during the editing process; externally published documents never have this third two-digit value 14 
included. 15 
The present document specifies the data model and the data types that are used in the body of the procedures in the A1 16 
interface. It is part of a TS-family covering the O-RAN WG2: A1 interface as identified below: “General Aspects and 17 
Principles”. “Transport Protocol”. “Application Protocol”. “Type Definitions”. 18 
1.1.1 Compatibility of A1 type definitions 19 
The version number of the present document indicates that there may be implications for the compatibility between 20 
implementations of policy types and/or EI types defined in different versions of this specification. 21 
The first two-digit value of this specification is incremented when 22 
at least one policy type and/or one EI type has been added or removed 23 
and/or 24 
at least one policy type and/or one EI type has been updated in a non-backward compatible way. 25 
The first two-digit value of this specification is incremented when at least one policy type and/or one EI type has been 26 
updated in a backward compatible way. 27 
Policy type compatibility is described in Chapter 5 and EI type compatibility is described in Chapter 7.    28 
The compatibility of A1 implementations in Non/Near-RT RICs depends on which policy types and/or EI types that are 29 
implemented. The present document handles the compatibility for data types used by the A1 services while A1 30 
interface: Application Protocol [4] handles the A1 service compatibility aspects. 31

Chapter 2 A1 Application data model 24 
This document together with the A1 interface: Application Protocol [4] contains a REST method realization of the A1 25 
interface architecture and the A1 procedures identified in A1 interface: Generic Aspects and Principles [3].  26 
The A1 interface: Application Protocol specification [4] contains the service description, service operations, resource 27 
indicators and the API definition (including the OpenAPI document) for the A1 services. The present document 28 
contains the data model and the defintions of the objects transported in the procedures defined  for the A1 services. 29 
The data types defined in this document are life cycled independently from the A1 services defined in the A1 interface: 30 
Application Protocol [4]. The documentation of the A1 data models follows the application data model used for API 31 
definitions in 3GPP TS 29.501 [5] and data types specified in 3GPP TS 29.571 [6]. It is based on structured data types 32 
and objects as specified in IETF RFC 8259 [7] and JSON schema [8]. 33 
Chapter 3 Generic aspects and common data types 34

The encoding of 3GPP attributes into JSON is based on their original definitions, and their value ranges, rather than on 36 
encodings used in other protocols or solutions sets. The definitions are referred to in the data type definitions (see 37 
clauses 4.2 and 4.3) and the corresponding encodings are seen in the type definitions (see clause 5.1.2 and 5.2).  38

4.1 Introduction 2 
This clause specifies the application data model and data types supported by the A1-P API (see A1 interface: 3 
Application Protocol [4]). The data model is based on policy statements that include attributes and are combined with a 4 
scope identifier into policy objects. 5 
For policy objectives, policy statements for the following characteristics are defined:  6 
- QoS targets; 7 
- QoE targets;  8 
-     UE level targets; 9 
- Slice SLA targets. 10 
For policy resources, policy statements for the following characteristics are defined:  11 
-  Traffic steering optimization; 12 
- Slice SLA assurance. 13

Chapter 5 A1-P Data Types (Policy Types) 20 
A policy type is identified by a PolicyTypeId as defined in A1 interface: Application Protocol [4]). The PolicyTypeId is 21 
a string that consists of two parts: a typename and a version. 22 
When updating a policy type, the version in the PolicyTypeId is updated according to SemVer [21] to reflect its 23 
compatibility with other policy types that has the same typename. 24 
Two policy types are considered as different if the PolicyTypeId is different, i.e. even if the typename is the same and 25 
the version only differs in the patch version digit.  26 
Two policy types are compatible in case the typename is the same and the major version digit in the version is the 27 
same.  28

A1-EI Data Model 1 
6.1 Introduction 2 
This clause specifies the application data model supported by the A1 -EI API (see A1 interface: Application Protocol 3 
[4]). The purpose of the data model is to be the basis for 4 
- Definition of EI Types; 5 
- The EI representation objects that are transported in the body of the A1 -EI procedures. 6 
There are two kinds of EI Types: those defined by O-RAN and those defined by another entity. EI Types need to define: 7 
- An EiTypeIdentifier for usage in the A1-EI procedures and URI structure; 8 
- The content to be transported in the body of the A1-EI procedures. 9 
The content is referred to as EI representation objects for the O-RAN defined EI Types and is defined by using the A1-10 
EI data model specified in this subclause. An EI Type defined outside of O-RAN may use the A1-EI data model or 11 
another model that covers the content corresponding to the schemas and objects. 12 
This specification covers the data model for O-RAN defined EI types. The O-RAN defined EI Types are based on the 13 
statements and attributes defined in the data model and may extend it with EI type specific rules and attributes.  14 
The O-RAN defined EI Types are defined based on JSON schemas [8]. An EI Type is defined by three schemas for EI 15 
job, EI job status and EI job result. The schemas are used to validate the EI representation objects transferred in the 16

Chapter 6 A1-EI Data Types (EI Types) 9 
An EI type is identified by a EiTypeId as defined in A1 interface: Application Protocol [4]). The EiTypeId is a string 10 
that consists of two parts: a typename and a version. 11 
When updating an EI type, the version in the EiTypeId is updated according to SemVer [21] to reflect its 12 
compatibility with other EI types that has the same typename. 13 
Two EI types are considered as different if the EiTypeId is different, i.e. even if the typename is the same and the 14 
version only differs in the patch version digit.  15 
Two EI types are compatible in case the typename is the same and the major version digit in the version is the 16 
same.  17
