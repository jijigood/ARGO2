4 General  1
4.1 Procedure Specification Principles 2
The principle for specifying the procedure logic is to specify the functional behaviour of the terminating node exactly 3
and completely. Any rule that specifies the behaviour of the originating node shall be possible to be verified with 4
information that is visible within the system. 5
The following specification principles have been applied for the procedure text in clause 8:  6
- The procedure text discriminates between: 7
1) Functionality which "shall" be executed. 8
 The procedure text indicates that the receiving node "shall" perform a certain function Y under a certain 9
condition. If the receiving node supports procedure X but cannot perform functionality Y requested in the 10
REQUEST message of a Class 1 EP, the receiving node shall respond with the message used to report  11
unsuccessful outcome for this procedure, containing an appropriate cause value.  12
2) Functionality which "shall, if supported" be executed. 13
 The procedure text indicates that the receiving node "shall, if supported," perform a certain function Y under 14
a certain condition. If the receiving node supports procedure X, but does not support functionality Y, the 15
receiving node shall proceed with the execution of the EP, possibly informing the requesting node about the 16
not supported functionality. 17
- Any required inclusion of an optional IE in a response message is explicitly indicated in the procedure text. If the 18
procedure text does not explicitly indicate that an optional IE shall be included in a response message, the 19
optional IE shall not be included. For requirements on including Criticality Diagnostics IE, see clause 10. 20
4.2 Forwards and Backwards Compatibility 21
The forwards and backwards compatibility of the protocol is assured by a mechanism where all current and future 22
messages, and IEs or groups of related IEs, include ID and criticality fields that are coded in a standard format that will 23
not be changed in the future. These parts can always be decoded regardless of the standard version.  24

6 RAN Function Service Model Description 1
6.1 RAN Function Overview 2
For the purposed of this E2 Service Model, E2SM-NI, the E2 Node terminating the E2 Interface is assumed to host the 3
RAN Function NI “Network Interface” which performs the following functionalities: 4
- Exposure of Network Interfaces 5
- Modification of both incoming and outgoing network interface message contents 6
- Execution of policies that may result in change of network behavior 7
This E2SM specification provides a set of RAN Function exposure services described in clause 6.2 and has been 8
prepared with the assumption that the same E2SM may be used to describe either a single RAN Function handling all 9
Network Interfaces or more than one RAN function with each one handling a subset of the Network Interfaces 10
terminated on the E2 Node. 11
6.2 RAN Function exposure services 12
6.2.1 REPORT service 13
The “Network Interface” RAN Function provides selective support of the following REPORT services: 14
- Copy of Complete message with header providing network interface type, identifier and direction with optional 15
network interface timestamp 16
- Copy of Partial message selected by target message protocol IE identifier with with header providing network 17
interface type, identifier, direction and message type with optional network interface timestamp 18
These services may be initiated according to: 19
- Target network interface type, identifier, direction and message type along with the presence of a sequence of 20
one or more message protocol IE identifiers with optional message protocol IE contents selection rules.  21
6.2.2 INSERT service 22
The “Network Interface” RAN Function provides selective support of the following INSERT services: 23
- Forwarding of Complete message with header providing network interface type, identifier and direction with 24
optional network interface timestamp  25
This service may be initiated according to: 26
- Target network interface type, identifier and direction, message type and presence of a sequence of one or more 27
message protocol IE identifiers with optional message protocol IE contents selection rules.  28
6.2.3 CONTROL service 29
The “Network Interface” RAN Function provides selective support of the following CONTROL services: 30
- Injection of Complete message with header providing target network interface type, identifier and direction and 31
optional RIC Control Message Priority 32
6.2.4 POLICY service 33
The “Network Interface” RAN Function provides selective support of the following POLICY services: 34
- Handling of Message Admission process according to a list of parameters to be applied to different RAN UE 35
groups 36

Annex A: Further information on RAN Function Network 23

A.1 Background information 25
The RAN function “Network Interface” is used to provide RIC Service expose of the network interface logical function 26
of the E2 Node.   27
It is assumed that the network interfaces are terminated on one of more logical functions with the E2 Node. For each 28
logical function the E2 Node shall use the RAN Function Description to declare the list of supported Network Interfaces 29
and the corresponding set of supported RIC Services (REPORT, INSERT, CONTROL and POLICY) and for each 30
supported RIC Service the list of supported Styles. 31
Each Network Interface is uniquely identified by: 32
- Network Interface Type defined by reference to the name of the interface specification (S1, F1, etc.) 33
- Network Interface Identifier defined by the Global identifier of the network node terminating the network 34
interface 35
Furthermore, it is assumed that a given network interface id bidirectional and so has an incoming direction terminated 36
by the E2 Node and an outgoing direction originated by the E2 Node. 37
Messages on the network interface are assumed to be encoded in terms of Message Types and each message is assumed 38
to contain one or more Protocol Information Elements (IE) each of which may de identified by the unique Protocol IE 39
Identifier as defined in the corresponding network interface specification. 40
Event triggers, Indication and Control procedure message contents and Policy Action D efinitions are encoded in terms 41
of the corresponding network interface specification. 42
