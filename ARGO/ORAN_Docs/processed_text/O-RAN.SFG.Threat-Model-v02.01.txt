The contents of the present document are subject to continuing work within O- RAN and may change following formal 3

The present document specifies the O-RAN Security Threat Modeling and Remediation Analysis. It identifies assets to 15 
be protected, analyzes the O-RAN components for vulnerabilities, examines potential threats associated with those 16 
vulnerabilities and provides security principles which stakeholders should adress when building a secure end-to-end O-17 
RAN system. 18 
The purpose of this version of the document is to present a list of system-level security principles as a basic guidline to 19 
be considered in the design, development, and operation of the O-RAN system. The principles are short and concise and 20 
can be used by organizations to improve the security of their O-RAN components. 21 
In next future versions of the document, the aim is to accomplish a normative security referential by: 22 
• Elaborating on each of security principles in more details,  23 
• Setting precisely what is required (SHALL), recommended (SHOULD) or optional (MAY) , 24 
• Refining security principles into requirements, recommendations and potential countermeasures. 25

Chapter 2 Overview 1 
2.1 Objective and structure 2 
O-RAN architecture [13] is quite different from the architecture of 3GPP RAN, where new components, interfaces and 3 
technologies are proposed, new actors (stakeholders) arise and novel business models are made possible. The attack 4 
surface is bigger and we foresee that O-RAN design and deployment will raise numerous security challenges and 5 
resulting risks because of the new O-RAN specific interfaces and components, the virtualization/containerization 6 
techniques, the support of open source code, the capability to support an AI/ML model, etc.  7 
Therefore, the security analysis and the threat model for O-RAN must be carefully studied and relevant 8 
assets/stakeholders/vulnerabilities/threats/requirements/countermeasures/recommendations  must be identified to reduce 9 
risk exposure and mitigate any harmful effects expected. 10 
The material presented in this analysis aims at supporting various O-RAN stakeholders understanding the relevant 11 
threats resulting to an exposure of O-RAN assets by exploiting the vulnerabilities. 12 
The objective is to give a comprehensive and high-level view on how security is organized in O-RAN system. In this 13 
document, we assume that 3GPP security requirements are met. Unless explicitly stated, features relate to O -RAN 14 
specifications. 15 
This analysis is consolidated from various relevant sources, including main 5G standardization documents and 16 
telecommunication best practices (e.g. 3GPP, ETSI, NIST, ENISA and GSMA). 17 
The first main part outlines the main stockholder roles involved in managing and using O -RAN system. Further, it 18 
addresses the prerequisites and assumptions needed to securely implement and run O -RAN systems. It also identifies 19 
the list of critical assets to be protected in integrity, availability, confidentiality, replay and authenticity.  20 
The second main part addresses the threat model. It identifies the threat agents, determines the threat surface, identifies 21 
vulnerabilities, and lists the threats for each O-RAN component or interface. For each threat, the description, threatened 22 
asset(s), vulnerabilities, threat agents and affected components are given. 23 
The third main part describes the security principles to be achieved to counter the identified threats. In addition, it 24 
illustrates the coverage between threats and security principles. 25 
In next future versions of this document, security requirements, recommendations and countermeasures will be dervived 26 
from security principles. Further, a risk assessment and remediation analaysis will be carried out. 27 
Note that this document focuses only on the components, interfaces and protocols specified by O-RAN alliance. 28 
Components, interfaces and protocols specified by 3GPP are only referenced where needed but are not within the scope 29 
of this specific O-RAN security analysis. 30 
The current document is a working document with the need to update the content on a regular basis following the risk 31 
evaluation. 32 
Note: In this document, the term NF (Network Function) is used to designate either VNF (Virtual Network Function), 33 
CNF (Containerized or cloud-native Network Function) or PNF (Physical Network Function). 34 
Note: Terms “Containers” and “Virtual Machines” are used interchangeably in this document as the implementation of 35 
O-RAN SW components could either be container-based, VM-based or hybrid (Containers and VMs together). 36

1) Identification (scope of this version of the document) 1 
a. Identify stakeholders: First, we need to identify the stakeholders involved in the implementation, 2 
management, operation and maintenance of the O-RAN system. Roles and responsibilities of each 3 
stakeholder are given. 4 
b. Define assumptions: The list of minimum prerequisites and assumptions need to be defined for the 5 
operational environment (not under the control of the O-RAN system) required to successfully 6 
operate the O-RAN system. 7 
c. Identify assets: First, we need to locate relevant assets the O-RAN system hold and give details about 8 
the type (Data, component, etc.), the security properties (CIA) at rest and in transit and location.  9 
d. Identify threats: We need to identify the relevant threats associated with the new O-RAN 10 
components, interfaces and technologies. In addition, the threat surface and agents are given. 11 
e. Identify vulnerabilities: O-RAN system may have weaknesses in its new O-RAN components, 12 
interfaces and technologies which need to be identified. 13 
f. Define security principles: We need to define security principles to be achieved in order to reduce 14 
risk exposure. 15 
g. Elaborate and refine security principles: Each security principle needs to be detailed and refined 16 
into requirements, recommendations and countermeasures. 17 
h. Identify existing/ongoing countermeasures: We need to identify all of O-RAN existing/ongoing 18 
controls and to take into account the protection provided by these controls before applying any new 19 
ones. 20 
2) Risk assessment (will be provided in a future version) 21 
a. After the identification stage, we need to focus efforts on the biggest threats, so we should use the 22 
information you’ve gathered during the identification stage to prioritize the biggest risks. There are 23 
many ways to do this, but the most common approach involves the following equat ion: 24 
Risk = (the probability of a threat exploiting a vulnerability) x (total impact of the vulnerability being exploited)  25 
3) Risk treatment (will be provided in a future version) 26 
a. Now that we know the level of risk that each threat poses, we need to decide how we’ll treat them. 27 
There are four options: 28 
i. Modify the risk by implementing a control to reduce the likelihood of it occurring. 29 
ii. Avoid the risk by ceasing any activity that creates it. This response is appropriate if the risk 30 
is too big to manage with a security control. 31 
iii. Share the risk with a third party. There are two ways we can do this: by outsourcing the 32 
security efforts to another organization or by purchasing cyber insurance to ensure we have 33 
the funds to respond appropriately in the event of a disaster. Neither option is ideal, because 34 
we are ultimately responsible for our organization’s security, but they might be the best 35 
solutions if we lack the resources to tackle the risk. 36 
iv. Retain the risk. This means that our organization accepts the risk and believes that the cost 37 
of treating it is greater than the damage that it would cause. 38

Chapter 5 Threat model 1 
As the industry evolves towards RAN virtualization, it is important that a risk-based approach is taken to adequately 2 
address O-RAN security risks. Building upon the foundation set forth by 3GPP, O-RAN is standardized by the O-RAN 3 
Alliance with new functions and open, interoperable interfaces. With any nascent technology, including O -RAN, 4 
security should be built upon security-by-design using a risk-based approach, with consideration of potential zero-day 5 
attacks.  The O-RAN architecture [13] has security risks beyond those in a 3GPP architecture due to its expanded threat 6 
surface from new functions, additional interfaces, and the Lower Layer Split (LLS).  The disaggregation of hardware 7 
and software, virtualization, and use of open source components also present security risks that must be addressed.  8 
5.1 Threat surface 9 
The O-RAN architecture [13] introduces new functions and interfaces. The introduction of additional interfaces and 10 
nodes, and the decoupling of hardware and software, expands the threat and attack surface of the network.  Threat 11

Threat ID T-OCLOUD-01 
Threat title An attacker compromises VNF/CNF images and embedded secrets 
Threat 
description 
Because images are effectively static archive modules that include all the components used to run a given O-RAN 
VNF/CNF, modules within an image may be having vulnerabilities, introducing malware, missing critical 
security updates or are otherwise outdated. Thus, could be used to attack other VMs/CNs or hosts within the 
environment. 
In addition, images may also have configuration defects. For example, an image may not be configured with a 
specific user account to “run as” and thus run with greater privileges than needed. 
Because images are just collections of files packaged together, malicious files could be included intentionally or 
inadvertently within them. Thus, could be used to attack other VMs/CNs or hosts within the environment. 
Many O-RAN VNFs/CNFs require secrets to enable authentication, access control and secure communication 
between components. For example, an O-RAN VNF/CNF may need a username and password to connect to a 
backend server/database. Other examples of embedded secrets include connection strings, SSH private keys, and 
X.509 private keys. When an O-RAN VNF/CNF is packaged into an image, these secrets can be embedded 
directly into the image file system. However, this practice creates a security risk because anyone with access to 
the image can easily parse it to learn these secrets. In addition, insufficient authentication and authorization can 
lead to intellectual property loss and expose significant technical details about an O-RAN VNF/CNF image to an 
attacker. Even more critically, because registries of images are typically trusted as a source of valid, approved 
software, compromise of a registry can potentially lead to compromise of downstream VMs/CNs and hosts. 
Images often contain sensitive components like an organization’s proprietary software, and embedded secrets and 
administrator credentials. If connections to registries are performed over insecure channels, man-in-the-middle 
attacks could intercept network traffic and therefore the contents integrity and confidentiality of images may be 
compromised. There is also an increased risk of man-in-the-middle attacks that could intercept network traffic 
intended for registries and steal developer or administrator credentials within that traffic. Thus, could be used to 
provide fraudulent or outdated images to orchestrators, etc.

Threat title DoS attacks on cognitive radio networks [22] 
Threat 
description 
Cognitive radio (CR) technology, which is designed to enhance spectrum utilization, depends on the success of 
opportunistic access, where unlicensed secondary users (SUs) exploit spectrum void unoccupied by primary users

Chapter 6 Security principles 1 
This chapter consists of a set of security principles that the O-RAN system should achieve. They provide high level and 2 
abstract statement of the intended solution to countering potential Threats.  In the future, this chapter is expected to be 3 
normative, therefore these security principles will be developed and refined into security requirements, 4 
recommendations and potential countermeseares in order to clearly identify what is required (SHALL), recommended 5 
(SHOULD) or optional (MAY). In the current version, only ‘SHOULD’ is used within the security principl es to make 6 
recommendations.  7 
6.1 Principles (SP) 8 
6.1.1 SP-AUTH Mutual Authentication  9 
• Mutual authentication SHOULD be established to allow the O-RAN system verifying who performs what, thus 10 
possible to detect fake base stations, unauthorized or malicious components, malicious applications and 11 
malicious users/administrators. To authenticate each component, a unique identifier and one  or more 12 
credentials that SHOULD be kept secret are needed. 13 
6.1.2 SP-ACC Access Control  14 
• The O-RAN system SHOULD forbid unauthorized administrators or components to access O-RAN resources 15 
or services anytime and anywhere. Access controls are required for: 16 
o Network Access Controls for filtering unauthorized/unexpected traffic in the O -RAN components 17 
over their interfaces. 18 
o Access controls to restrict access to component configurations. 19 
o Access controls for hardware to maintain the trust chain. 20 
6.1.3 SP-CRYPTO Secure cryptographic, key management and PKI  21 
• Well-known, standardized, secure and unbroken cryptographic schemes and protocols SHOULD be used. 22 
Proprietary schemes and protocols SHOULD be avoided.  23 
• A secure key management of O-RAN keys (KgNB, KRRC-enc, KRRC-int, KUP-int, and KUP-enc, ksn) 24 
SHOULD be implemented to manage all the steps of key lifecycle: key generation using an appropriate level 25 
of entropy from a reliable source, secure key storage, key rotation and revocation, secure key destruction, e tc. 26 
• Reliable PKI for authentication and data encryption SHOULD be used. Public CAs SHOULD be supported. 27 
The certificates SHOULD be issued by a trusted or rooted Certificate Authority (CA). The CA implements the 28 
Certificate Policy which specifies the rules and policies about who may or may not receive a Certificate. 29 
Relying parties can access the Certificate Policy to determine what validation/verification checks were 30 
performed prior to certificate issuance.  31 
• Each O-RU SHOULD be configured with lists of algorithms which are allowed for usage. There SHOULD be 32 
one list for integrity algorithms, and one for ciphering algorithms. These lists SHOULD be ordered according 33 
to a priority decided by the operator. 34 
6.1.4 SP-TCOMM Trusted Communication  35 
• Integrity, confidentiality, availability, authenticity and replay protection of resources SHOULD be ensured in 36 
transit (see ‘Critical Assets) over O-RAN interfaces according to industry best practices. 37

6.1.5 SP-SS Secure storage 1 
• Integrity, confidentiality, availability protection of resources SHOULD be ensured at rest (see ‘Critical Assets) 2 
according to industry best practices. Special attention SHOULD be paid to the protection of private keys, 3 
certificates and hash-values at rest. 4 
6.1.6 SP-SB Secure boot and self-configuration 5 
• O-RAN components SHOULD secure their underlying firmware and configuration to provide the opportunity 6 
for trust to be extended higher in the software stack. Verified platform firmware can, in turn, verify the 7 
operating system (OS) boot loader, which can then verify other softwar e components all the way up to the OS 8 
itself, the hypervisor or container runtime layers and O-RAN components. The transitive trust SHOULD be 9 
consistent with the concept of the chain of trust (CoT)-a method where each piece of code in the boot process 10 
measures and checks the signature of the next stage of the boot process before the software boots.  11 
• The secure boot process, signature verification and self-configuration SHOULD be securely and properly 12 
implemented for all O-RAN components to authenticate them before loading. 13 
6.1.7 SP-UPDT Secure Update 14 
• A secure update management process SHOULD be implemented for introducing a new component or software 15 
change into the O-RAN system. The process SHOULD consider the ability to update the cryptographic 16 
algorithms and to adapt to upcoming O-RAN security challenges. O-RAN vendors SHOULD constantly 17 
provide ongoing security monitoring and patches for all of O-RAN components. A timely update cycles 18 
if vulnerabilities are discovered SHOULD be in place. 19 
6.1.8 SP-RECO Recoverability & Backup 20 
• Recoverability process in terms of the capacity of recovery in case of denial of service SHOULD be 21 
implemented. An approach for detecting and mitigating DoS attacks SHOULD be in place. 22 
• O-RAN vendors SHOULD define a recovery plan that resets the O-RAN components to a trustworthy state in 23 
case of a malfunction or an attack (e.g. DoS). 24 
• Backup systems SHOULD be in place to allow data or component on the O-RAN to be secured. Backup 25 
systems SHOULD ensure a suitable level of data or component availability and reliability. 26 
6.1.9 SP-OPNS Security management of risks in open source components 27 
• Vendors using open source code SHOULD enhance its security by applying industry coding best practices. It 28 
is recommended that vendors practice a higher level of due diligence for exposure to public exploits when 29 
using Open Source code.  30 
• A Software Bill of Materials (SBOM) SHOULD be maintained to understand which open source components 31 
are in use and where. 32 
• Security Analysis (Audit, vulnerability scan, etc.) SHOULD be performed to ensure all identified components 33 
are free of security vulnerabilities. 34 
• Because of the pull operation mode of the open source software, a proper policy and process SHOULD be  in 35 
place for identifying and patching known issues with the open source components. Open source software 36 
components SHOULD be kept up to date and patched. 37

6.1.11 SP-PRV Privacy 9 
• In O-RAN, the privacy of end users SHOULD be considered. The privacy of end users can be divided into data 10 
privacy, identity privacy and personal information privacy. Most Communication services are to gather data 11 
and personal information around end users themselves, which may reveal i nformation sensitive to their 12 
privacy. Adversaries would further extract more personal information about end users, such as UE priority, 13 
location information, trajectory, and preference. 14 
6.1.12 SP-SLC Continuous security development, testing, logging, monitoring 15 
and vulnerability handling 16 
• Continuous development and continuous integration (CD/CI) with continuous regression testing and software 17 
security auditing SHOULD be implemented. 18 
• Relevant activities events SHOULD be logged and logs collected SHOULD be analyzed in real time for the 19 
identification of potential security attacks and for security auditing.  20 
• Continuous monitoring SHOULD be implemented to verify that the wanted security state is maintained 21 
throughout the lifecycle of deployed O-RAN components. 22 
• Vulnerability management SHOULD be in place with intelligence to continuously track, identify and 23 
remediate vulnerable applications. Vendors SHOULD keep track of any new vulnerabilities discovered and is 24 
ready to act on customer product security incidents and reported security issues affecting O-RAN components. 25 
6.1.13 SP-ISO Robust Isolation 26 
• In a multi-vendor environment, intra-domain host isolation SHOULD be enforced. In the same host, VMs, 27 
CNs, virtualization/container layer, CPU, storage, and network security isolation of resources SHOULD be 28 
ensured by implementing system security orchestration, segmentation, lifecycle management, time scheduling, 29 
monitoring and audit on the management, signaling, control and data planes, and the execution of virtualized 30 
O-RAN components. 31 
6.1.14 SP-PHY Physical security 32 
• The O-RAN system SHOULD be located at physically secure environment in a way that minimizes the risk of 33 
resource theft and destruction. It SHOULD support secure storage of sensitive data (cryptographic keys and 34 
configuration data), execution of sensitive functions (encryption/decryption, authentication), and execution of 35 
boot and update processes. 36 
• Special attention SHOULD be paid to the site intrusion and physical access threats against O-RU sites. 37 
Consequently O-RU equipment SHOULD disable all unnecessary physical and logical ports, protocols and 38 
interfaces. In addition, secure physical connections to O-RU for O&M operations SHOULD be implemented 39 
(e.g. secure laptop with secure credentials). 40

6.1.15 SP-CLD Secure cloud computing and virtualization 1 
• Defense methods SHOULD be implemented: virtual machine-based intrusion detection, virtual machine-based 2

• To get a fully trusted virtualized application, all the layers in the stack from hardware to firmware to 5 
virtualized software SHOULD be trust, as it is impossible to protect a virtual machine or containers from the 6 
host system. 7 
6.1.17 SP-ROB Robustness 8 
• The O-RAN system SHOULD not only ensure the robustness of software or hardware resources, but also 9 
guarantee the robustness of the cognitive radio channel for meeting the QoS of communication services 10 
required by users. In some scenarios, the robustness of spectrum sensing SHOULD be enhanced when some 11 
sensing nodes (e.g. O-RU) are easily malfunctioned. Robustness is an essential consideration for overcoming 12 
the security threats caused by jamming, DoS or DDoS attacks.  13
