1.1.1 Compatibility of A1 versions 17 
The version number of the present document indicates that there may be implications for the compatibility between A1 18 
implementations in non/near-RT RICs that are based on different versions of this specification. 19 
An incremented first digit of this specification could indicate that a new major feature (e.g. new A1 service) has been 20 
added or that an incompatible change has been made to an A1 service. An incremented second digit could indicate that 21 
an optional feature has been added, or that clarifications or corrections have been made. 22 
The compatibility of A1 implementations in non/near-RT RICs depends on which A1 services that are implemented and 23 
which version(s) of each A1 service that are implemented. The version of an A1 service is indicated by the API version 24 
in the URI (see chapter 4) and compatibility is governed by the version of the OpenAPI document for the A1 service 25 
(see Annex A). 26

Chapter 2 A1 Application Protocol 1 
This document contains a REST method realization of the interface architecture and policy procedures defined in [3]. It 2 
is based on HTTP transport as defined in [4]. 3 
This definition of the A1 Application Protocol (A1AP) is based on the 3GPP service framework for network functions 4 
[5]. It corresponds to a REST-based Solution Set and is based on the structure in the 3GPP specification 29.501 [6] and 5 
the related template [7]. The design patterns for HTTP procedures are based on [8] and the design patterns for JSON 6 
objects are based on [9]. 7

3.2.1.2 Policy representation 15 
The following principles are used for A1 policies when JSON is used as resource representation format : 16 
- A policy corresponds to a resource (in the REST sense); 17 
- A policy is represented as a JSON object referred to as a policy object; 18 
- A policy object contains a scope identifier and at least one policy statement (e.g. one or more policy objective 19 
statements and/or one or more policy resource statements); 20 
- A policy is identified by a policyId that is included in the URI when operation is for a single policy; 21 
- The policyId is assigned by the A1-P Consumer when the policy is created; 22 
- The A1-P Producer cannot modify or delete a policy; 23 
- Policy feedback for a specifc policy is subscribed to when the policy is created by providing a callback URI in 24 
the create policy operation; 25 
- A policy object does not contain any information related to which internal function in the near -RT RIC that is 26 
to evaluate the policy;  27

The following JSON objects are used within the service operations of the A1 -P service: 5

ProblemDetails 12 
The problem details object is the JSON representation of the content in a response message with other HTTP error 13 
response codes (4xx/5xx). 14

3.2.2.1 Introduction 2 
The following sections describe the policy operations. For details on the policy objects ( in JSON format) transferred in 3 
the HTTP message bodies, see Annex B. 4 
The policy scope in a policy object contains a scope identifier that can be e.g. a ueId, a groupId or a cellId. The A1-P 5 
Consumer needs to map policyIds to scope identifiers in order to manage e.g. all policies applicable to a specific 6

made to manage that specific individual in a sequence of single policy operations. 8 
The A1-P Producer allows the A1-P Consumer to create policies of specific types and the A1-P Consumer can discover 9 
the supported policy types using the Query policy type procedures. The A1-P Consumer then indicates the policy type 10

This Annex specifies the formal definition of the A1 API(s). It consists of OpenAPI documents in YAML format that 3

A.1.1 Versioning of A1 OpenAPI documents 7 
The OpenAPI documents for the A1 services found in this chapter are versioned by Semantic Versioning 2.0.0 [24] as 8 
described in the OpenAPI Specification [23]. When included in the present specification, the OpenAPI documents are 9 
considered as released and are versioned using three digit major.minor.patch where the main compatibility expectations 10 
stated in [24] implies: 11 
major version is stepped up when incompatible API changes are made to the OpenAPI document. This corresponds 12 
to saying that implementations of an A1 service in non/near-RT RICs are incompatible in case the API version is 13 
different. The major version in the OpenAPI document corresponds to the A PI version in the URI for of the A1 14 
service defined in chapter 4. 15 
minor version is stepped up when features are added to the OpenAPI document in way that keeps implementations 16 
compatible although all features are not supported by both the service producer and the service consumer of the A1 17 
service. 18 
patch version is stepped up when errors are corrected in a backward compatible way, when or editorial changes are 19 
made to the OpenAPI document, but no features are added.  20 
Note: non/near-RT RIC products that implement various API versions of an A1 service can be compatible by 21 
supporting several API versions of an A1 service. The present specification specifies only one API version, and 22 
contains only one OpenAPI document, for each A1 service. 23
