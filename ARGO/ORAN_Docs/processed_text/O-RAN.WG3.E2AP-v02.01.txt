1 Scope 1 
The present document specifies the Near-RT RIC layer signalling protocol for the E2 interface.  2 
The E2 interface provides means for interconnecting a Near-RT RIC and an E2 Node. The E2 Application Protocol 3 
(E2AP) supports the functions of E2 interface by signalling procedures defined in the present document. E2AP is 4 
developed in accordance to the general principles stated in O-RAN E2 General Aspects & Principles [2]. 5

3 Definitions and Abbreviations 10 
3.1 Definitions 11 
For the purposes of the present document, the terms and definitions given in  3GPP TR 21.905 [1] and the following 12 
apply.   13 
A term defined in the present document takes precedence over the definition of the same term, if any, in 3GPP TR 14 
21.905 [1]. 15 
A1: Interface between non-RT RIC and Near-RT RIC to enable policy-driven guidance of Near-RT RIC 16 
applications/functions, and support AI/ML workflow [4].  17 
E2: Interface connecting the Near-RT RIC and one or more O-CU-CPs, one or more O-CU-UPs, and one or more O-18 
DUs [2]. 19 
E2 Node: a logical node terminating E2 interface. In this version of the specification, ORAN nodes terminating E2 20 
interface are: 21 
- for NR access: O-CU-CP, O-CU-UP, O-DU or any combination as defined in [9];  22 
- for E-UTRA access: O-eNB. 23 
non-RT RIC (O-RAN non-real-time RAN Intelligent Controller): a logical function that enables non-real-time 24 
control and optimization of RAN elements and resources, AI/ML workflow including model training and updates, and 25 
policy-based guidance of applications/features in Near-RT RIC. 26 
Near-RT RIC (O-RAN near-real-time RAN Intelligent Controller): a logical function that enables near-real-time 27 
control and optimization of RAN elements and resources via fine-grained (e.g. UE basis, Cell basis) data collection and 28 
actions over E2 interface. 29 
O-CU: (O-RAN Central Unit): a logical node hosting RRC, SDAP and PDCP protocols [7]. 30 
O-CU-CP: (O-RAN Central Unit – Control Plane): a logical node hosting the RRC and the control plane part of the 31 
PDCP protocol [7]. 32 
O-CU-UP: (O-RAN Central Unit – User Plane): a logical node hosting the user plane part of the PDCP protocol and the 33 
SDAP protocol [7]. 34 
O-DU: (O-RAN Distributed Unit): a logical node hosting RLC/MAC/High-PHY layers based on a lower layer 35 
functional split. 36 
O-eNB: an eNB [6] or ng-eNB [18] that supports E2 interface. 37 
O-RU: (O-RAN Radio Unit): a logical node hosting Low-PHY layer and RF processing based on a lower layer 38 
functional split.  This is similar to 3GPP’s “TRP” or “RRH” but more specific in including the Low -PHY layer 39 
(FFT/iFFT, PRACH extraction).   40

4 General  17 
4.1 Procedure Specification Principles 18 
The principle for specifying the procedure logic is to specify the functional behaviour of the terminating node exactly 19 
and completely. Any rule that specifies the behaviour of the originating node shall be pos sible to be verified with 20 
information that is visible within the system. 21 
The following specification principles have been applied for the procedure text in clause 8:  22 
- The procedure text discriminates between: 23 
1) Functionality which "shall" be executed. 24 
 The procedure text indicates that the receiving node "shall" perform a certain function Y under a certain 25 
condition. If the receiving node supports procedure X but cannot perform functionality Y requested in the 26 
REQUEST message of a Class 1 EP, the receiving node shall respond with the message used to report 27 
unsuccessful outcome for this procedure, containing an appropriate cause value.  28 
2) Functionality which "shall, if supported" be executed. 29 
 The procedure text indicates that the receiving node "shall, if supported," perform a certain function Y under 30 
a certain condition. If the receiving node supports procedure X, but does not support functionality Y, the 31 
receiving node shall proceed with the execution of the EP, possibly informing the requesting node about the 32 
not supported functionality. 33 
- Any required inclusion of an optional IE in a response message is explicitly indicated in the procedure text. If the 34 
procedure text does not explicitly indicate that an optional IE shall be included in a response message, the 35 
optional IE shall not be included. For requirements on including Criticality Diagnostics  IE, see clause 10. 36

4.2 Forwards and Backwards Compatibility 1 
The forwards and backwards compatibility of the protocol is assured by mechanism where all current and future 2 
messages, and IEs or groups of related IEs, include ID and criticality fields that are coded in a standard format that will 3 
not be changed in the future. These parts can always be decoded regardless of the standard version.  4 
4.3 Specification Notations 5 
For the purposes of the present document, the following notations apply: 6 
Procedure When referring to an elementary procedure in the specification the Procedure Name is written with 7 
the first letters in each word in upper case characters followed by the word "procedure", e.g. 8 
Handover Preparation procedure. 9 
Message When referring to a message in the specification the MESSAGE NAME is written with all letters 10 
in upper case characters followed by the word "message", e.g. HANDOVER REQUEST message.  11 
IE When referring to an information element (IE) in the specification the Information Element Name  12 
is written with the first letters in each word in upper case characters and all letters in Italic font 13 
followed by the abbreviation "IE", e.g. E-RAB ID IE. 14 
Value of an IE When referring to the value of an information element (IE) in the specification the "Value" is 15 
written as it is specified in the specification enclosed by quotation marks, e.g. "Value".  16 
4.4 Identifiers 17 
For the purposes of the present document, the following identifiers apply: 18 
Global E2 Node ID: Global identifier of an E2 Node.  Defined as the global eNB or gNB identifier and an optional 19 
local identifier of an CU-UP or DU which is required when and if an individual DU or CU-UP supports a direct E2 20 
interface. 21 
Global RIC ID: Global identifier of a Near-RT RIC. 22 
RAN Function ID: Local identifier of a specific RAN Function within an E2 Node that supports one or more RIC 23 
Services using a specific E2 Service Model.  Note that same E2SM may be used by more than one RAN Function in the 24 
same E2 Node.  25 
RAN Function OID: RAN Function Object Identifier.  Used to identify specific RAN function definition (i.e. E2SM 26 
used by specific RAN Function). 27 
RIC Action ID: Local identifier used Near-RT RIC to identify a specific Action within a specific RIC Subscription 28 
Request, used by E2 Node in subsequent RIC Indication messages. 29 
RIC Call Process ID: Local identifier used by E2 Node to identify the suspended associated procedure instance  during 30 
a RIC Service “Insert”, used by Near-RT RIC in subsequent RIC Control procedure. 31 
RIC Request ID: Local identifier used by the Near-RT RIC to identify a specific RIC Subscription procedure or RIC 32 
Control procedure, used by E2 Node in subsequent RIC Indication messages. 33

5 E2AP Services 1 
The present clause describes the services an E2 Node offers to the Near-RT RIC. 2 
5.1 E2AP procedure modules 3 
The E2 interface E2AP procedures are divided into two modules as follows:  4 
1. E2AP Near-RT RIC Functional Procedures; 5 
2. E2AP Global Procedures; 6 
The E2AP Near-RT RIC functional procedures module contains procedures used to pass application specific messages 7 
between Near-RT RIC applications and a target function in an E2 node [2] 8 
The Global Procedures module contains procedures that are not directly related to a specific application. 9 
5.2 Parallel transactions 10 
Parallel transactions, that is, multiple ongoing E2AP procedures related to the same Application and E2 node, are 11 
supported. 12

9.3.1 General 12 
E2AP ASN.1 definition conforms to ITU-T Rec. X.691 [15], ITU-T Rec. X.680 [16] and ITU-T Rec. X.681 [17]. 13 
The ASN.1 definition specifies the structure and content of E2AP messages. E2AP messages can contain any IEs 14 
specified in the object set definitions for that message without the order or number of occurrence being restricted by 15 
ASN.1. However, for this version of the standard, a sending entity shall construct an E2 AP message according to the 16 
PDU definitions module and with the following additional rules: 17 
- IEs shall be ordered (in an IE container) in the order they appear in object set definitions.  18 
- Object set definitions specify how many times IEs may appear. An IE shall appear ex actly once if the presence field 19 
in an object has value "mandatory". An IE may appear at most once if the presence field in an object has value 20 
"optional" or "conditional". If in a tabular format there is multiplicity specified for an IE (i.e., an IE list)  then in the 21 
corresponding ASN.1 definition the list definition is separated into two parts. The first part defines an IE container list 22 

O-RAN.WG3.E2AP-v02.01 
 
Copyright © 2022 by the O-RAN ALLIANCE e.V. Your use is subject to the terms of the O-RAN Adopter License Agreement in the Annex ZZZ 
59 
where the list elements reside. The second part defines list elements. The IE container list appears as an IE of its ow n. 1 
For this version of the standard an IE container list may contain only one kind of list elements.  2 
NOTE: In the above "IE" means an IE in the object set with an explicit ID. If one IE needs to appear more than once 3
