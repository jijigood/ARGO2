Chapter 2 Objectives and scope 1
This security test specification is focused on: 2
- Validating the proper implementation of security protocols requirements specified in [2] . 3
- Emulating security attacks against the O-RAN component(s), interfaces, and the system to measure the 4
robustness of the O-RAN system and the service impact(s). 5
- Validating the effectiveness of the security mitigation method(s) to protect the O -RAN system and the services 6
it offers. 7
- It is based on the priority of the risk assessment of the O-RAN security threats and security requirements of the 8
O-RAN system. 9

3.3 Test and measurement equipment and tools 13
A list of possible test and measurement equipment and tools required to execute the security tests. 14
Test tool Description
Real UE and/or
UE emulator
The UE (Real UE or UE emulator) is used to establish stateful end -to-end connection and to
generate or receive data traffic.
The real UE used in this context as a test tool is typically a UE which is designed for commercial
or testing applications with certain test and diagnostic functions enabled for test and
measurements purposes. Such test and diagnostic functions should not affect the performance.
This real UE requires an (emulated) SIM card which is pre- provisioned with subscriber profiles.
A UE emulator or multiple real UEs can be used in multi- UE test scenarios requiring multiple
UEs sessions. The UE is connected to the SUT either via RF cables or via an over the air (OTA)
connection. In a lab environment, the UE should be placed inside an RF shielded box/room to
avoid interference from external signals.
The logging tool connected to the UE is used to capture measurements and KPI logs for test
validation and reporting.
4G/5G Core or
Core emulator
The 4G/5G core or core emulator is used to terminate 4G/5G NAS sessions, and to support core
network procedures required for RAN (SUT) testing. 4G/5G core or core emulator must support
end-to-end connection and data transfer between Application server and Real UE/UE emulator.
Application
(traffic) server
The application (traffic) server is used as an endpoint for generation and/or termination of various
data traffic streams to/from Real UE(s)/UE emulator. The application server should be capable to
generate data traffic for the services under test.
The application server should be placed as close as possible to the core/core emulator and
connected to the core/core emulator via a transport link with sufficient capacity.
Network
impairment
emulator
The network impairment emulator is used for tests which require insertion of impairment (packet
delay and/or jitter) at the network interface (e.g. OpenFH).
Packet generation
tool / DoS
emulator
The packet generation tool / Denial of Service (DoS) emulator is used for DoS traffic generation
of security tests. The tool must support crafting network traffic on network layers from 2 to 7,
which conform to network protocols such as: Ethernet, IP, UDP, TCP, PTP, eCPRI, TLS,
HTTP/HTTPS. The tool is intended to be dep loyable in various network segments
(communication planes) according to the testing needs.

Packet capture
tool
The packet capture tool is used to capture samples of data traffic for validation, analysis, and
troubleshooting. In the case of security test cases it can be used to capture samples of legitimate
traffic, which then can be used as templates for fuzzing at tacks. The tool must support capturing
network traffic on network layers from 2 to 7, which conform to network protocols such as:
Ethernet, IP, UDP, TCP, PTP, eCPRI, TLS, QUIC, HTTP/HTTPS. The tool is intended to be
deployable in various network segments (communication planes) according to the testing needs.
Network tap A network tap is a hardware or software device which provides access and visibility to the data
flowing across a computer network.
Fuzzing tool The protocol fuzzing tool is used for unexpected protocol input generation of security tests. The
tool must support mutating and replaying of captured network traffic on network layers from 2 to
7, which conform to network protocols such as: Ethernet, IP, UDP, TCP, PTP, eCPRI, TLS,
HTTP/HTTPS.
The tool is intended to be deployable in various network segments
(communication planes) according to the testing needs.
Vulnerability
scanning tool
The vulnerability scanning tool is used for blind exploitation of well-known vulnerabilities during
security tests. The tool should rely on cyclically updated database of known vulnerabilities based
on Common Vulnerabilities and Exposures (CVE) and should support scanning network services
running on TCP/IP stack of protocols. The tool is intended to be deployable in various network
segments (communication planes) according to the testing needs.
NFV
benchmarking
and resource
exhaustion tool
The Network Function Virtualization (NFV) tool is used for O -Cloud system performance
measurement and resource exhaustion type of DoS attack generation. This tool should be able to
be deployed on any types of O-Cloud environment (public or private) with testing VNF(s) and/or
CNF(s) support.
SSH audit tool SSH audit tool is used to verify properties like version of protocol, cipher suites, and known

TLS scanning tool TLS scanning tool is used to verify properties like version of protocol, cipher suites, and known

DTLS scanning
tool
DTLS scanning tool is used to verify properties like version of protocol, cipher suites, and
known vulnerabilities in configuration of server DTLS software.
IKE scanning tool IKE scanning tool is used to verify properties like version of protocol, cipher suites, and known

4.2 STC-Chapter 4-001: SSH Server & Client 5
Requirement Name: Network Security Protocol - SSH  6
Requirement Reference:  Section 2.1, O-RAN Security Protocols Specification 3.0 [2] 7
Requirement Description: Robust protocol implementation with adequately strong cipher suites is being required for 8
SSH 9
Threat References: T-O-RAN-05 10
O-RAN Component References: O-DU, O-RU 11
4.2.1 Test description and applicability 12
The purpose of the test is to verify the proper implementation of the secure communication protocol SSH as specified in 13
[2]. 14
Properties requiring validation (on both: server and client side): 15
- Supported version of SSH protocol (v2) 16
- Robustness of cryptographic algorithms used for/as: 17
1. Host key algorithms (also known as public key signature algorithms) 18
2. Symmetric algorithms for encrypting data (also known as symmetric ciphe rs) 19
3. Key exchange algorithms 20
4. Message authentication codes (MACs) 21
- Lack of existence of well-known vulnerabilities in leveraged SSH client and server-side implementations (e.g. 22
OpenSSH) according to Common Vulnerabilities and Exposures (CVE) listed at [10]  23
 24
4.2.2 Test setup and configuration 25
The test must be executed against both server and client endpoints of communication used by SUT on O -RAN/3GPP 26
defined interfaces. 27
- Test prerequisites: SSH audit tool with capabilities as defined in Chapter 3.3 28
- Network access to SSH server services from tester’s PC (Used for server -side testing) 29
- OS level (shell) access to hosts acting as SSH clients (Used for client-side testing) 30
 31
4.2.3 Test procedure 32
Server-side testing: 33
- Run SSH audit tool in server audit mode against each SSH server service from tester’s PC  34
- Compare the tool’s output with the list of approved SSH protocols and algorithms (for host key, symmetric 35
encryption, key exchange, and MACs) as defined by Security Protocols Specifications 36
- Review the tool’s output for reported vulnerabilities 37
Client-side testing: 38
- Run SSH audit tool on each SSH clients in client audit mode 39
- Compare the tool’s output with the list of approved SSH protocols and algorithms (for host key, symmetric 40
encryption, key exchange, and MACs) as defined by Security Protocols Specifications 41

4.3 STC-Chapter 4-002: TLS 11
Requirement Name: Network Security Protocol - TLS  12
Requirement Reference: Section 2.2, O-RAN Security Protocols Specification 3.0 [2] 13
Requirement Description: Support TLS v1.2 and/or TLS v1.3 with proper protocol profiles  14
Threat References: T-O-RAN-05, T-SMO-01 15

4.3.1 Test description and applicability 17
The purpose of the test is to verify the proper implementation of the secure communication protocol TLS as specified in 18
[2]. 19
Properties requiring validation for the TLS service on O-RAN component(s): 20
- Supported version of TLS v1.2 or TLS v1.3 21
- Proper TLS protocol profiles specs listed in Section 2.2.2 of [2], i.e.: 22
1. Intermediate profile for TLS v1.2 23
2. Intermediate profile for TLS v1.3 24
3. Modern profile for TLS v1.3 25
- Lack of existence of TLS well-known vulnerabilities in TLS implementations (e.g. OpenSSL) according to 26
Common Vulnerabilities and Exposures (CVE) listed [10] 27
 28
4.3.2 Test setup and configuration 29

Requirement Description: A list of network protocols and services supported on the O -RAN component shall be clearly 13

5.2.1.1 Test description and applicability 17
Service enumeration is the discovery of network services that are managed or handled by the DUT. Service 18
enumeration for O-RAN component first requires both internal and external "port scans" to discover ports that are 19
servicing application and network protocols. 20
- TCP Port Scanning 21
o Sending a TCP SYN packet to each port. A response with SYN/ACK indicates "port open"  22
- SCTP Port Scanning 23
o Sending a SCTP INIT packet to each possible port. A response with INIT_ACK indicates "port open" 24
- UDP Port Scanning 25
o Sending a UDP packet containing data to each port. A non-response of ICMP Port Unreachable 26
indicates that a service MIGHT be listening. 27
 28
After each set of ports is scanned, a set of service discovery protocol handshake attempts should be sent to validate if a 29
service is listening or available. For each port, the most likely protocol(s) should be sent to validate if they are present 30
and flagged. If there is a non-match, the data received should be reported in both hexadecimal form and ASCII 31
printable. A few examples: 32
- A TCP port on port 80 is discovered as "open". An HTTP 1.1 GET should be sent, as well as a HTTP2 GET  33
- UDP Port 69 was deemed open. A TFTP read for a random file should be sent, looking for a TFTP ERROR 34
Packet. 35
- SCTP port 38412 was discovered as open. A NG Reset packet should be sent to that port looking for the 36
appropriate response. 37

5.4 Fuzzing 11
Fuzzing is mostly an automated randomized robustness testing technique using malformed data injected to either a DUT 12
or SUT to monitor for potential crashes or deviation from expected behavior in software.  13
Fuzzing is effective for finding vulnerabilities because while most modern programs have extremely large input fields, 14
the test coverage of these areas are relatively small. Pre-release fuzzing is particularly useful, as it prevents malicious 15
actors from using the same tools to find bugs to exploit. Additionally, constantly changing the software base can help 16
detect unexpected bugs early. A major benefit of fuzzing is that it often generates actual positive tests for errors, not just 17
static warnings. 18
Fuzzing can be a powerful capability, but it needs to be sufficiently defined and implemented throughout the system or 19
software lifecycle to be useful as a certification tool in a multivendor environment.  In the case of O-RAN system, 20
fuzzing shall cover the following sets of protocols:  21
