This Technical Specification has been produced by the O-RAN Alliance. 3

The present document describes the Transport Management Plane for the Cooperative Transport Interface (CTI). CTI is 14 
an interface between O-DUs and Transport Nodes of a packet-based transport network that is used to interconnect the O-15 
DUs to a variety of O -RUs. CTI specifically targets transport Nodes that manage a shared point -to-multipoint access 16 
network. Intermediate transport nodes (routers and switches) that only manage point -to-point links do not exchange CTI 17 
messages with the O -DUs. CTI consists of a Transport Control plane (TC) a nd a Transport Management Plane (TM). 18 
This document specifies the TM-plane. The TC-plane is described in [1]. The reader is referred to the TC-plane document 19 
for a description of the protocol as well as background, context, motivation and topologies with CTI.  20 
This document focuses on the management aspect of CTI. The goals are to define the CTI-related configurable parameters 21 
at TN and at O-DU, to define a suitable set of YANG modules to capture these parameters, to describe necessary steps in 22 
the configuration of the parameters, and to define an auto -discovery method to automate the correlation between TN 23 
specific parameters and CTI parameters. 24 
Disclaimer: the CTI configuration parameters and their corresponding YANG modules are subject to evolution in the next 25 
version of this specification. 26 
The wording for requirements in this document follows the guidelines of RFC 2119.  27

• CTI client: a process in the O-DU that exchanges CTI messages with one or multiple CTI servers, e.g. to request 23

• CTI server: a process in the Transport Node that exchanges CTI messages with one or multiple CTI clients, e.g. 25 
to receive capacity requests. 26 
• CTI message sender: CTI client or CTI server generating a CTI message 27 
• CTI message receiver; CTI client or CTI server receiving a CTI message 28

2 High Level Description 1 
2.1 General CTI Management Architecture 2 
2.1.1 Foreword on management domains  3 
This document refers to separate “transport domain OSS” and “ RAN domain SMO”, without any implication of 4 
ownership of the domain s or use of particular processes or protocols in the transport domain OSS. E ach domain is 5 
responsible for managing its nodes and can do so independently, but for CTI some level of mutual coordination is needed. 6 
Note that using two domains in this document does not preclude a lternative approaches such as an integrated OSS for 7 
both domains, or an overarching orchestrat ing function that would select the parameters and forward them to the OSS 8 
domains.  9 
2.1.2 Summary of architecture  10 
As the configuration of CTI parameters in the O-DU is non real-time, it will be done over the O-RAN O1 interface using 11 
NETCONF/YANG. 12 
This document does not make assumptions about the management process in the Transport OSS. 13 
Several CTI parameters are common for O-DU and TN side and must be aligned. There is a need for coordination between 14 
both operation domains. This document specifies the needs, but does not make assumptions about the process or protocols 15 
to ensure such coordination. The only process described in this document is an automated auto -discovery which allows 16 
to link a TN specific transport parameter with each individual CTI session ID. 17 
This document does not make assumptions about the integration of functional features into physical nodes. For example, 18 
the document does not restrict the realization of the RAN domain DHCP server, which may be integrated with the O-DU, 19 
may be provided by the transport system, or may be accessed via a relay. 20 
 21

2.1.3 Parameters to be configured  23 
There are two types of parameters related to CTI: 24 
• Parameters to establish L2 or L2 + L3/L4 connectivity for CTI messages between CTI servers and CTI clients , 25 
to be configured by Transport OSS and by RAN SMO, and their corresponding YANG models 26 
 27 
• Parameters for the operation of CTI, to be configured by Transport OSS and by RAN SMO, and their 28 
corresponding YANG models 29

4 CTI procedures 1 
Proper interpretation of CTI messages by the TNs requires a level of cooperation between the CTI server side and CTI 2 
client side to guarantee consistency of the parameter values and and correlation between the parameters. Although it is 3 
possible to rely on manual correlation actions, it is beneficial to automate the process. This section describes an automated 4 
autodiscovery method of the correlation of the CTI session IDs with the TU/TN transport parameter they correspond to.  5 
The same method can also be used to determine which CTI server must be associated per gi ven CTI session ID. 6 
4.1 Correlations to be performed 7 
For proper operation of CTI, several parameters need to be consistent at both TN and at O-DU sides. This means that the 8 
same values must be consistently configured at both CTI server and CTI client side by t heir respective management 9 
systems. The method or protocols for this coordination between Transport OSS and RAN SMO are out of scope of this 10 
document. 11 
Static parameters such as timer values, CTI Flow ID filters, CTI Pattern IDs etc… can be coordinated at O SS level 12 
independently of the topology of the O-RUs and transport nodes. 13 
Dynamically allocated parameters such as IP addresses of CTI servers and CTI clients are dependent of the 14 
interconnection topology of TNs and O -DUs, but not of the topology of the O -RUs. They can be coordinated at OSS 15 
level. 16 
The topology of O-RU deployments will require the following correlations for every CTI session ID, based which TU 17 
interface each O -RU interface is connected to  (see Figure 4-1 for a logical view of the correlations. This figure is 18 
conceptual only and does not represent how or where the correlation must be executed (in the TN or in its management 19 
system)): 20 
• The TN mus t determine  which transport parameter must be linked to the corresponding CTI session ID 21 
(combined with a CTI flow ID if used) in the CTI report messages (note that multiple transport parameters could 22 
be configured on a TU interface, only one pertains to a  combination CTI session ID + CTI flow ID).  Note that 23 
CTI flow IDs are only unique per CTI client (different clients could use the same Flow ID for different 24 
meanings), so the CTI client must be taken into account in the correlation. 25 
• The TN must also keep track of which CTI client is reporting the CTI session ID, to send messages related to 26 
that CTI session ID. A single CTI client can report about multiple CTI session IDs.  27 
• At O-DU side, the CTI client must correlate which CTI session ID to use for each O -RU interface it reports 28 
about. This is a one-to-one relationship. For reporting between different flows on the same CTI session ID the 29 
O-DU must also be aware of the class of service associated to each flow, which in turn is represented by a given 30 
set of L2/ L3/L4 filter parameters reflecting how uplink traffic is tagged by the O -RU. This tagging could be 31 
different in different transport networks (a single O -DU can in principle manage O -RUs in multiple transport 32 
networks), so the CTI server (reflecting the TN) must be taken into account in the correlation.  33 
• The O-DU must also know to which CTI server to send the reports for a given CTI session ID. Multiple reports 34 
could have to be sent to the same CTI server. 35 
 36 
Note that it is recommended to use the physical MAC address of the O -RU port over which the O -RU has M -plane 37 
connectivity as the value of the CTI session ID. 38

4.2 Auto Discovery for correlation of CTI session ID to transport 4 
parameter in TN 5 
This optional method is a means to automate the correlation of CTI session ID with the required transport paramet er on 6 
the corresponding TU (or TU UNI) on the corresponding TU or TU UNI . It avoids relying on manual correlation during 7 
the installation phase of the O-RU to a given TU interface. 8 
The method is based on  re-using the DHCP exchange from the O -RU that is performed during start up of the O -RU to 9 
achieve IP connectivity for the M -plane [4]. Both cases of layer 2 and layer 3 connectivity are very similar , using the 10 
DHCP relay agent in the TN to add the agent circuit ID as the key for the correlation. The only differences are the use of 11 
giaddr by the DHCP relay agent in case of layer 3 connectivity, and the mandatory use of IP in the C -plane and U-plane 12 
traffic in case of layer 3 connectivity. 13 
If the M-plane uses IPv4, the DHCP relay agent adds Option 82 [5], which contains the agent circuit ID (sub option code 14 
1), which is a string uniquely identifying the TU interface on which the DHCP request was received.  15 
If the M-plane uses IPv6, the DHCPv6 relay agent adds Option 18 [6], which contains the agent circuit ID. 16 
Additionally, the possible use of DHCP Option 60 [7] and DHCP Option 124 [8] by the O-RU is specified in [4]. Even if 17 
such options are not used, the O-RU interface will be discovered by the RAN SMO during the autoconfiguration of the 18 
O-RU through M-plane exchanges. 19 
Once the O-RU is configured, the RAN SMO can correlate the agent circuit ID with the O-RU interface ID based on the 20 
allocated IP address. In turn the O-RU interface is uniquely linked to a CTI session ID. The correlation [agent circuit ID, 21 
CTI session ID ] can then be notified back to the transport OSS, which pushes the correlation to the corresponding TN 22 
that terminates the given TU. 23 
It is recommended to use the  unique MAC address of the  O-RU interface as the CTI session ID. For IPv4 t his is 24 
communicated to the DHCP server in the field “chaddr” of DHCP option 82. For IPv6 it can be communicated as “client 25 
link layer address” in DHCP v6 Option 79 [9]. If the O -RU does no t use Option 79 in DHCPv6, there needs to be a 26 
preconfigured association [O-RU ID; MAC address] in the RAN SMO. 27 
The following sections 4.2.1 and 4.2.2 supersede section 3.2.1.2 in [1]. 28

5 CTI YANG Model Definition 1 
This document specifies the CTI YANG model to be used in the O-DU 2 
This document also proposes a CTI YANG model to be  used in a TN  from a generic perspective, with a  specific 3 
declination for PON OLT and DOCSIS CMTS . The TN YANG model is provided here for further distribution towards 4 
the corresponding external bodies for their consideration (Broadband Forum for PON, Cable Labs for DOCSIS). 5

The YANG model follows a modular approach.  7 
Structures that are used in the O-DU and that are also proposed for consideration at the TN are defined in module o-ran-8 
cti-common.  9 
The specific structures for the O -DU are defined in module o-ran-cti-o-du. Together, the modules o-ran-cti-10

cti-tn-docsis respectively. A TN then has to combine o-ran-cti-common, o-ran-cti-tn-generic and one tn-14

5.1 Approach to uniqueness of parameters  20 
Depending on its position in the YANG structure, a parameter can be unique for the whole node, or only within a smaller 21 
context in the node allowing for multiple values to be configured inside the same node. Additionally, if a parameter must 22 
be unique across the whole network, this consistency must be assured by the SMO system to configure the same value in 23 
all its managed nodes. 24 
 25 
CTI Session ID:  26 
The CTI Session ID must be unique across the whole network. Recommendation is to use the MAC address of the O-RU 27 
interface it is linked to. 28 
The CTI session ID element is used as the cornerstone in the TN and O-DU YANG models. 29

Filter values linked to a CTI Flow ID:  9 
From O-DU point of view a Flow ID can either represent the  same filter values over the whole network (all TNs), or the 10 
same ID could represent different filter values in different transport networks (eg if TNs in different transport networks 11 
use different VLAN tagging). Therefore the CTI Flow ID is defined at per-CTI server level in the O-DU.  12 
From the TN point of view a Flow ID can either represent the same filter values over the whole network (all O -DUs), or 13 
the same ID could represent different filter values in different RAN networks, eg if the same ID would be  re-used by 14 
accident by two different RAN networks (sets of O -DUs). Therefore the CTI Flow ID is defined at per -CTI client level 15 
in the TN. 16 
 17 
Pattern characteristics linked to a CTI Pattern ID:  18 
From O-DU point of view a Pattern ID will represent the same pattern characteristics no matter on which TN an O-RU is 19 
connected to. The Pattern ID can therefore be unique at O-DU node level. It is up to the RAN SMO to configure consistent 20 
values across the different O-DUs. 21 
From TN point of view a same Pattern ID could be re -used by accident by two different RAN networks (sets of O-DUs) 22 
while representing different pattern characteristics. The CTI Pattern ID is therefore defined at a per -CTI client level in 23 
the TN, allowing to make such differentiation.  24 
 25 
L2, L3, L4 Parameters for CTI connectivity: 26 
As mentioned in section 3.2, different sorts of connectivity can be used  between pairs of clients and servers. Therefore 27 
the parameters are grouped into a so -called “connectivity profile”, and pair of CTI client – CTI server can have its own 28 
connectivity profile. In the O-DU the connectivity profiles are defined at top level, and each CTI session ID can refer to 29 
a given connectivity profile. The same approach is done in the TN: connectivity profile at top level, and each CTI session 30 
ID can refer to a given one.    31 
Note that there is no distinction between message types (all message between a CTI client and CTI server use the same  32 
connectivity parameters).  33 
 34 
CTI timers 35 
It is assumed that there will be no difference in timer values throughout the network, their values will be unique. The 36 
timers are defined at the top level in the TN and O-DU YANG modules. 37 
 38 
UDP listening port 39 
When UDP/IP is used for CTI transport between a CTI client and a CTI server, all messages between all CTI servers and 40 
all CTI clients over all networks  will use the sam e destination port value. Without a well -known IANA port, a single 41 
value must be consistently configured in all TN and O-DU nodes.  42
