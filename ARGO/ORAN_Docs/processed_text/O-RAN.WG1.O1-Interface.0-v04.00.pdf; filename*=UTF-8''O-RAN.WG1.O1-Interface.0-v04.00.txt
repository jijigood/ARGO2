1.4 Philosophy 170
It is expected that O-RAN Managed Elements, specified in O-RAN Operations and Maintenance Architecture [29], 171
comply with the O1 Interface Specification. 172
The O-RAN O1 management services follow existing standards wherever possible.  The focus of this document is to 173
identify the use cases which conform to existing standards, identify gaps in management services for O -RAN and define 174
needed extensions.  For identified gaps, the goal is to modify the standards to include the needed O-RAN extensions 175
and update the references in this document as the standards evolve to cover the gaps. If extensions and gaps are not 176
specified, it is expected that the management services providers and consumers are conforming to referenced 3GPP 177
specifications. 178
1.5 Open Points 179
As each Management Service is evaluated, the Use Cases and relevant specifications need to be assessed and 180
augmented as needed to support O-RAN. The current list of Use Cases in Chapter 2 below is not exhaustive, and the list 181
of specification references may not be complete. 182
Some Use Cases referred to in the standard may not be applicable to O-RAN and this needs to be addressed, as an 183
exception.  184
Future clarifications may be added to specify when citing a reference whether it is being used as a citation (meaning it 185
will be strictly followed) or as a reference.  More precise terminology may be included as this draft matures. 186
O-RAN Security Task Group (STG) plans to develop an O-RAN security architecture, as well as security guidelines 187
and requirements for all O-RAN Working Groups (WG) and O-RAN entities.  The security architecture, guidelines and 188
requirements will be incorporated into existing O-RAN documents, as appropriate. The O1 Interface Specification 189
intends to comply with the security architecture, guidelines and requirements that are applicable to the O1 interface and 190
will provide appropriate references, when available. 191
It is mandatory for the O-RU to comply with the O-RAN Fronthaul Management Plane Specification [30] for 192
management services.  There is a joint WG1/WG4 Work Item (WI) in progress to determine how the O -RU can support 193
management services in the O1 Interface Specification, in what time frame, and under what conditions.  Futu re versions 194
of the O1 Interface Specification will be updated as necessary to reflect the decisions of this WI.  195

Chapter 2. Management Services 1
2.1 Provisioning Management Services 2
Provisioning management services allow a Provisioning MnS Consumer to configure attributes of managed objects on 3
the Provisioning MnS Provider that modify the Provisioning MnS Provider’s capabilities in its role in end-to-end 4
network services and allows a Provisioning MnS Provider to report configuration changes to the Provisioning MnS 5
Consumer. NETCONF is used for the Provisioning Management Services to Create Managed Object Instance, Delete 6
Managed Object Instance, Modify Managed Object Instance Attributes and Read Managed Object Instance Attributes. 7
A REST/HTTPS event is used to notify the Provisioning MnS subscribed Consumers when a configuration change 8
occurs.   9
Stage 1 Provisioning management services are specified in 3GPP TS 28.531 [3] section 6.3. 10
Stage 2 CM operations and notifications are specified in 3GPP TS 28.532 [4] section 11.1.1. 11
Stage 3 Provisioning operations for YANG/NETCONF solution set are specified in 3GPP TS 28.532 [4] section 12.1.3. 12
Stage 3 CM notifications for RESTful HTTP-based solution set are specified in 3GPP TS 28.532 [4] section A.1.1. 13
IETF reference documents for NETCONF and YANG include RFC 6241, “Network Configuration Protocol 14
(NETCONF)” [33] and RFC 7950, “The YANG 1.1 Data Modeling Language” [34]. 15

2.2.2.1 Description 447

optional filter to retrieve selected AlarmRecords based on the value of attributes in the AlarmRecord.  For example, 453

for the O-RAN Fault Supervision MnS Provider to support: 459

Rationale: There is no Use Case that requires a NF to acknowledge an alarm.  This operation is best done 461
at the SMO level.  If the NF does not support alarm acknowledgement from the MnS Consumer, then the 462
NF must consider cleared alarms as automatically acknowledged so that they may be removed from the 463
AlarmList. 464
 REQ-FSDC_NF-FUN-2 to support manual alarm clearing. 465
Rationale: Manual clearing of alarms is only for ADMC (Automatically Detected, Manually Cleared) 466

3GPP has published an informative Annex B in TS 28.532 [4] providing guidelines for the integration of 3GPP MnS 6
notifications with VES.  This Annex expands on the information provided by 3GPP.  7
When an O-RAN and 3GPP compliant ME supports VES stndDefined events for sending asynchronous notifications, 8
the native 3GPP notification, as defined by 3GPP, is included in the event. 9
A VES common event header, as defined by VES Event Specification v7.2 [31], is added to the notification. 10
In VES, the domain field in the common event header is used to route the event to the proper consumers and to map to a 11
schema for the event payload.  VES Event Specification v7.2 [31] added a new domain field enumeration value called 12
stndDefined that indicates that the event is complying with a schema defined by a standards body.  13
An additional field was added to the VES common event header called stndDefinedNamespace, whi ch contains a valid 14
namespace as defined by the standards body.  This field is only populated when the domain is stndDefined.  3GPP has 15
defined four namespaces in TS 28.532 [4] Annex B; namely 3GPP-Provisioning, 3GPP-Heartbeat, 3GPP-16
FaultSupervision and 3GPP-PerformanceAssurance.  A VES collector uses the stndDefinedNamespace, along with the 17
stndDefined domain, to route the event to the correct consumer.   18
A stndDefined VES event has a field structure called stndDefinedFields, sp ecified in VES Event Specification v7.2 [31].  19
This structure contains three properties: 20
 schemaReference (type = string, format = uri) 21
 data (JSON object which should be identical to the 3GPP notification) 22
 stndDefinedFieldsVersion (type = string, format = enum) 23

The schemaReference, if present, should be used to verify that the notification content is correct.  3GPP is publishing 25
the notification schemas, in JSON format, to a public repository, (https://forge.3gpp.org/rep/sa5) so that schema 26
references can be included in the event.  27
The data element contains the 3GPP notification, in JSON format, as specified in 28.532 [4]. 28

Scenario 7
1. Provisioning Management Service Consumer activates/configures Trace Session on Trace Provider.  This will 8
be accomplished using Provisioning Management services described in Section 2.1 of this document.  9
2. Trace Provider sends a notifyMOIChanges to indicate the new MOI is created. 10
3. Trace Provider needs to establish a connection to the Trace Consumer to set up a streaming connection 11
(streams are active at this time between the Provider and Consumer). This is done using the 12
establishStreamingConnection Operation via an HTTP POST request containing MetaData associated with this 13
Trace Session. 14
4. Trace Consumer responds with an acknowledgement that contains the ConnectionID needed by the Provider 15
when requesting that the connection be upgraded to a WebSocket to support streaming of the trace data. 16
5. Trace Provider requests the upgrade of the connection to a WebSocket using the ConnectionID and an HTTP 17
GET operation.  18
6. Trace Consumer accepts the upgrade and WebSocket is established.  WebSocket will remain connected until 19
the last streaming trace session active on the Trace Provider is ended.  Note in this example, only one 20
streaming trace session is active. 21
7. Trace Provider starts trace session, waiting for triggering event to occur.  22
8. Triggering event occurs and a new trace recording session is started on the Trace Provider.  Each trace 23
recording session has a unique Trace Recording Session (TRS) Reference associated with it.  24
9. Trace Provider sends Trace Data to Trace Consumer indicating that this is the start of a new trace record 25
session. The start trace recording session information is included in the reportStreamingData Operation.  26
10. While this trace record is active, the Trace Provider collects trace data. 27
11. When the reporting timer expires or enough trace data is available, the Trace Provider sends a trace data report 28
to the Trace Consumer containing trace record data for active recording sessions in a trace session.  These 29
records are the payload of the reportStreamingData operation. 30
12. If the timer expires, but no trace data is present on the Trace Provider, the producer will send an alive message 31
to the Trace Consumer to confirm that the session is still active.  The alive message is the payload of the 32
reportStreamingData operation. 33
13. When the criteria for the trace recording session completion occurs (call ends, etc.), the Trace Provider stops 34
collecting data for this trace recording session. 35
14. Trace Provider creates a record that indicates this trace recording session has ended and includes this record in 36
the payload of the reportStreamingData operation to the Trace Consumer. 37
15. Provisioning Management Service Consumer deactivates the trace via procedures defined in section 2.1 of this 38
document.  Deactivation means that the trace data collection should cease, and the Trace P rovider should stop 39
all active trace recording sessions and send data that it has collected up to this point, if any, for each active 40
trace recording to the Trace Consumer. 41
16. Trace Provider initiates the termination of the trace session. 42
17. For each active session, Trace Provider initiates a Stop Trace Recording Session. 43
18. If there are outstanding record(s) for this trace recording session that have not been streamed to the Trace 44
Consumer, Trace Provider sends them as the payload of the reportStreamingData operati on. 45
19. Trace Provider informs the Trace Consumer that this Trace Recording Session has ended by sending the trace 46
record termination via the reportStreamingData Operation. The producer repeats this until all trace recording 47
sessions for this trace session have been terminated. 48
20. In this example, only one stream was on the connection, so the WebSocket should be disconnected.  To 49
accomplish this, Trace Provider sends the Trace Consumer the deleteStream operation indicating that this 50
Trace Session has been terminated. 51
21. When all active Trace Sessions between Trace Provider and Trace Consumer have ended, the WebSocket 52
connection is to be torn down.  In this example, this occurs when this session terminates because only one 53
session is active.  To terminate the session, Trace Provider sends the Trace Consumer the 54
terminateSignalingConnection Operation which is a WebSocket close frame. 55
22. WebSocket connection is torn down. 56
