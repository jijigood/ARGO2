1 Scope 1 
The present document specifies the Near-RT RIC layer signalling protocol for the E2 interface.  2 
The E2 interface provides means for interconnecting a Near-RT RIC and an E2 Node. The E2 Application Protocol 3 
(E2AP) supports the functions of E2 interface by signalling procedures defined in the present document. E2AP is 4 
developed in accordance to the general principles stated in O-RAN E2 General Aspects & Principles [2]. 5

3.1 Definitions 5 
For the purposes of the present document, the terms and definitions given in 3GPP TR 21.905 [1] and the following 6 
apply.   7 
A term defined in the present document takes precedence over the definition of the same term, if any, in 3GPP TR 8 
21.905 [1]. 9 
A1: Interface between non-RT RIC and Near-RT RIC to enable policy-driven guidance of Near-RT RIC 10 
applications/functions, and support AI/ML workflow [4].  11 
E2: Interface connecting the Near-RT RIC and one or more O-CU-CPs, one or more O-CU-UPs, and one or more O-12 
DUs [2]. 13 
E2 Node: a logical node terminating E2 interface. In this version of the specification, ORAN nodes terminating E2 14 
interface are: 15 
- for NR access: O-CU-CP, O-CU-UP, O-DU or any combination as defined in [9];  16 
- for E-UTRA access: O-eNB. 17 
non-RT RIC (O-RAN non-real-time RAN Intelligent Controller): a logical function that enables non-real-time 18 
control and optimization of RAN elements and resources, AI/ML workflow including model training and updates, and 19 
policy-based guidance of applications/features in Near-RT RIC. 20 
Near-RT RIC (O-RAN near-real-time RAN Intelligent Controller): a logical function that enables near-real-time 21 
control and optimization of RAN elements and resources via fine-grained (e.g. UE basis, Cell basis) data collection and 22 
actions over E2 interface. 23 
O-CU: (O-RAN Central Unit): a logical node hosting RRC, SDAP and PDCP protocols [7] 24 
O-CU-CP: (O-RAN Central Unit – Control Plane): a logical node hosting the RRC and the control plane part of the 25 
PDCP protocol [7] 26 
O-CU-UP: (O-RAN Central Unit – User Plane): a logical node hosting the user plane part of the PDCP protocol and the 27 
SDAP protocol [7] 28 
O-DU: (O-RAN Distributed Unit): a logical node hosting RLC/MAC/High-PHY layers based on a lower layer 29 
functional split. 30 
O-eNB: an eNB [6] or ng-eNB [18] that supports E2 interface. 31 
O-RU: (O-RAN Radio Unit): a logical node hosting Low-PHY layer and RF processing based on a lower layer 32 
functional split.  This is similar to 3GPP’s “TRP” or “RRH” but more specific in including the Low -PHY layer 33 
(FFT/iFFT, PRACH extraction).   34 
O1: Interface between orchestration & management entities (Orchestration/NMS) and O -RAN managed elements, for 35 
operation and management, by which FCAPS management, Software management, File management and other similar 36 
functions shall be achieved.  37 
RAN Function: A specific Function in a E2 Node; examples include network interfaces (i.e. X2AP  [8], F1 [12], S1AP 38 
[13], Xn [11], NGc [10]) interfaces and RAN internal functions handling UEs, Cells, etc. 39

4 General  14 
4.1 Procedure Specification Principles 15 
The principle for specifying the procedure logic is to specify the functional behaviour of the terminating node exactly 16 
and completely. Any rule that specifies the behaviour of the originating node shall be p ossible to be verified with 17 
information that is visible within the system. 18 
The following specification principles have been applied for the procedure text in clause 8:  19 
- The procedure text discriminates between: 20 
1) Functionality which "shall" be executed. 21 
 The procedure text indicates that the receiving node "shall" perform a certain function Y under a certain 22 
condition. If the receiving node supports procedure X but cannot perform functionality Y requested in the 23 
REQUEST message of a Class 1 EP, the receiving node shall respond with the message used to report 24 
unsuccessful outcome for this procedure, containing an appropriate cause value.  25 
2) Functionality which "shall, if supported" be executed. 26 
 The procedure text indicates that the receiving node "shall, if supported," perform a certain function Y under 27 
a certain condition. If the receiving node supports procedure X, but does not support functionality Y, the 28 
receiving node shall proceed with the execution of the EP, possibly informing the requesting node about the 29 
not supported functionality. 30 
- Any required inclusion of an optional IE in a response message is explicitly indicated in the procedure text. If the 31 
procedure text does not explicitly indicate that an optional IE shall be included in a res ponse message, the 32 
optional IE shall not be included. For requirements on including Criticality Diagnostics IE, see clause 10. 33

4.2 Forwards and Backwards Compatibility 1 
The forwards and backwards compatibility of the protocol is assured by mechanism where all current and future 2 
messages, and IEs or groups of related IEs, include ID and criticality fields that are coded in a standard format that will 3 
not be changed in the future. These parts can always be decoded regardless of the standard version.  4

5 E2AP Services 1 
The present clause describes the services an E2 Node offers to the Near-RT RIC. 2 
5.1 E2AP procedure modules 3 
The E2 interface E2AP procedures are divided into two modules as follows:  4 
1. E2AP RIC Functional Procedures; 5 
2. E2AP Global Procedures; 6 
The E2AP RIC functional procedures module contains procedures used to pass application specific messages between 7 
Near-RT RIC applications and a target function in an E2 node [2] 8 
The Global Procedures module contains procedures that are not directly related to a specifi c application. 9 
5.2 Parallel transactions 10 
Parallel transactions, that is, multiple ongoing E2AP procedures related to the same Application and E2 node, are 11 
supported. 12

8.2.1.2 Successful Operation 1 
 2 
Figure 8.2.1.2-1: Near-RT RIC Subscription procedure, successful operation 3 
The Near-RT RIC initiates the procedure by sending the RIC SUBSCRIPTION REQUEST message containing a 4 
unique RIC Request ID to the target E2 Node. When the Near-RT RIC sends the RIC SUBSCRIPTION message, it 5 
shall start the timer TRICEVENTcreate. 6 
At reception of the RIC SUBSCRIPTION REQUEST message the target E2 Node shall: 7 
- Determine the target function using the information in the RAN Function ID IE and configure the requested 8 
event trigger using information in the RIC Subscription Details IE. 9 
- If one or more Report, Insert and/or Policy RIC service actions are included in the RIC Subscription Details IE 10 
then the target function shall validate the event trigger and requested action sequence and, if accepted, store the 11 
required RIC Request ID, RIC Event Trigger Definition IE and sequence of RIC Action ID IE, RIC Action Type 12 
IE, RIC Action Definition, RIC Subsequent Action IE. 13 
If the requested trigger and at least one required Action are accepted by the E2 Node, the target E2 Node shall reserve 14 
necessary resources and send the RIC SUBSCRIPTION RESPONSE message back to the Near-RT RIC. The target E2 15 
Node shall include in the response message the Actions for which resources have been prepared at the target E2 Node in 16 
the RIC Actions Admitted List IE. The target E2 Node shall include the Actions that have not been admitted in the RIC 17 
Actions Not Admitted List IE with an appropriate Cause IE value. 18 
Upon reception of the RIC SUBSCRIPTION RESPONSE message the Near-RT RIC shall stop the timer TRICEVENTcreate 19 
and terminate the Subscription Request procedure. 20 
8.2.1.3 Unsuccessful Operation 21 
 22 
Figure 8.2.1.3-1: Near-RT RIC Subscription procedure, unsuccessful operation 23 
If the target E2 Node does not admit at least one requested Action, detects an inconsistency in the sequence of Action 24 
and Subsequent Action definitions, or a failure occurs during the RIC Subscription procedure, the target E2 Node shall 25 
send the RIC SUBSCRIPTION FAILURE message to the Near-RT RIC. The E2 Node shall include the RIC Actions 26 
Not Admitted List IE with an appropriate Cause IE value. 27 
Upon reception of the RIC SUBSCRIPTION FAILURE message the Near-RT RIC shall stop the timer TRICEVENTcreate 28 
and terminate the Subscription Request procedure. 29

8.2.1.4 Abnormal Conditions 10 
If the target E2 Node receives a RIC SUBSCRIPTION REQUEST message containing RIC Subscription IE that does 11 
not align with the Near-RT RIC Service Model [3], the target E2 Node shall send the RIC SUBSCRIPTION FAILURE 12 
message to the Near-RT RIC. 13 
If the target E2 Node receives a RIC SUBSCRIPTION REQUEST message which contains a RAN Function ID IE that 14 
was not previously announced as a supported RAN function in the E2 Setup procedure and RIC Service Update 15 
procedure, the target E2 Node shall send the RIC SUBSCRIPTION FAILURE message to the Near-RT RIC. 16 
If the target E2 Node receives a RIC SUBSCRIPTION REQUEST message containing identical contents, that is, same 17 
RAN Function ID IE, same RIC Event Trigger Definition IE and same sequence of actions, the target E2 Node shall 18 
send the RIC SUBSCRIPTION FAILURE message to the Near-RT RIC. 19

8.2.2.1 General 21 
This procedure is used to delete E2 subscriptions on E2 Node. 22 
8.2.2.2 Successful Operation 23 
 24 
Figure 8.2.1.2-1: Near-RT RIC Subscription Delete procedure, successful operation 25 
The Near-RT RIC initiates the procedure by sending the RIC SUBSCRIPTION DELETE REQUEST message to the 26

When specifying information elements which are to be represented by bit strings, if not otherwise specifically stated in 6 
the semantics description of the concerned IE or elsewhere, the following principle applies with regards to the ordering 7 
of bits: 8 
- The first bit (leftmost bit) contains the most significant bit (MSB); 9 
- The last bit (rightmost bit) contains the least significant bit (LSB); 10 
- When importing bit strings from other specifications, the first bit of the bit string contains the first bit of the 11

ASN.1) 15 
9.3.1 General 16 
E2AP ASN.1 definition conforms to ITU-T Rec. X.691 [15], ITU-T Rec. X.680 [16] and ITU-T Rec. X.681 [17]. 17 
The ASN.1 definition specifies the structure and content of E2AP messages. E2AP messages can contain any IEs 18 
specified in the object set definitions for that message without the order or number of occurrence being restricted by 19 
ASN.1. However, for this version of the standard, a sending entity shall construct an E2AP message according to the 20 
PDU definitions module and with the following additional rules: 21 
- IEs shall be ordered (in an IE container) in the order they appear in object set definitions.  22 

 
________________________________________________________________________________________________ 
Copyright © 2020 by the O-RAN Alliance e.V. Your use is subject to the terms of the O-RAN Adopter License Agreement in the Annex ZZZ. 41 
ORAN-WG3.E2AP-v01.00.00 
- Object set definitions specify how many times IEs may appear. An IE shall appear exactly once if the presence field 1 
in an object has value "mandatory". An IE may appear at most once if the presence field in an object has value 2 
"optional" or "conditional". If in a tabular format there is multiplicity specified for an IE (i.e., an IE list) then in the 3 
corresponding ASN.1 definition the list definition is separated into two parts. The first part defines an IE container list 4 
where the list elements reside. The second part defines list elements. The IE container list appears as an IE of its own. 5 
For this version of the standard an IE container list may contain only one kind of list elements.  6 
NOTE: In the above "IE" means an IE in the object set with an explicit ID. If one IE needs to appear more than once 7 
in one object set, then the different occurrences will have different IE IDs.  8 
If an E2AP message that is not constructed as defined above is received, this shall be considered as Abstract Syntax 9 
Error, and the message shall be handled as defined for Abstract Syntax Error in subclause 10.3.6.  10
