For the purposes of the present document, the terms and definitions given in the following apply 14
Badging: the process by which interoperability for a pair of products or E2E system integration for a group of products 15
are assessed, and the involved products are authorized to carry a specific set of credentials that indicate successful results. 16
Badging information, which is common to all products involved, will include information on all products (including 17
hardware version and software version) which were involved in interoperability testing or E2E system testing .  18
Blueprint: the set of inputs which is used to describe a specific deployment of O-RAN-based system or solution from 19
several aspects. These aspects include the sp ecification of the O-RAN deployment at the system level (e.g. architecture, 20
performance metrics) as well as subsystem level and the interfaces between the specified subsystems. This would then 21
allow definition and documentation of the testing methodology and the test cases using the blueprint specification. 22
Catalogue: the centralized database providing basic (non-sensitive) information about all issued certificates, IOT badges 23
and E2E badges. Th is catalogue is publicly available at O -RAN ALLIANCE web site a nd maintain ed by O -RAN 24
ALLIANCE. 25
Certification: the process by which compliance with O-RAN interface or reference design specifications for a product is 26
verified, and the product is authorized to carry a specific set of credentials that indicate that it is  conformant. 27
Compliance folder: the set of detailed measurement results demonstrating an O-RAN ALLIANCE device’s compliance 28
to the Badging and Certification Criteria, which includes the ICS, test reports, technical documentation, change 29
management documentation and certification information. 30
Implementation Conformance Statement (ICS) : statement of conformance provided by vendors who bring their 31
DUTs/SUT for badging and/or certification service, which is the key element in determining which test cases will b e 32
executed on a particular product. 33
IOT profile: a specific selection of parameters, optional features, default values and ranges of configurable attributes and 34
mechanisms from O-RAN interface specification(s) that must be supported and used in order to guarantee interoperability 35
between implementations from different vendors. 36

Chapter 2. Certification and Badging Overview 1
2.1 Objectives 2
The motivation of O-RAN certification and badging is to minimize the repetition of fundamental and common tests which 3
should be performed to verify and validate O-RAN compliant products, systems and solutions before their deployment in 4
operator networks thus reduce the test effort. The idea is to test only once and reuse /adopt the test results multiple times 5
whenever it is possible. In addition, the use of O -RAN certificates and badges for network components can be a very 6
important way to promote the O -RAN ecosystem and to encourage implementation and testing of O -RAN products, 7
systems and solutions and their deployment in operator networks. 8
The purpose of O-RAN certification is to validate and certify that a vendor product is compliant to a related set of O -9
RAN defined interface or reference design specifications through a set of O-RAN defined conformance tests using unified 10
test proc edures and templates for sharing of results. The O-RAN certificate is awarded to a product for one or more 11
interfaces (e.g. O-RU CUSM-OpenFH, O-RU O1, O-DU O1, O-cloud O2). 12
The purpose of O-RAN IOT badging is to prove interoperability of a pair of products  based on the most fundamental 13
and operator neutral (common) features and parameters through a set of O -RAN defined interoperability tests using 14
unified test procedures and templates for sharing of results. In addition, O -RAN IOT badging can also provide an 15
indication of interoperability of a pair of products based on optio nal/advanced and operator specific features and 16
parameters through a set of O-RAN defined interoperability tests using unified test procedures and templates for sharing 17
of results. The O-RAN IOT badge is awarded to both products in the DUT pair and is applicable for an O-RAN interface. 18
The purpose of O-RAN end-to-end badging is to demonstrate and verify that a (multi-vendor) end-to-end system meets 19
minimum requirements in terms of functionality, security and other mandatory features defined in related O -RAN test 20
specifications. In addition, O -RAN end-to-end badging can also provide an indication of end -to-end performance and 21
other optional end-to-end capabilities though a set of O-RAN defined end-to-end tests using unified test procedures and 22
templates and d ata formats for sharing of results. The O-RAN E2E badge is awarded to all products in the SUT and is 23
applicable to the specific scope tested. 24
With certification and badging in place, operator awareness of a vendor O -RAN-based system or solution is expected  to 25
increase, and the operator own testing is expected to reduce because the common tests can be performed only once and 26
would not need to be repeated. 27
O-RAN certification and badging may also be helpful for vendors as it provides some level of evidence that their O-RAN-28
based systems and solutions work, which could be used to promote their systems and solutions to operators. Further, for 29
vendors looking for other vendors to interoperate with, certification and badging may also be beneficial to increase 30
acceptance from other vendors. 31
O-RAN certification and badging should not be used as a substitute for operator-specific testing of O-RAN based systems 32
and solutions prior to deployment in operator networks. O-RAN certificate and badge do not mean all the necessary tests 33
have been performed and passed,  and O-RAN certified/badged system or solution can be automatically deployed in an 34
operator network  without any additional testing.  Rather, O -RAN certification and badging can improve operator 35
confidence in their chosen O-RAN based blueprint and can reduce the complexity and duration of pre-deployment testing.  36

2.2 Scope 1
The certification and badging described in this document defines the processes and associated technical procedures in the 2
following areas: 3
- Compliance verification of DUTs using O-RAN interface or an implementation of O-RAN reference design based 4
on conformance test specifications (conformance certification); 5
- Interoperability assessment of DUT pairs using O-RAN interfaces based on IOT test specifications (IOT 6
badging); 7
- E2E system integration assessment of SUT using O-RAN interfaces based on E2E test specifications ( E2E 8
badging). 9
The reason to use the term “ certification” for conformance testing and “badging” for interoperability and E2E system 10
testing is as follows: 11
- Conformance tests are controllable as it only involves a single DUT and verifies whether that DUT behaves in 12
accordance with a technical specification (e.g. open fronthaul interface specification); 13
- IOT and E2E system tests have some variability as it involves multiple DUTs /SUT (from different vendors) and 14
the results may depend on the DUT/SUT combinations, deployment blueprints and test profiles. 15

2.3 Entities Involved 1
2.3.1 O-RAN ALLIANCE 2
In terms of certification and badging, O-RAN Test and Integration Focus Group (TIFG) only specifies the processes and 3
associated detailed technical procedures, identifies the relevant O -RAN test specifications and test cases specified by O -4
RAN WGs/FGs, including end-to-end tests defined by O-RAN TIFG [7].  5
Under no circumstances is the ce rtification/badging entity under the charter or governed/sponsored by O -RAN/TIFG. It 6
is the underlying certification/badging entity that will take full responsibility on all certification/badging related activities 7
including certification/badging implementation, verification/assessment as well as certificate/badge issuance. 8
2.3.2 Certification/Badging entity 9
O-RAN certification and badging can only be performed by approved O -RAN Open Testing and Integration Centers 10
(OTICs), and consequently the O -RAN Certificates and Badges (with O -RAN approved logos for certification and 11
badging) can only be issued by OTICs. In addition, the O-RAN Certificates and Badges (with O -RAN approved logos 12
for certification and badging) can only be awarded to a product provided by O-RAN member, O-RAN contributor, or O-13
RAN academic contributor.  14
OTICs provide a collaborative, open, and impartial working environment, and the criteria and guidelines for OTICs are 15
specified in [1]. These guidelines stipulate that:       16
- O-RAN certification and badging test reports and detailed measurement results can be freely accessed by O -17
RAN operator members for the purpose of evaluating the equipment for use in their networks  18
- An OTIC may or may not choose to provide certification and/or badging services. If an OTIC intends to provide 19
O-RAN certification/badging services, it is mandatory for them to follow the processes and associated technical 20
procedures specified in this document. 21
- Prerequisites for OTICs to be able to provide certification/badging services are defined in [1].  22
O-RAN allows the testing to be also performed outside of the OTICs using O-RAN test specifications and following the 23
certification and badging procedures as defined in this document. However, such test  results cannot carry an O-RAN 24
Certificate or O-RAN Badge with O-RAN approved logos but may state that their equipment has been  "tested according 25
to O-RAN Certification and Badging Process and Procedures document." 26
The certification/badging entity may require certain pre -requisite testing be done by vendors before the start of the 27
certification/badging process. Some examples are illustrated as follows 28
- 3GPP compliance validated by vendor, 3rd party, or the same certification entity as add -on service may be a pre-29
requisite for any O-RAN conformance certification 30
- O-RAN conformance certification by vendor, 3rd party, or the same certification/badging entity as add-on service 31
may be a prerequisite for O-RAN IOT / E2E badging.  32
 33

3.1 General 2
3.1.1 Objectives of defining certification/badging processes 3
The objective is to have in place some level of uniformity on the certification/badging processes i n order to: 4
- Ensure quality of the certification/badging service provided by different certification/badging entities;  5
- Align procedures that users of the certification/badging service (both vendors bringing in DUTs /SUT for 6
certification/badging, and ope rators requesting access to test reports and detailed measurement results of 7
certified/badged DUTs/SUT) experience at different certification/badging entities. 8
3.1.2 Principles 9
The certification/badging process sub-chapter adheres to the following principles: 10
- A distributed approach is adopted in which (a) roles pertaining to certification/badging are allocated to 11
certification/badging entities and (b) related details are left to certification/badging entities’ discretion and 12
responsibility. Rather than having rigorous centralized O-RAN control and management, this approach helps avoid 13
imposing heavy requirements and allows freedom of operation to the certification/badging entities. However, 14
centralized approach will also be adopted where required, e.g. to ensure  quality of certification/badging, sharing 15
of test reports and detailed measurement results, etc, 16
- Defining certification/badging process elements to address near -term needs of the community, while also 17
establishing a framework that can evolve and scale w ith the ecosystem growth , i.e. this sub -chapter can evolve 18
over time. 19
- O-RAN will, as necessary, issue new versions of its functional specifications. As a result, O-RAN may determine 20
that it will launch an updated version of its certification/badging prog ram. 21
- O-RAN will, as necessary, determine that an existing version of its functional specification is no longer viable and 22
should be taken out of service for product development and certification/badging. As a result, O -RAN may 23
determine when the specification is to be taken out of service. 24

6.1 End-to-End assessment overview 2
The O-RAN End-to-End System Test Specification, currently under development in TIFG, is to define the E2E system 3

illustrated in Figure 6.1-1.  5
 6
Figure 6.1-1   The O-RAN system as System under Test (SUT) 7
It is expected that all involved O -RAN functions and interfaces can properly interoperate together, and end -to-end 8
communication link can be established between end -user device and application servers/other end -user devices. It is not 9
mandatory that all O -RAN DUTs /SUTs are conformance certified or recipient of IOT badge. However , if relevant 10
conformance and IOT test cases defined in the chapter 4 and 5 of this document have been identified to be applicable to 11
the underlying DUTs/SUTs, then it is recommended that underlying DUTs/SUTs go through these conformance and IOT 12
testing processes first to avoid unnecessary troubleshooting and prolonged testing period in the end-to-end environment.  13

defines the E2E KPIs across whole end -to-end communication link betwee n end-user device and application (traffic) 15
servers/other end-user devices.  16

assessment and validation. 18
• Assessment of single DUT subsystem in the E2E assessment environment (Subsystem Replacement Testing) 19

6.5 Security  1
This chapter describes the tests evaluating and assessing the security aspects of the E2E of a radio access network.  2
Because the whole O-RAN system is the System under Test (SUT) and can be viewed as an integrated black box in the 3
context of the E2E testing, the security test cases listed here fall into the following categories: 4
1. gNB security assurance specification required by 3GPP SA3. 5
2. Denial of Service (DoS) and fuzzing types of security test cases against the O-RAN system with easy to access 6
IP target information and a routable path. 7
3. Resource exhaustion types of security test case against the underlying cloud infrastructure hosting the O -RAN 8
system and its network slice(s). 9
Exact test cases for security are described in Chapter 7 of E2E Test Specifications [7].  10

6.5.2 DoS and fuzzing types of security test 1

transport protocols and major interfaces of O-RAN system become easy targets for the potential attackers. Security 3
attacks like DoS and fuzzing types are easy to launch, require little information on the target system, and could cause 4
significant performance degradation, or even the service interruption if not properly mitigated. 5
As the WG1 Security Task Group (STG) is continuously working on the O -RAN system security analysis and security 6
requirements specifications, the proposed security test cases in this section will be optional for this E2E test 7
