3 Scope 1
The document is intended to describe best practices for O-RAN Architecture and Solution for X-haul timing 2
& synchronization. Beyond the solutions described in this document, other Architectures and Solutions may 3
be adequate for X-haul timing & synchronization and can be considered in future versions of this document. 4
As far as possible it tries to make no assumptions, rather define overall Open X-haul synchronization solution 5
and architecture model, to enable the operators to understand different synchronization options and deployment 6
models and help them to come up with right network sync model that can support the different 5G services, 7
the different radio architectures.  8
 9
The section 6 concentrates on a general description of timing and synchronization technology, different options 10
using different timing profiles with recommendations . The next section 7 describes the different network 11
models, synchronization budgeting, right use of class of devices for both boundary and grandmaster clocks , 12
solution guidelines for network operations including holdover, redundancy etc. The section 8 describes about 13
timing solution and use cases, redundancy and resiliency network models. The annex section describes other 14
technology aspects like, Microwave, QoS, security, PON etc.  15
 16
This document makes explicit recommendations using keyword “Reco” to insist what is officially being 17
recommended by this specification. 18
 19

other standard bodies and industry associations.  21

6.1 Building blocks of network-based synchronization 3
This section covers different building blocks required for network -based synchronization. This includes 4

6.1.1 Synchronous Ethernet and Enhanced Synchronous Ethernet 6
6.1.1.1 Synchronous Ethernet Clock 7
Synchronous Ethernet Clock is also referred as Ethernet Equipment Clock (EEC). The ITU-T standard G.8262 8
[14] specification defines both synchronous Ethernet Equipment Clock (EEC) and OTN Equipment Clock 9
(OEC).  10
 11
There are two options available for Synchronous Equipment Clocks. The first option, referred to as “Option 12
1”, applies to synchronous equipment designed to interwork with networks optimized for the 2048 kbits/s 13
hierarchy. The second option, referred to as “Option 2”, applies to synchronous equipment designed to 14
interwork with networks optimized for the 1544 kbits/s hierarchy. 15
 16
An EEC recovers the clock at physical layer level. The performance and recovery of clock at physical layer is 17
independent of packet layer. Synchronous is hop by hop clock recovery and drive model. Any one node in the 18
chain is not capable to support SyncE, it is considered to be the clock chain is broken.  19

6.1.2 PRC and clocks 30
The main function of a Primary Reference Clock (PRC), as specified in ITU -T G.811, is to provide the 31
reference signal for the timing or synchronization of other clocks within a network or section of a network, 32
including  the  clock specified in Recommendation ITU-T G.812 within the network nodes where the PRC is 33
located. The long -term accuracy of the PRC is in the order of  1 part in 10 11 or better with verification to 34
Coordinated Universal Time (UTC). PRCs are typically built using Caesium clocks. PRCs are at the top level 35
of the clock hierarchy with one of the highest accuracies [34][35]. 36

6.1.3 PRTC and Grandmaster clocks 38
 The main function of a PRTC, as defined in ITU-T G8272.1 amd1, is to deliver a primary time reference to 39
be used in time and/or phase synchronization by other clocks of the network.  40

can also optionally take a frequency input reference traceable to a PRC to maintain the local representation of 44
the timescale during outages of the input time reference (i.e., extend the phase/time holdover period of the 45
clock). 46

6.3.2 Time Error budget calculation 1
6.3.2.1 General Budgeting model 2
When time error budget is calculated there are three different aspects to that: 3
1. Time Error from Time source (Ex. T-GM) to O-RU (Input to O-RU and up until Radio Interface) 4
2. Time Error from Time source (Ex: T-GM) to O-DU 5
3. Time Error between O-RU to O-RU radio interfaces – Also called as Relative Time Alignment Error 6
(TAE). 7
 8
For each type described above one needs to start with overall time error budget as the end number to start with 9
and calculate back by subtracting the individual budgets for each of the following that are applicable: 10
1. Half of asymmetry (caused by Network, Fiber, Wavelength used or Optics) 11
2. Holdover budget (for Radio, Network nodes, GM or combination of these) 12
3. Number of hops and cTE and dTEL of each of the network nodes based on clock types and time error 13
of T-GM/PRTC based on PRTC type. 14
4. Sync-E/Physical layer clock switchover phase error 15
 16
 17
For example:  18
T : Target Time Error budget (Ex. 1.5 microseconds for TDD network) 19
T(g) :  time error of PRTC+GM 20
T(n) :  time error for all network nodes (boundary clocks) 21
T(r) : time error of Radio device 22
T(h) : holdover timer error budget 23
T(a) : time error budget for asymmetry 24
T(s) : time error budget for SyncE re-arrangement. 25
T(c) : Total calculated time error budget 26
 27
Then, sum of all time errors allocated for GM, network nodes, asymmetry, holdover, SyncE re-arrangement 28
must be less than the Total Target budget (T) to successfully plan and deploy the network (as shown below). 29
 30
      T(c)   = T(g) + T(n) + T(r) + T(h) + T(a) + T(s)  31
 32
Then T(c) < T 33
 34
Note1: If there are multiple PRTC/GMs in the network design the total budget T must not be exceeded 35
whichever path and whichever GM is selected. 36

Note2: Similarly, the time error must be calculated for the longest chain of network pat h/hops rather shortest 38

7 Synchronization network models 2
7.1 Factors to be considered for synchronization network design 3
7.1.1 Source of clock and location of clock source 4
The source of timing should be traceable to a recognized primary time standard such as the Coordinated 5
Universal Time (UTC) or a global navigation satellite system (GNSS) . The GNSS time offset from UTC is 6
contained in the GNSS broadcast message. UTC is the international reference time that is computed by the 7
Bureau International des Poids et Mesures (BIPM)  from hundreds of atomic clocks maintained in national 8
laboratories worldwide. Local representations of UTC, commonly called UTC(k) time scales, are maintained 9
by national measurement institutes and time laboratories.  GNSS uses a constellation of low -orbit satellites 10
that covers the entire Earth’s surface.  11
 12
It should be noted that for the purpose of meeting the 3GPP synchronization requirements (e.g., CPS), there is 13
no need to recover UTC time even when UTC traceability is required. 14
 15
ePRTCs are typically distributed in the 5G Core location to protect their timing networks against regional 16
GNSS and global GNSS outages. An ePRTC system provides in the core of the network an independent and 17
autonomous timescale aligned with GNSS to deliver both frequency, phase and time. 18
 19
PRTCs are typically deployed in a CRAN Hub location where they distribute timing to the Boundary Clocks 20
and O-DUs. The PRTCs can also, in an APTS configuration, receive timing from other PRTCs further down 21
in the pre-aggregation/aggregation network.  22
 23
7.1.2 GM/clock source resiliency 24
Timing is a mission critical service that needs to be protected by designing a highly available timing 25
infrastructure so that no failure will cause the timing service to become unavailable. The timing infrastructure 26
is typically dependent upon GNSS as the timing source. The latter is a single point of failure if the GNSS 27
signal is jammed or spoofed at the PRTC-B location (e.g., Central Office, Mobile Telephone Switching Office 28
-MTSO, etc). It is important to take appropriate measures to mitigate the risks against GNSS failure. 29
 30
Several mechanisms of resiliency can be implemented to ensure the continuity of the timing service: 31
• multi-constellation GNSS to protect against one constellation failure. 32
• GNSS Anti-jamming/spoofing on the PRTC-B GNSS antenna (GNSS failure) 33
• e-PRTC-A to provide up to 14-days of holdover while maintaining up to 100 ns of accuracy. 34
• PRTC-B equipped with Rubidium oscillator to extend the holdover period. 35
• Alternate BTCA (PRTC/T-GM failure) for the timing network to automatically select an alternate 36
PRTC-B in a different location. 37
• High Availability PRTC/T-GM to automatically transfer the IP address of the PRTC-B to another one 38
in a different building. 39
 40
7.1.3 Holdover requirements 41
The duration for which the radio should continue to operate in normal operating mode when  the 42
synchronization clock source is down. This can happen when  GNSS, T -GM or network node in the 43
synchronization path goes down.  44
 45

 41
Major criteria to consider in determining holdover budget requirements: 1
• Regulatory requirement from the government: each country/government may have different 2
requirements as to how long the service should be up and running when there is a GNSS failure. 3
• Operator requirement to meet nominal operation of the service when sync goes down.  4
• How soon an operator can address the sync issue caused by network or GNSS failure. 5
• Sync redundancy model put in place.  6
• How often the GNSS failure may occur? This may be caused by jamming, spoofing or some 7
neighbouring countries planned/unplanned intervention. 8
 9
One number does not fit for all. Each operator needs to carefully plan and determine the required holdover 10
budget. Once required holdover budget is determined, it must be used to calculate  overall synchronization 11
budget from end to end (T-GM to base-station node). 12
Note: CUS specification [33] does not make any explicit recommendation for holdover at Radio/base station 13
 14
Ways to mitigate the holdover condition: 15
• Sync redundancy through the alternate network path in case of network node failure 16
• APTS in case of GNSS used at every cell site. 17
• Alternate flow for PTP and Sync-E in the network path 18
• High stratum oscillator in the end base station 19
• Extended holdover support at source of the sync (Ex. T-GM with extended holdover) 20
 21
7.1.4 Usage of packet rates 22
Based on the network deployment and sync precision requirements of the clock, the PTP packet rates may 23
need to be exercised differently.  24
 25
The factors to be considered in configuring higher packet rate: 26
• High Jitter/PDV in the network 27
• One or more PTP unaware nodes used in the sync network. 28
• Network that is expected to have burst traffic. 29
 30
Different Telecom profiles and packet rate usage: 31
• ITU-T G.8275.1 [1] 32
o Packet rate is fixed for this profile. 33
o 16 Sync, Delay-request and Delay-response and 8 Announce packets per second. 34
o User cannot change the packet rate in this profile mode of operation. Both TimeTransmitter 35
and TimeReceiver clocks shall be able to support and function properly with this packet rate. 36
• ITU-T G.8275.2 [3] 37
o Packet rate is configurable for this profile. 38
o Allows up to 128 packets per second for Sync, Delay-request, Delay-Response and 8 announce 39
packets per second. 40
o Packet rates plays critical role based on the clock recovery algorithm used in this profile mode 41
of operation. 42
 43
Note-1: Packet rate can also affect the bandwidth utilization on the link hence the network. Selecting 44
appropriate packet rate without compromising Sync performance is critical for good network operation. 45

7.1.7 Asymmetry  1
What is asymmetry? 2
• Difference in propagation delay between the forward and reverse path of PTP timeReceiver node from 3
its upstream timeTransmitter node. 4
• Half of uncorrected asymmetry would translate to time error offset in the packet timeReceiver clock 5
from its timeTransmitter clock. 6
 7
Types of asymmetries: 8
• Static asymmetry 9
The (propagation) delay is constant or remains same irrespective of reboot of the node/optics or 10
system. 11
• Dynamic (or semi static) asymmetry 12
The delay is not constant, or it would vary from reboot or reset of the node or interface or optics 13
module. 14
 15
Note: Refer Appendix -III, IV, V of ITU -T G.8271 [7] for general details about asymmetry and how it can 16
impact the time/sync recovery by a timeReceiver clock. 17

7.1.7.1.2 Asymmetry in optics (Grey optics) 21
The propagation delay inside the optics module is not zero. Especially the Tx and Rx may not be equal within 22
a given optics. This introduces static asymmetry within the optics. This is typically seen as small value unlike 23
the fiber asymmetry, but every nanosecond counts for high precision sync requirement. 24

7.1.7.1.3 Wavelength Asymmetry 26
The asymmetry due to the use of different wavelength is obtained by calculating the group delay applicable to 27
wavelengths used in the forward and in the reverse direction. 28
 29
Asymmetry A = df – dr = L * (nr – nf)/c 30
 31
• L is the distance (fiber length) 32
• c is the speed of light. 33
• df and dr are the forward and reverse transmission delay. 34
• nr and nf are the group refractive indexes applicable at the wavelength used in the forward and 35
reverse direction, respectively. 36
 37
The evaluation of the refractive indexes can be done either using known chromatic dispersion data (e.g., from 38
the optical fiber data sheet) or, in the case that the dispersion is unknown, making a direct delay measurement 39
at three different wavelengths (the refractive index for an arbitrary wavelength can then be derived by quadratic 40
interpolation). 41
 42
These data can then be used to derive the group delay of a generic wavelength. In particular, in the case of an 43
ITU-T G.652 compliant fiber, the group delay at the applicable wavelengths can be calculated making use of 44
the Sellmeier equations as described in ITU-T G.652 standard. 45
 46
Note: For additional details refer Appendix-III in ITU-T G.8271 [7] standard and group delay specification 47
and measurement method are specified in G.671 [40], Clause 3.2.2.25. 48

7.1.9 Selection of timing profile 1
In general profile selection is driven based on following criteria: 2
• Target Synchronization Precision requirement 3
o The target precision requirement plays critical role in selection of timing profile.  4
o For high precise application (+/-130 nsec), ITU-T G.8275.1 [1] profile is recommended. 5
o For end to end 1.5usec target – it may be possible to achieve using ITU-T G.8275.2 [3] profile 6
with proper planning and budgeting. 7
o Additionally, FFO limits must be considered. 8
• Transport mechanism used in the transport network. 9
o The usage of L3 versus L2 protocols in the transport nodes sometimes lead to selection of profile 10
as G.8275.1 [1] (L2) or G.8275.2 (L3) [3]. 11
o The end nodes capability to support the specific timing profile will also play an important role in 12
selecting timing for profile for the transport network nodes. 13
• The sync capability of the network nodes used in the transport. 14
o The capability of every node supporting synchronization in the transport network can be a factor 15
in deciding the right profile.  16
o In green field network it is possible to use full timing support profile (G.8275.1 [1]) but in brown 17
field network it might be possible with G.8275.2 [3] profile. 18
• Access to GNSS at cell site and associated CapEx and OpEx leverage 19
o Sync can be delivered directly or close to the base -station nodes based on the accessibility and 20
availability of GNSS either at cell site or close to the cell-site. 21
o When associated cost is not a bigger concern for installation and maintenance of the high number 22
of GNSS/T-GMs in the network. 23
• Network hops in the sync network 24
o In some cases, it may be practically limiting to achieve the targeted sync precision if too many 25
network hops between the T-GM and O-RU (base station nodes). 26
o This can lead to usage of LLS-C4 option with GNSS directly connected to base-station. 27
• Asymmetry and control over asymmetry in the network 28
o If asymmetry (packet, path, link asymmetry) expected to be difficult to control, choosing right 29
profile would play a critical role. 30
o For example – full timing support profile gives better control to address the network asymmetry 31
than partial timing support profiles. 32
• Administrator control of the synchronization network path 33
o Any network hops/cloud(s) in the middle of synchronization path that does not belong to mobile 34
operator administrative control can be risky to deploy sync. 35
o Example – if mobile operator selects full timing support profile, and intermediate cloud/network 36
provider does not support timing in their network, it will break the synchronization chain. 37
 38
Reco: This ORAN specification recommends G.8275.1 [1] full timing support profile and hence plan for 39
the transport network to accommodate the transport mechanism described in this profile (PTP over 40
Ethernet Multicast). 41

7.2.4 Comparison of Centralized versus Distributed GM network model 4
This section describes the disadvantages of distributed and network-based sync models: 5
 6
Direct GNSS to Base station: 7
• Single point of failure 8
• Line of sight access to GNSS can be an issue if it is urban deployment. 9
• Jamming and/or spoofing can bring down the entire cell site down. 10
• No extended holdover possible as it depends on oscillator used in the base-station nodes. 11
• High precision sync application that needs precision between the cluster of cell-sites to be precise with 12
less than or equal to +/-130 or +/-65 nanoseconds (ex: Carrier aggregation, NR MIMO, LTE MIMO) 13
may not be achievable with direct GNSS based sync. 14
Ex1: PRTC-A it is difficult to achieve even +/-130 nsec even with enhanced Radio. 15
Ex2: PRTC-B – it is difficult to achieve +/-65 nsec even with enhanced Radio. 16
•  OpEx and CapEx may be high as it depends on how many base stations deployed. 17
• Cost of monitoring and downtime is high. 18
 19
Network based synchronization using one or few GMs located at centralized location: 20
• Every node in the network must support PTP/SyncE in case of full timing support profile deployment 21
(exception can be O-RU) 22
• Asymmetry in the network can cause time/phase recovery error. 23
• Any node behaves incorrectly in the chain of nodes from GM to base -station, can affect entire chain 24
of downstream nodes performance. 25
• KPI can be complex in wholesale environment especially with G.8275.2 [3] profile deployment case. 26

8.2.4.4  ePRTC Resiliency 1
High availability is one of the key requirements for 5G. It is important to design the 5G synchronization to be 2
highly available with redundancy integrated at all levels of the synchronization infrastructure starting from the 3
source of timing. The resiliency of the timing source is critical to achieve an overall goal of a survivable and 4
fault tolerant synchronization infrastructure. It is essential that the timing infrastructure continues to maintain 5
an acceptable level of service in the event of failures and faults affecting normal operations.  6
 7
There is a growing number of intentional and unintentional GNSS incidents. During such extended GNSS 8
outages, ePRTC-A systems operating as an autonomous primary reference clocks can be deployed to maintain 9
time and frequency service in a geographical area. ePRTC -A delivers a high -level of service reliability to 10
ensure operators maintain an acceptable time and frequency service performance for a long period of GNSS 11
unavailability. The frequency stability of a Cesium atomic clock serves as a reference for the ePRTC-A Time 12
Scale. 13
 14
ePRTC-A offers the following features: 15
• Reliability: Immunity from local jamming or outages  16
• Autonomy: Atomic clock sustained timescale with & without GNSS connection  17
• Holdover: 14-day time holdover <= 100 ns 18
 19
When an ePRTC-A loses all its input phase and time references, it enters the phase/time holdover state and 20
relies on an autonomous primary frequency reference input (e.g., 2MHz, 10MHz, etc.) to deliver time and 21
phase. This autonomous primary reference clock is typically a Cesium atomic clock. Refer Figure 8.2.4-19, 22
for an ePRTC-A, from the start of phase/time holdover, after 30 days of continuous normal operation, the time 23
output should be accurate, when verified against the applicable primary time standard (e.g., UTC), to within a 24
value increasing linearly from 30 ns to 100 ns over a 14-day period. 25

A.3 Impact of interference  2
Like all radio communications systems, Point -to-Point (PP) and Point -to-Multiple-Point (PMP) microwave 3
and mmWave radio transport links are susceptible to radio interference from adjacent and parallel links as well 4
from other services, such as radar, Radio LAN and Short-Range devices, that could be sharing the band or 5
operating in adjacent band to the radio transport fixed service. Communication failure may occur due to the 6
influence of this interference, and the (TE) characteristics may deteriorate accordingly. The severity of TE 7
degradation depends on the equipment and the techniques used to mitigate the impact of interference. As such 8
the design and selection of the bands should take into considering TE degradation due to the external 9
environment. Examples of considerations to mitigate the effect of interference, include avoiding the use of 10
license exempt bands for time sensitive applications, use high quality antenna with very low side lobes (e.g., 11
ETSI Class 4), and apply for high protection and availability link licenses, i.e., 99.99% or over. 12
A.4 Impact of dynamic capacity variations 13
Microwave and mmWave  Point to Point radio systems are designed to operate with high availability by 14
allowing adequate fade margin in their link budget.  The fade margin is calculated to compensate the 15
propagation loss depending on the rain intensity in the area they intended to operate. Modern PP radio systems 16
apply Adaptive Coding and Modulation (ACM) technique to boost its link capacity during the clear sky period 17
taking advantage of the fade margin to apply higher order modulation scheme with higher capacity than the 18
one needed during the worst raining period. This allows the PP radio system to transmit at its maximum 19
transmission capacity allowed by the changing weather condition. This dynamic variations in the modulation 20
method can cause deterioration to the TE, because the data size that is processed by the Modem vary 21
significantly between the different ACM schemes. In general, in ACM technology, the packet delay varies 22
with the modulation level. This is caused by changes in the transmission bandwidth, which results by buffer 23
retention of packets and the mapping process to wireless frames. It is therefore necessary to ensure that 24
equipment selected in the network for time sensitive application does not deteriorate the TE characteristics as 25
a result of operation of the ACM technology.  26
A.5 Impact of Band and Carrier Aggregation 27
Wireless transport systems operate in a variety of bands ranging from the lower microwave spectrum up to the 28
mmWave above 100 GHz. The characteristics of these bands are summarised in [x1], showing large bandwidth 29
availability in mmWave spectrum but with shorter link length, while the systems operating in the lower bands 30
of the microwave spectrum have longer links with narrower channel bandwidth is available. Band and Carrier 31
Aggregation (BCA) technique combines different channels that may be even in different bands, providing a 32
single big capacity pipe. In particular, BCA allows the combined benefits of the longer hop distance of 33
microwave systems with the high capacity in multiple Gigabits per second of the mmWave bands such as the 34
E-bands and above. However, this BCA pipe will have different propagations losses between the portion of 35
the links operating in the microwave band and the one operating in the mmWave band. Furthermore, these 36
links with multi -band operation would result in part of the links in one direction to disconnect resulting in 37
imbalance between the go and return of the wireless transport link. As such, it is necessary to consider the 38
effects of delay asymmetry and the impact of disconnection in part of the sub-channels over the BCA pipe. It 39
is therefore necessary to carefully verify the effect of the asymmetric effects and the imbalance in the 40
subchannels on the possible deterioration of TE characteristics. 41
A.6 Point to Multi Point (PMP) radio system 42
There are three types of systems that are used for the wireless transport system: Point-to-Point (PP), Point-to-43
Multi-Point (PMP), and Multi -Point-to-Multi-Point (MPMP) such as Mesh radio systems. PMP and MPMP 44
systems have asymmetric UL/DL latency. This limits, its capabilities in achieving good TE characteristics. 45
Equipment that are designed and able to achieve good TE characteristics should only be used for time sensitive 46
application with tight TE requirements. 47

The DL/UL delay in the Radio Interface of PP Radio systems are generally symmetric, and the degradation of 3
cTE is small. However, it should be noted that TE due to delay asymmetry of several ns to several tens of ns 4
is inevitable due to radio circuit configuration, filter group delay, and other factors in the radio processing part 5
of the PP system. This phenomenon varies depending on the system configuration such as its channel 6
bandwidth and the environment of the band in which they operate.  7
If the total TE of the NW has enough margin, these factors can be regarded as minor errors. However, when 8
designing for a tight total TE of the NW, it is desirable to use a MW node that has the ability to perform static 9
correction of the cTE. 10

A.8 Holdover Spec of BC function on the wireless transport node 12
The holdover function of the microwave and mmWave node is required as TE tolerance for the temporary 13
unlock state of a few tens of seconds during timeTransmitter clock rearrangement.  However, since the U -14

meaningless. A TE holdover characteristic of a few hundred ns per few tens of seconds is sufficient as shown 16
in G.8262 [14] or G.8262.1  [15], however, this depends on the assignment of the TE value of the 17
Rearrangement event in the Total TE Budget of the NW. 18

Annex B Radio operation when synchronization is lost 1
This section describes the radio operational impacts during the sync loss and recommendations or best practices 2
for Sync plane for handling the Radio operation during the sync loss. All the data processing clocks in O-RU 3
are driven by the S-plane Reference. If S-plane is down, we might lose sync. If O -RU implements Holdover 4
O-RU moves to Holdover state if not O -RU moves to FREE-RUN state. The M-Plane/OAM (Operation and 5
Maintenance modules) detects this S -plane state changes and might initiate shutting down data processing 6
paths and cells will be brought down to inactive state.  7
 8
B.1 Potential impacts due to sync loss on O-RUs 9
Usually O-RUs upon losing the lock (due to PTP/Sync-E down) might move to HOLDOVER. At this state 10
cells are still active and continue to be operational as much as they can. The holdover duration needed to 11
maintain the cells to be intact is subject to the holdover characteristics of the Oscillator (ageing, holdover per 12
day etc..), sync accuracy. An oscillator with good holdover characteristics would lead to a slow drift such that 13
the frequency reference is still within tolerance and the clocks don't change too much then we can continue as 14
the cells are effectively still in sync even though they have lost the clock reference.  If the frequency is out by 15
too much then the carrier starts drifting across the spectrum - potentially starting to encroach on adjacent 16
signals and causing signal corruption if it drifts beyond guard band. If there is a phase drift happening, then 17
the cell timing is out, and handsets will then exhibit jumps in range that they are not expecting and handover 18
from one cell to another may fail. Further static handsets may see ranging errors increase. During the sync loss 19
its essential to be able to meet 50ppb frequency limit and 3us of Time error (3GPP thresholds) for the cells to 20
be operational without any problems. 21
 22
B.1.1 TAE errors beyond the allowed range during sync loss 23
For the case of O -DU connected to multiple O -RUs, Sync loss on one of the connected O -RUs will lead to 24
TAE (Time Alignment errors) between the radio ports of O-RUs crossing the allowed thresholds based on the 25
chosen Air interface targets which will eventually impacts the connected cells and thus bringing down the UEs. 26
 27
B.1.2 Impact on Handover/Handoff 28
It is a very basic requirement of the system that as the mobile handset moves out of one cell to the next, it must 29
be possible to hand the call over from the base station of the first cell, to that of the next with no discernible 30
disruption to the call. This is termed as cell handover/handoff. It is necessary to ensure handoff can be 31
performed reliably and without disruption to any calls. handover or handoff is one of the key performance 32
indicators monitored so that a robust cellular handover / handoff regime is maintained on the cellular network. 33
Sync loss on O-RU will impact the cell handoff as the RU starts drifting in phase/frequency when the handset 34
moves from connected cell to the other and leading to call drops. 35

Annex C QoS Considerations for PTP packets 1
To achieve high degree of accuracy of the synchronization clock recovered from PTP, important aspect of the 2
overall solution is the precise time stamping of PTP packets. 3
 4
To achieve Class C requirements , p hysical layer time stamping must be implemented, since other time 5
stamping methods, or PTP packets without timestamping at all, do not provide suitable accuracy. From the 6
QoS perspective, PTP packets with physical layer time stamping do not require strict priority queueing to 7
optimize packet’s latency/jitter, since the queueing time is accounted by the physical layer time stamps. The 8
only requirement is some QoS queue with guaranteed bandwidth, to avoid PTP packet drop during congestion 9
events. 10

for PTPoE packets) and ITU-T G.8275.2 [3] PTP profile (with multi-hop PTPoIP sessions) are used across the 14
transport network. Depending on the transport network element capabilities, it can happen that the PTPoIP 15
packets are not time -stamped (i.e., T-BC/T-TC function to timestamp PTPoIP packets is missing) on transit 16

Annex E Synchronization over TDM PON  1
E.1 Short introduction to TDM PON 2
A TDM PON system is composed of an Optical Liner Termination (OLT) with one or more network ports and 3
one or more PON ports, the point-to-multipoint optical distribution networks (ODNs) terminated by the OLT, 4
and a set of Optical Network Units (ONUs) on each ODN. Each ONU then provides network connectivity to 5
one or more “user” devices connected to it. In the case of Mobile X-haul such devices are gNBs, O-CUs, O-6
DUs and/or O-RUs.  7
 8
TDM PONs are characterized by the shared medium nature of the ODN connecting multiple ONUs to a single 9
OLT port. The common bandwidth is shared in TDM fashion in downstream (DS) and TDMA fashion in 10
upstream (US), with a WDM separation between downstream and upstream signals for full duplex operation. 11
For upstream the TDMA is  performed by means of a Dynamic Bandwidth Assignment (DBA) algorithm in 12
the OLT that controls when each ONU can send a burst of data, so that bursts from different ONUs are 13
interleaved at the OLT receiver without overlaps. In downstream the TDM is done by each ONU only selecting 14
the packets that are destined to itself. 15
 16
 17
 18
Figure E-1: TDM PON system 19
 20
The latency for data transport is inherently asymmetrical in TDM PON; 21
• DS : latency by buffering in the OLT, FEC over PON, serialization of control words at line rate, fiber 22
propagation time, queuing & scheduling in ONU 23
• US: latency by TDMA in ONU (scheduling done by DBA in OLT), FEC over PON, serialization of 24
control words at line rate, fiber propagation time, buffering in OLT 25
 26
The delay discrepancy between US and DS can be large and this means that PTP cannot be used across a TDM 27
PON. The PON interface uses its own MAC encapsulation (over which Ethernet-based packets are transported 28
transparently), meaning that plain Sync-E as such is not readily available at the ONU PON interface. Therefore, 29
TDM PON systems have built in alternative methods for frequency and time synchronization.  30
 31
According to [1], a medium-specific access section can still act as a link in the synchronization chain by means 32
of a pair of converters between PTP and the medium, also known as Inter -Working Functions IWF. When 33
mapping this on a TDM PON system, the ODN is the access medium, the Transport Protocol Specific 34

E.3.2 TDM PON capabilities 7
 8
Mechanisms 9
 10
Synchronization over TDM PON is made possible by the fact that the system is inherently synchronized 11
between the OLT and its ONUs.  12
 13
First, each ONU is detected and automatically “ranged”, whereby the OLT measures its distance and sets time 14
equalization per ONU, to align on it on the US TDMA scheme. For this time alignment each ONU has access 15
to a common (arbitrary) time reference with the OLT (ITU PON: D/S 125µs frame boundary).  16
 17
Then the correlation to ToD (received at the OLT by PTP from the network) is added; each ONU retrieves the 18
absolute ToD for an association of a given event with a given timestamp. This association is communicated 19
by the OLT to the ONU in a management message. In ITU PON the ToD (at a hypothetical ONU, at start of 20
PON frame X) is communicated via OMCI to each ONU.  21
 22
Dependencies 23
 24

▪ RTT estimation depends on speed of light, which depends on n(λ) 26
▪ Variation on λ: the PON WDM bands in US and DS depend on the PON technology 27
▪ Accuracy of PON time equalization of ONUs (correction measure to deal with ONU drifts) 28
▪ Knowledge of internal delays in OLT 29
▪ Knowledge of internal delays in ONU 30

on: 33
▪ Location of the reference clock common to both O-RUs, which depends on the topology: 34
• Common PON card (same PON port or different PON ports): local card clock 35
• Common OLT node: local OLT clock 36

Centralized Monitoring System 2
H.1 Introduction 3
The precision of a frequency/phase and time recovered at the end of the synchronization chain depends on any 4
single node or combination of multiple nodes in the sync network. The sum of the timing errors introduced at 5
all nodes along the synchronization path needs to be within the required timing budget for a given application. 6
If there is an issue with nodes synchronization in a sync network chain, it is very complex to identify which 7
node had introduced the issue.   8
 9
End-to-end Sync monitoring feature can be used to identify synchronization issues in the network, correlation 10
of timing events and much more. It can be used to monitor the health of the nodes in synchronization network 11
and assess the timing performance of the node as well as complete synchronization network chain.  12
 13
End-to-end Sync monitoring feature is exercised in the Centralized Monitoring System (CMS). Every node in 14
the network chain will be monitored for the synchronization events and their impact on the node as well as on 15
the network.   16

H.2 Various elements of e2e sync monitoring 1
H.2.1 Nodes 2
An element in the network chain referred as a Node. Node can act as a PTP boundary clock, grandmaster clock 3
and/or end-time-receiver clock (shortly called as sync -nodes) in the synchronization chain. A Node can also 4
extract clock information (Physical Layer Frequency) from physical interface.  5
 6
Nodes are responsible for either generating, consuming or propagating frequency and time information over 7
the synchronization network.  8
 9
H.2.2 The Centralized Monitoring System (CMS) 10
This is a monitoring service running on a server or similar platform in a cloud or on-premise, to receive timing 11
telemetry data from the nodes in the synchronization network chain.  It is critical for the CMS to get timing 12
telemetry data from all nodes in the synchronization Network. 13
 14
The CMS shall analyse the telemetry data, do monitoring and reporting the health of the synchronization 15
network. 16
 17
H.3 CMS Implementation 18
Each Node in synchronization chain exports synchronization protocol (PTP) and state machine attributes. Most 19
of these attributes are referenced from IEEE1588 2019 [42] and ITU-T G.781 [19],  G.7721.1 [43] standards. 20
Some of the attributes are newly defined for End -to-end Sync Monitoring feature to support monitoring and 21
reporting requirements. The ITU -T PTP profiles (G.8275.1 [1] and G.8275.2 [3]) include the Performance 22
Monitoring option defined in G.8275 [41] Annex F that is based on IEEE1588-2019 Annex J [42].  23
 24
Yang data model -based telemetry mechanism shall be used to receive synchronization attributes from the 25
nodes. 26
 27
The CMS and nodes can use gNMI or NETCONF for streaming the data. 28
 29
Each node in the synchronization chain has a secure telemetry session with the CMS. This session can be 30
initiated either by a node or the CMS based on the management model. If a session is terminated, the CMS 31
shall report and log accordingly. Session can be re-established as per node accessibility for secure session as 32
per the transport protocol used. 33
 34
The telemetry interface shall use push subscription model to deliver data asynchronously. A request to “send 35
data” only once by the CMS to stream periodic updates by the nodes (Periodic subscriptions). 36
 37
Periodic streaming is less useful for time -critical events, in such instances, the CMS shall be configured for 38
“On Change/Notification” to receive streaming information whenever operational state changes (On-Change 39
Subscriptions). 40
  41

synchronization network chain and perform the static analysis as specified in section H.3.2. The protocol and 43
state attributes/data sets are shown in Table H.4-1, Table H.4-2 and Table H.4-3. 44
 45
