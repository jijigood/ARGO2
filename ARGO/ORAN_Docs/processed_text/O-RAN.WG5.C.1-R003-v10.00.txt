1 Scope 1 
The present document provides profiles for C-plane procedures and functions to achieve interoperability among different 2 
vendors. The profile specifies the expected behavior of each node, e.g., message flow of each use case, definitions of IEs, 3 
etc. which is not specified in 3GPP specifications. The profile specification provided in this document does not violate 4 
3GPP specifications. 5 
The use cases in this specification cover selected 3GPP functionalities.  6 
In the IE tables: 7 
- If optional IEs are essential for a use case to work, they are categorized ‘included’ in the IE tables.  8 
- If optional IEs are used under certain conditions applicable for this use case, t hey are categorized ‘optionally 9 
included’. 10 
- If optional IEs are not essential for the use case, they are set to ‘not included’.  11 
 12 
One or more use cases described in this document can be initiated in a single procedure if this is allowed by 3GPP 13 
specifications. In these cases, the IE tables need to be augmented and the ‘included’ IEs of the procedure are the 14 
combination of ‘included’ IEs of the individual use cases, unless otherwise specified in 3GPP.  15 
In addition, it is also possible to combine the initiation of one or multiple use cases of this document with other functions 16 
of the 3GPP procedure as defined in the relevant 3GPP specifications. 17 
The use cases are described based on following two architectures defined by 3GPP for 5G.  18 
- For a description of the overall architecture in E-UTRA NR Dual Connectivity (EN-DC) also known as NR 19 
Non-Standalone (NSA) as depicted in Figure 1-1 see TS 37.340 [1]. 20

1.1 UE connectivity options used in this document 1 
The 3GPP architecture provides different options in the way a UE can connect to the network. Throughout this 2 
document the configuration options as explained in the following chapters are used. Note that these chapters focus on 3 
explaining the connectivity for the profiled use case. There might be additional bearers such as MCG or default bearers 4 
that are not shown in the figures. 5 
1.1.1 NR Non-Standalone (NSA) 6 
In the NSA architecture the UE relies on LTE access to attach to EPC network and uses NR as secondary RAT for 7 
increased throughput using dual connectivity. In this version of the specification EN -DC is used with different options 8 
to establish the UP bearer(s).  9 
1.1.1.1 SN terminated SCG bearer (‘Option 3a’) 10 
First UE connectivity option is typically referred to as ‘Option 3a’. The PDCP and S1-U terminate in en-gNB acting as 11 
Secondary Node (SN). From there SCG bearers are used to convey UP data to/from the UE (‘SN t erminated SCG 12 
bearer’). 13

1.1.2 NR Standalone (SA) 9 
In NR SA, also referred to as ‘Option 2’, the NG-RAN based on gNB RAN nodes only is connected to 5GC, i.e. no 10 
EPC is involved and ng-eNBs are not considered to be part of NG-RAN. 11 
1.1.2.1 NR-DC with SN terminated Split bearer 12 
In case a UE connects with option 2 and uses NR-NR dual connectivity (NR-DC) this specification in its current version 13 
is profiling cases where MN and SN are gNBs. 3GPP shows options to terminate NG-U on MN, SN or both. Current 14 
version of this specification considers use cases with PDCP terminated on SN. SN splits the UP bearer in a leg via Xn -15 
U and, if applicable F1-U, to convey UP data via bearer using RLC in both, MCG and SCG (‘SN terminated split 16 
bearer’). 17 
When profiling the NR SA SN terminated split bearer cases it was further assumed that MN is serving FR1 cells with a 18 
larger coverage area than FR2 cells served by SN.  By that the amount of unwanted mobility inter MN mobility events 19 
is reduced.  20
