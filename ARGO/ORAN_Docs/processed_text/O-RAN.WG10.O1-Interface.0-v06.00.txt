 Philosophy 174 
It is expected that O-RAN Managed Elements, specified in O-RAN Operations and Maintenance Architecture [32], 175 
comply with the O1 Interface Specification, except the O-RU which complies with the O-RAN Fronthaul Management 176 
Plane Specification Error! Reference source not found.. 177 
The O-RAN O1 management services follow existing 3GPP standards wherever possible.  The focus of this document 178 
is to identify the use cases which conform to existing 3GPP standards, identify gaps in management services for O-179 
RAN and define needed extensions.  For identified gaps, the goal is to modify the 3GPP standards to include the needed 180 
O-RAN extensions and update the references in this document as the 3GPP standards evolve to cover the gaps. In cases 181 
where the 3GPP standards are not modified, O-RAN extensions are specified in this, and other, O-RAN documents. O-182 
RAN extensions should be compatible with 3GPP standards as much as possible to avoid divergence. If extensions and 183 
gaps are not specified, it is expected that the management services providers and consumers are conforming to 184 
referenced 3GPP specifications. 185 
This O1 Interface Specification specifies the management services (MnS) supported in the O-RAN architecture between 186 
O1 compliant Managed Elements (MnS providers) and the SMO (MnS consumer).  It defines common MnS 187 
descriptions, requirements, procedures, operations, and notifications. In addition, there are complementary O1 Interface 188 
Specifications for the Near-RT RIC Error! Reference source not found., the O-CU-UP and O-CU-CP Error! 189 
Reference source not found. and the O-DU Error! Reference source not found. that define the O1 specificities, 190 
extensions, and restrictions for that particular managed function (MF).   191 
For example,  192 
 O1 Interface Specification for a MF may contain the alarms and performance measurements generated by that 193 
MF,   194 
 O1 Interface Specification for O-DU may include extensions needed by O-DU for managing O-RU in 195 
hierarchical mode, 196 
 O1 Interface Specification for Near-RT RIC may document the O1 MnS services that it does not provide and 197 
may reference a related specification containing procedures and APIs for managing xApps. 198

 Open Points 200 
As each Management Service is evaluated, the Use Cases and relevant specifications need to be assessed and 201 
augmented as needed to support O-RAN. The current list of Use Cases in Chapter 2 below is not exhaustive, and the list 202 
of specification references may not be complete. 203 
Some Use Cases referred to in the standard may not be applicable to O-RAN and this needs to be addressed, as an 204 
exception.  205 
Future clarifications may be added to specify when citing a reference whether it is being used as a citation (meaning it 206 
will be strictly followed) or as a reference.  More precise terminology may be included as this draft matures. 207

and requirements for all O-RAN Working Groups (WG) and O-RAN entities.  The security architecture, guidelines and 209 
requirements will be incorporated into existing O-RAN documents, as appropriate. The O1 Interface Specification 210 
intends to comply with the security architecture, guidelines and requirements that are applicable to the O1 interface and 211 
will provide appropriate references, when available. 212

As specified in Section 1.6.6 below, it is agreed that a PNF MnS provider must be able to inform a PNF MnS consumer 217

 General Requirements 223 
This section contains general requirements that are applicable to many O1 Interface Management Services. 224 
1.6.1 Service Management and Orchestration (SMO) 225 
REQ-SMO-FUN-1:  O-RAN compliant SMOs SHALL support the O1 interfaces as defined in this document. 226 
1.6.2 Transport Layer Security (TLS) 227 
TLS requirements are specified in O-RAN Security Protocol Specifications [34] section 2.2 and 2.3. 228 
1.6.3 HyperText Transfer Protocol (HTTP) 229 
REQ-HTP-FUN-1:  Management Service providers and consumers that use HTTP SHALL support HTTP v1.1 or 230 
higher.  HTTP v2.0 is preferred. 231 
1.6.4 Secure Shell (SSH) 232 
SSH requirements are specified in O-RAN Security Protocol Specifications [34] section 2.1.  233 
1.6.5 Least Privilege Access Control 234 
Least privilege access control requirements for the O1 interface are specified in O-RAN Security Requirements 235 
Specification [35] section 3.2.2.  236 
1.6.6 Reset Notification and Reset Reason 237 
REQ-RN-FUN-1: A PNF MnS Provider shall support the capability to inform a PNF MnS Consumer that a reset has 238 
occurred and the reason that a HW unit has reset. 239 
REQ-RN-FUN-2: A PNF MnS Provider shall save the reason for a reset persistently before resetting.  Note that this 240 
requirement applies to resets that occur under the control of the PNF.   241 
REQ-RN-FUN-3: If a reset reason has not been saved persistently, the PNF MnS Provider shall set the reset reason to 242 
unknown in the notification.  Note that this requirement applies to resets that occur unexpectedly before the reset reason 243 
could be stored. 244

An SDO O1 format notification is an O1 notification formatted as specified by a Standards Defining Organization (SDO). 251 
Currently, O1 supports SDO O1 format notifications that are either 3GPP-specified or O-RAN-specified. SDO O1 format 252 
notifications are formatted as specified by the SDO and sent without a VES header. 253 
3GPP-specified O1 notifications are specified in 3GPP TS 28.532 [4], clause 11 and clause 12. 254 
O-RAN-specified O1 notifications will be specified in the O-RAN Information Model and Data Models Specification 255 
[33]. 256 
O-RAN-specified O1 notifications should follow 3GPP naming and format where possible to reduce the number of 257 
variants that need to be supported.  Specifically, O-RAN-specified O1 notifications should: 258 
 be named “o1NotifyXxx” 259 
 include the common 3GPP notification fields objectClass, objectInstance, notificationId, notificationType, 260 
eventTime and systemDN 261 
 include an additionalText and/or additionalInformation field when appropriate. 262 
A VES O1 format notification is an O1 notification formatted as specified by VES Event Listener Specification [36], 263 
consisting of a common event header and domain-specific event fields. VES O1 format notifications are categorized into 264 
2 types, based on domain: 265 
 Harmonized VES 266 
 Legacy VES 267 
Harmonized VES refers to the stndDefined VES event specified in VES Event Listener Specification [36] that allows a 268 
VES event to carry, as its payload, a notification specified by an SDO.  In the case of O-RAN O1 Interface Specification, 269 
a harmonized stndDefined VES event carries either a 3GPP-specified O1 notification or an O-RAN specified O1 270 
notification as its payload. 271 
Legacy VES refers to any VES event specified in the VES Event Listener Specification [36], except for stndDefined.  272 
Legacy VES events are fully defined in [36] and don’t rely on an SDO to specify the content of the payload.  The Legacy 273 
VES events supported by O1 Interface Specification are Fault, Heartbeat, File Ready and PNF Registration. 274 
Legacy VES events are supported for backward compatibility.  However, harmonized VES events are preferred.  Use of 275 
harmonized VES events results in less notification variants for the providers and the consumers because a harmonized 276 
VES O1 format notification is effectively an SDO O1 format notification wrapped in a VES common event header.   277 
Two attributes are used to communicate the notification format between MnS provider and MnS consumer. 278 
 o1NotifyFormatCapabilities indicates whether the MnS provider supports the capability to send notifications in 279 
SDO O1 format, VES O1 format or both.  This attribute is set by the MnS provider at the Managed Element 280 
level, meaning the capability is for all O1 notifications sent by that MnS provider.  It is not per notification 281 
type. This attribute is read-only for the MnS consumer. 282 
 o1NotifyFormatConfig indicates whether the MnS consumer wants to receive notifications in SDO O1 format 283 
or VES O1 format. This attribute is configured by the MnS consumer at the Managed Element level, meaning 284 
the configuration is for all O1 notifications sent by that MnS provider.  It is not per notification type.  If the 285 
MnS provider supports both formats, the MnS provider sets the value for this attribute to the default value of 286 
VES O1 format when the MOI is created and the MnS consumer is permitted to change the value to SDO O1 287 
format if desired. Otherwise, if the MnS provider only supports one notification format, this attribute is absent. 288 
These two attributes will be specified in the O-RAN Information Model and Data Models Specification [33]. 289 
It is not necessary to have an attribute to indicate whether harmonized VES or legacy VES is sent for VES format 290 
because the domain of the event is provided in the VES common event header and the schema of the event is provided 291 
by the Network Function at onboarding time.292

Trace management services allow a Trace MnS Provider to report file-based or streaming trace records to the Trace 615 
MnS Consumer.  Trace Control provides the ability for the Trace Consumer to start a trace session by configuring a 616

Trace Sessions are configured on the provider with information on where and how to send the trace information to the 624 
consumer.  The provider creates trace records within a trace session as the trigger mechanism occurs.  Trace records are 625 
produced and provided to the consumer until the trace session is terminated. 626 
File-based trace collects trace records in files that are available to the consumer with a time delay.  In the case of 627 
streaming trace, the data is sent in bursts across a WebSocket connection to the consumer, maintaining the relevance of 628 
the data while minimizing transport overhead. 629

2.8.1 Software Package Naming and Content 1107 
PNF Software Package naming, content and format are vendor specific and do not require standardization in O-RAN.  1108 
A PNF Software Package may contain one or more files.  Some of the files in the Software Package may be optional for 1109 
the PNF (example: a file that has not changed version).  The PNF is aware of the content and format of its available 1110 
Software Packages and can determine which files it needs to download. 1111 
The softwarePackage Managed Object Class (MOC) contains attributes about a software package such as:  software 1112 
package name, version, fileList, integrityStatus (valid, invalid, empty), runningState (active, passive), vendor, 1113 
productName, softwareType (operational, factory), etc. This MOC is applicable to VNFs and PNFs and is a generic 1114 
term that O-RAN will use to refer to the software available on the PNF rather than the legacy term of software slot 1115 
The PNF creates one instance of softwarePackage for each software package supported concurrently on the PNF.  1116 
Typically, a PNF will have two softwarePackage MOIs for operational software; one with runningState = active and 1117 
one with runningState = passive.  Some PNFs also have a softwarePackage MOI for the factory software which would 1118 
be read only.  O-RAN may have PNFs that support more than one passive slot.  In this case the inventory query result 1119 
would show multiple MOIs with runningState=passive. 1120

2.9.1 PNF Reset Command 1309 
 Description 1310 
The PNF Reset Command procedure allows a PNF Reset MnS Consumer to trigger a reset of a HW unit of a PNF Reset 1311 
MnS Provider on command.  Any HW unit that is resettable via a reset command must be represented by a managed 1312 
object instance (MOI) and is able to be identified by a distinguished name (DN).  The NETCONF RPC <reset> 1313 
command identifies the unit to reset by the DN.  The unit to reset may be the entire PNF or a resettable HW 1314 
subcomponent of the PNF.  A resettable HW subcomponent of a PNF is a subcomponent of a PNF that is able to be 1315 
independently reset and whose PNF Reset MnS Provider supports a reset command for the subcomponent. It is vendor 1316 
and PNF-specific whether a PNF has resettable HW subcomponents.  The reset command also has an attribute to 1317 
identify the type of reset requested.  The types of reset commands that a PNF supports are vendor and PNF specific.  O-1318 
RAN O1 Interface Specification specifies two mandatory reset command types that every PNF must support: 1319 
conditional and forced.  A conditional reset command can be rejected by the PNF Reset MnS Provider depending on the 1320 
conditions on the PNF, for example if the unit to reset is not in a proper state to reset, such as, if there is an emergency 1321 
call in progress on the unit.  A valid forced reset command cannot be rejected.  Valid means that the unit to reset 1322 
supports a reset command. Invalid forced resets will be rejected, for example, if the unit to reset is not a resettable HW 1323 
unit, such as a cell.  Vendors are allowed to extend the O1 specified reset command types to add vendor and PNF 1324 
specific reset command types. 1325

A stndDefined VES event, as specified in VES Event Listener Specification [36], allows a VES event to carry, as its 6 
payload, a notification specified by an SDO.  In the case of O-RAN O1 Interface Specification, a harmonized 7 
stndDefined VES event carries either a 3GPP-specified O1 notification or an O-RAN-specified O1 notification as its 8 
payload. 9 
3GPP has published an informative Annex B in TS 28.532 [4] providing guidelines for the integration of 3GPP-10 
specified notifications with VES.  This Annex expands on the information provided by 3GPP, including information on 11 
how to include O-RAN-specified O1 notifications in a VES stndDefined event. 12 
When an O-RAN and 3GPP compliant ME supports VES stndDefined events for sending asynchronous notifications, 13 
a3GPP-specified O1 notification, as defined by 3GPP, or an O-RAN-specified O1 notification, as defined by O-RAN, is 14 
included in the event. 15 
A VES common event header, as defined by VES Event Listener Specification [36], is added to the notification. 16 
In VES, the domain field in the common event header is used to route the event to the proper consumers and to map to a 17 
schema for the event payload.  VES Event Listener Specification [36] added a new domain field enumeration value 18 
called stndDefined that indicates that the event is complying with a schema defined by a standards body.  19 
An additional field was added to the VES common event header called stndDefinedNamespace, which contains a valid 20 
namespace as defined by the standards body.  This field is only populated when the domain is stndDefined.  3GPP has 21 
defined four namespaces in TS 28.532 [4] Annex B; namely 3GPP-Provisioning, 3GPP-Heartbeat, 3GPP-22 
FaultSupervision and 3GPP-PerformanceAssurance.  Likewise, O-RAN must specify namespaces for the notifications it 23 
defines.  A VES collector uses the stndDefinedNamespace, along with the stndDefined domain, to route the event to the 24 
correct consumer.   25 
A stndDefined VES event has a field structure called stndDefinedFields, specified in VES Event Listener Specification 26 
[36].  This structure contains three properties: 27 
 schemaReference (type = string, format = uri) 28 
 data (JSON object which should be identical to the 3GPP or O-RAN notification) 29 
 stndDefinedFieldsVersion (type = string, format = enum) 30 
 31 
The schemaReference, if present, should be used to verify that the notification content is correct.  3GPP is publishing 32 
the notification schemas, in JSON format, to a public repository, (https://forge.3gpp.org/rep/sa5) so that schema 33 
references can be included in the event.  Likewise, O-RAN will publish its notification schemas, in JSON format, in a 34 
public repository.  This repository is still to be created.  35 
The data element contains either a 3GPP-specified O1 notification, in JSON format, as specified in 28.532 [4] or an O-36 
RAN-specified O1 notification, in JSON format, as specified in O-RAN Information Model and Data Models 37 
Specification [33]. 38 
The stndDefinedFieldsVersion provides the version of the stndDefinedFields structure, as defined by VES Event 39 
Listener Specification [36].  40

Scenario 7 
1. Provisioning Management Service Consumer activates/configures Trace Session on Trace Provider.  This will 8 
be accomplished using Provisioning Management services described in Section 2.1 of this document. 9 
2. Trace Provider sends a notifyMOIChanges to indicate the new MOI is created. 10 
3. Trace Provider needs to establish a connection to the Trace Consumer to set up a streaming connection 11 
(streams are active at this time between the Provider and Consumer). This is done using the 12 
establishStreamingConnection Operation via an HTTP POST request containing MetaData associated with this 13 
Trace Session. 14 
4. Trace Consumer responds with an acknowledgement that contains the ConnectionID needed by the Provider 15 
when requesting that the connection be upgraded to a WebSocket to support streaming of the trace data. 16 
5. Trace Provider requests the upgrade of the connection to a WebSocket using the ConnectionID and an HTTP 17 
GET operation.  18 
6. Trace Consumer accepts the upgrade and WebSocket is established.  WebSocket will remain connected until 19 
the last streaming trace session active on the Trace Provider is ended.  Note in this example, only one 20 
streaming trace session is active. 21 
7. Trace Provider starts trace session, waiting for triggering event to occur. 22 
8. Triggering event occurs and a new trace recording session is started on the Trace Provider.  Each trace 23 
recording session has a unique Trace Recording Session (TRS) Reference associated with it. 24 
9. Trace Provider sends Trace Data to Trace Consumer indicating that this is the start of a new trace record 25 
session. The start trace recording session information is included in the reportStreamingData Operation.  26 
10. While this trace record is active, the Trace Provider collects trace data. 27 
11. When the reporting timer expires or enough trace data is available, the Trace Provider sends a trace data report 28 
to the Trace Consumer containing trace record data for active recording sessions in a trace session.  These 29 
records are the payload of the reportStreamingData operation. 30 
12. If the timer expires, but no trace data is present on the Trace Provider, the producer will send an alive message 31 
to the Trace Consumer to confirm that the session is still active.  The alive message is the payload of the 32 
reportStreamingData operation. 33 
13. When the criteria for the trace recording session completion occurs (call ends, etc.), the Trace Provider stops 34 
collecting data for this trace recording session. 35 
14. Trace Provider creates a record that indicates this trace recording session has ended and includes this record in 36 
the payload of the reportStreamingData operation to the Trace Consumer. 37 
15. Provisioning Management Service Consumer deactivates the trace via procedures defined in section 2.1 of this 38 
document.  Deactivation means that the trace data collection should cease, and the Trace Provider should stop 39 
all active trace recording sessions and send data that it has collected up to this point, if any, for each active 40 
trace recording to the Trace Consumer. 41 
16. Trace Provider initiates the termination of the trace session. 42 
17. For each active session, Trace Provider initiates a Stop Trace Recording Session. 43 
18. If there are outstanding record(s) for this trace recording session that have not been streamed to the Trace 44 
Consumer, Trace Provider sends them as the payload of the reportStreamingData operation. 45 
19. Trace Provider informs the Trace Consumer that this Trace Recording Session has ended by sending the trace 46 
record termination via the reportStreamingData Operation. The producer repeats this until all trace recording 47 
sessions for this trace session have been terminated. 48 
20. In this example, only one stream was on the connection, so the WebSocket should be disconnected.  To 49 
accomplish this, Trace Provider sends the Trace Consumer the deleteStream operation indicating that this 50 
Trace Session has been terminated. 51 
21. When all active Trace Sessions between Trace Provider and Trace Consumer have ended, the WebSocket 52 
connection is to be torn down.  In this example, this occurs when this session terminates because only one 53 
session is active.  To terminate the session, Trace Provider sends the Trace Consumer the 54 
terminateSignalingConnection Operation which is a WebSocket close frame. 55 
22. WebSocket connection is torn down. 56
